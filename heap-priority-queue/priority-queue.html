Priority Queues in Practice

Imagine the line at a hospital emergency room. People don't get seen in the order they walked in — the patient with the most urgent injury goes first, no matter when they arrived. That's a priority queue: a line that's always sorted by importance, not by arrival time.

Formally, a priority queue (PQ) is an abstract data type — a contract that says "give me the most important item next." It doesn't dictate how that's done. A heap is the usual machinery underneath, but in real code you almost never build it yourself: every language ships a priority queue in its standard library, and you just use it.

Contrast it with a normal queue. A queue is FIFO — first in, first out, like a line at a coffee shop. A priority queue pops by priority, not arrival order. The "best" element (smallest or largest, depending on the setup) always comes out next, even if it was the last one added.

The mental model: a PQ is a bag where you can throw items in any order, and when you reach in, you always pull out the highest-priority one. Two questions decide everything: (1) is "highest priority" the smallest or the largest value? (2) what counts as the priority when items are complex objects?

The Language Gotchas

This is the part that actually trips people up. The three big languages disagree on the defaults, and mixing them up is a classic interview and code-review bug. Let's nail down all three.

1. Default direction: min-heap or max-heap?

Out of the box, each language has an opinion about what "most important" means:

  • C++ std::priority_queue is a MAX-heap — top() gives you the largest element.
  • Java PriorityQueue is a MIN-heap — peek() gives you the smallest.
  • Python heapq is a MIN-heap too — and it's a set of functions over a plain list, not a class you instantiate.
// C++ std::priority_queue is a MAX-heap by default
#include <queue>

std::priority_queue<int> pq;

pq.push(5);
pq.push(1);
pq.push(9);          // holds {5, 1, 9}

int t = pq.top();    // peek -> 9 (the LARGEST)
pq.pop();             // removes 9. NOTE: pop() returns void
// now top() == 5

if (!pq.empty()) {    // always guard before top()/pop()
    int x = pq.top(); // 5
}

// Java PriorityQueue is a MIN-heap by default
PriorityQueue<Integer> pq = new PriorityQueue<>();

pq.offer(5);          // add() works too
pq.offer(1);
pq.offer(9);          // holds {5, 1, 9}

int t = pq.peek();    // peek -> 1 (the SMALLEST)
int x = pq.poll();    // removes AND returns 1
// now peek() == 5

if (!pq.isEmpty()) {   // peek()/poll() return null when empty
    int y = pq.peek(); // 5
}

# Python heapq is a MIN-heap, and it's FUNCTIONS over a list
import heapq

pq = []                  # just a plain list

heapq.heappush(pq, 5)
heapq.heappush(pq, 1)
heapq.heappush(pq, 9)    # heap-ordered list

top = pq[0]              # peek -> 1 (the SMALLEST), no removal
x = heapq.heappop(pq)    # removes AND returns 1
# now pq[0] == 5

if pq:                   # empty list is falsy -> your isEmpty check
    y = pq[0]          # 5

2. Flipping the direction

Need the opposite of the default? Each language has its own trick. The idea is always the same — change what "comes first" means.

// C++: a MIN-heap needs greater<> as the comparator
#include <queue>
#include <vector>

std::priority_queue<int, std::vector<int>, std::greater<int>> pq;

pq.push(5);
pq.push(1);
pq.push(9);

int t = pq.top();    // 1 (the SMALLEST) — now a min-heap

// Java: a MAX-heap uses Collections.reverseOrder()
PriorityQueue<Integer> pq =
    new PriorityQueue<>(Collections.reverseOrder());

pq.offer(5);
pq.offer(1);
pq.offer(9);

int t = pq.peek();   // 9 (the LARGEST) — now a max-heap

# Python: no max-heap built in — push NEGATED values
import heapq

pq = []
for v in (5, 1, 9):
    heapq.heappush(pq, -v)   # store the negatives

biggest = -heapq.heappop(pq) # negate back -> 9 (LARGEST)

# (CPython also has heapq._heapify_max, but it's private — avoid it.)

3. Custom comparators & ordering objects

Real items are rarely bare integers. Say each item is a task with a priority number, and you want the highest-priority task out first. Now you must tell the PQ which field to order by.

// C++: order tasks by priority with a lambda comparator
#include <queue>
#include <vector>
#include <string>

struct Task { int prio; std::string name; };

// returns true if a should come AFTER b -> highest prio on top
auto cmp = [](const Task& a, const Task& b) {
    return a.prio < b.prio;
};
std::priority_queue<Task, std::vector<Task>, decltype(cmp)> pq(cmp);

pq.push({2, "email"});
pq.push({5, "pager"});   // highest prio
pq.push({1, "log"});
// pq.top().name == "pager"

// Java: pass a Comparator (lambda) to order by a field
record Task(int prio, String name) {}

// max-heap on prio: highest priority polled first
PriorityQueue<Task> pq =
    new PriorityQueue<>(Comparator.comparingInt(Task::prio).reversed());

pq.offer(new Task(2, "email"));
pq.offer(new Task(5, "pager"));  // highest prio
pq.offer(new Task(1, "log"));
// pq.peek().name() == "pager"

# Python: push TUPLES (priority, tiebreaker, item)
import heapq
import itertools

pq = []
counter = itertools.count()   # unique, increasing tiebreaker

def add(prio, name):
    # negate prio for "highest first"; counter breaks ties
    heapq.heappush(pq, (-prio, next(counter), name))

add(2, "email")
add(5, "pager")            # highest prio
add(1, "log")

_, _, name = heapq.heappop(pq)  # name == "pager"
Python heapq survival kit:
  • Store (priority, item) tuples — heapq compares tuples element-by-element.
  • For a max-heap, negate the priority on the way in and out.
  • Add a tiebreaker (like a counter) between the priority and the item, so heapq never tries to compare two un-orderable objects when priorities tie: (priority, count, item).
The classic cross-language bug: C++'s default priority_queue is a MAX-heap, but Java and Python default to MIN-heaps. Porting code between them without flipping the comparator silently gives you the wrong "best" element every time.

Also: don't mutate an element's priority while it's still inside the PQ — the heap won't re-order itself and your ordering quietly breaks. Remove the item and re-add it with the new priority (the "lazy deletion" pattern) instead.

Where PQs Power Real Algorithms

Once you can "always grab the best option next" in O(log n), a whole family of greedy algorithms falls out almost for free. Here's where priority queues quietly do the heavy lifting.

Graphs & pathfinding

  • Dijkstra's shortest path — pop the closest unsettled node, relax its neighbors, repeat.
  • Prim's MST — pop the cheapest edge that grows the tree without forming a cycle.
  • A* search — pop the node with the best estimated total cost (distance so far + heuristic).

Scheduling & encoding

  • Huffman coding — repeatedly pop the two smallest frequencies and merge them into one node.
  • Event-driven simulation — pop the next event by timestamp to advance the clock.
  • Task scheduling — pop the highest-priority (or earliest-deadline) job to run next.

The common thread: each of these is a greedy algorithm that, at every step, needs the single best option out of a changing set. That "best of a changing set, fast" job is exactly what a priority queue was built for.

Complexity at a Glance

Because a PQ is backed by a heap, the costs are the heap's costs. Notice that peeking the best element is free — it's always sitting at the top.

OperationTime Complexity
push (insert)O(log n)
pop (remove best)O(log n)
peek / top (read best)O(1)
build from n itemsO(n)
sizeO(1)

Building from n items all at once (C++ constructor, Java constructor with a collection, Python heapify) is O(n) — cheaper than pushing them one at a time, which would be O(n log n).

Pattern to remember: if a problem says "always take the cheapest / closest / highest-priority next," "Kth largest," or "merge K sorted things," reach for a priority queue. Just remember to decide min vs max before you write the comparator.
