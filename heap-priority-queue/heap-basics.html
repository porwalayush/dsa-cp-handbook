Heaps: The Basics

Imagine a company org chart, or the bracket of a knockout tournament. Every manager sits above their direct reports, and there's a rule everyone follows: a boss is always "more important" than the people under them. Follow the chart all the way up and you land on one person at the very top — the most important of all, reachable in a single glance.

That's the whole idea behind a heap. It's a tree where every parent outranks its children by some measure. If "more important" means "smaller," the smallest value floats to the very top. If it means "larger," the largest does. Either way, the champion is always sitting right at the root, ready to be read instantly.

What makes a heap special isn't just the ordering — it's that this neat, predictable shape lets us pack the whole tree into a plain array with no pointers and no wasted space. That trick is the heart of this lesson.

First, a Complete Binary Tree

Before the ordering rule, a heap has a strict shape rule. A heap is always a complete binary tree: every level is completely full, except possibly the last one — and that last level fills in strictly left to right, with no gaps.

Think of seating people in a theater row by row. You fill the front row completely before anyone sits in the second row, and within a row you fill seats left to right. You never leave an empty seat with someone sitting to its right.

This "no gaps" guarantee is exactly why a heap fits so perfectly into an array. Because the nodes pack in left-to-right with no holes, you can lay them out in reading order — level by level, left to right — and the array has no empty slots to work around.

The Heap Property

Now the ordering rule, called the heap property. It comes in two flavors:

  • Min-heap — every parent is less than or equal to its children. The smallest value bubbles all the way to the root.
  • Max-heap — every parent is greater than or equal to its children. The largest value sits at the root.

That's the only constraint. Notice how loose it is: a parent must beat its own children, but siblings have no ordering at all, and a node on the left of the tree could be larger or smaller than some unrelated node on the right.

So a heap is not a sorted structure, and it is not a binary search tree. It makes one promise and one promise only: the best element is at the top. That single guarantee is enough to power priority queues, heap sort, Dijkstra, and more.

The Key Insight: A Tree Inside an Array

Here's the trick that makes heaps so efficient. Because a complete binary tree has no gaps, we can store it in a flat array by reading the tree level by level, left to right. The root goes at index 0, its two children at indices 1 and 2, the next level at 3, 4, 5, 6, and so on.

Once the nodes are laid out this way, you never need pointers. Given any index, plain arithmetic tells you exactly where a node's parent and children live (using 0-based indexing):

  • For a node at index i: left child = 2i + 1
  • right child = 2i + 2
  • parent = (i - 1) / 2 (integer division, i.e. floor)

Take the array [1, 3, 6, 5, 9, 8]. Index 0 holds 1 (the root). Its children are at indices 1 and 2 — values 3 and 6. The node at index 1 (3) has children at indices 3 and 4 — values 5 and 9. Every relationship is just a multiply-and-add away. No tree nodes, no pointers, no allocation per element.

0-indexed vs 1-indexed. The formulas above assume the array starts at index 0 (the convention in C++, Java, and Python). Some textbooks start the heap at index 1 instead, because the math looks cleaner:
  • 1-indexed: left = 2i, right = 2i + 1, parent = i / 2 (floor).
Both are correct — just be consistent and never mix the two in one implementation.

Min-Heap vs Max-Heap

Same shape, same array layout — the only difference is which direction the ordering points. Pick the one that puts the element you care about at the root.

Min-Heap

  • Every parent ≤ its children.
  • The smallest element is at the root (index 0).
  • peek() gives you the minimum in O(1).
  • Default in Python's heapq and Java's PriorityQueue.
  • Great for "give me the cheapest / closest / earliest next."

Max-Heap

  • Every parent ≥ its children.
  • The largest element is at the root (index 0).
  • peek() gives you the maximum in O(1).
  • Default in C++'s priority_queue.
  • Great for "give me the biggest / most urgent next."

A heap is NOT sorted, and NOT a binary search tree. This trips up almost everyone at first. In a BST, everything to the left of a node is smaller and everything to the right is larger — so you can binary-search it. A heap makes no such promise: siblings are unordered, and the left subtree can hold values larger than the right. The only guaranteed-fast lookup is the root (the min or max). Searching for any other value is O(n) — you have to scan. Do not try to binary-search a heap.

The Representation in Code

You rarely hand-roll a heap (the standard library gives you one), but seeing the index math makes the whole structure click. Here's how the array maps to a tree and the three little helper functions that navigate it.

// A heap is just an array. These helpers do all the navigation.
#include <vector>

std::vector<int> heap = {1, 3, 6, 5, 9, 8};
//   index:  0  1  2  3  4  5
//
//                 1            <- index 0 (root, the minimum)
//               /   \
//              3     6        <- index 1, 2
//             / \   /
//            5   9 8          <- index 3, 4, 5

int parent(int i) { return (i - 1) / 2; }   // floor division
int left  (int i) { return 2 * i + 1; }
int right (int i) { return 2 * i + 2; }

// node at index 1 (value 3): parent=0, left=3, right=4
int p = parent(1);   // 0  -> value 1
int l = left(1);     // 3  -> value 5
int r = right(1);    // 4  -> value 9

// a child index is only valid if it is < heap.size()

// A heap is just an array. These helpers do all the navigation.
int[] heap = { 1, 3, 6, 5, 9, 8 };
//   index:  0  1  2  3  4  5
//
//                 1            <- index 0 (root, the minimum)
//               /   \
//              3     6        <- index 1, 2
//             / \   /
//            5   9 8          <- index 3, 4, 5

static int parent(int i) { return (i - 1) / 2; }  // floor
static int left  (int i) { return 2 * i + 1; }
static int right (int i) { return 2 * i + 2; }

// node at index 1 (value 3): parent=0, left=3, right=4
int p = parent(1);   // 0  -> value 1
int l = left(1);     // 3  -> value 5
int r = right(1);    // 4  -> value 9

// a child index is only valid if it is < heap.length

# A heap is just a list. These helpers do all the navigation.
heap = [1, 3, 6, 5, 9, 8]
#  index:  0  1  2  3  4  5
#
#                1            <- index 0 (root, the minimum)
#              /   \
#             3     6        <- index 1, 2
#            / \   /
#           5   9 8          <- index 3, 4, 5

def parent(i): return (i - 1) // 2   # // is floor division
def left(i):   return 2 * i + 1
def right(i):  return 2 * i + 2

# node at index 1 (value 3): parent=0, left=3, right=4
p = parent(1)   # 0  -> value 1
l = left(1)     # 3  -> value 5
r = right(1)    # 4  -> value 9

# a child index is only valid if it is < len(heap)

Complexity at a Glance

The headline is the root: reading the min (or max) is instant. Everything that changes the heap costs O(log n) because a value may have to travel the height of the tree — and a complete tree of n nodes is only about log₂(n) levels tall. The operations themselves are covered in the next lesson; here are the costs to keep in mind.

OperationTime Complexity
peek / find-min (or max) — read the rootO(1)
insert (add an element)O(log n)
extract-min (or max) — remove the rootO(log n)
build-heap (heapify a whole array)O(n)
search for an arbitrary valueO(n)
spaceO(n)

Build-heap looks like it should be O(n log n), but a clever bottom-up heapify gets it down to O(n). Arbitrary search is O(n) because — as the warning above says — a heap gives you no shortcut to anything but the root.

Pattern to remember: whenever a problem keeps asking for the "smallest so far," the "largest," the "cheapest next," or the "top K," a heap is almost certainly the right tool. It hands you the extreme element instantly and re-balances itself in O(log n). The next lesson shows exactly how insert and extract keep the heap property intact.
