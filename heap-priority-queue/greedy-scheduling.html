Heaps for Greedy & Scheduling

Picture a busy coffee counter where you always serve whoever has been waiting longest, or a checkout line where the express lane keeps grabbing the customer with the fewest items. Each time, you make one move: pick the most extreme thing right now — the longest wait, the smallest cart, the cheapest option — act on it, then look again.

That repeated "take the best next" move is the heartbeat of greedy algorithms. And a heap is the perfect engine for it: it always keeps the current champion at the root, hands it to you in O(1), and re-settles itself in O(log n) after you pop it or push something new.

This lesson is about a single, reusable pattern: whenever a problem says "repeatedly take the smallest / largest / most-frequent / cheapest next," reach for a heap. We'll see it power tree-building, rope-merging, CPU scheduling, string layout, and room allocation — five problems that look unrelated until you notice they all run the same loop.

The unifying insight. A heap turns "repeatedly pick the extreme element" from O(n) per pick (scan the whole collection every time) into O(log n) per pick. That single speed-up is exactly why heaps power so many greedy algorithms — the greedy strategy decides what to grab next, and the heap makes grabbing it fast.

1. Huffman Coding — a Heap that Builds a Tree

Here's the centerpiece. Suppose you want to compress text. Letters that appear a lot (like e or a space) should get short codes; rare letters (like z) can afford long ones. That's the whole intuition behind Huffman coding: spend your short codes where they'll be used most.

How do we decide the exact codes? We build a binary tree from the bottom up, and a min-heap drives the construction. Start with one tiny node per symbol, keyed by its frequency. Then repeat:

  1. Pop the two lowest-frequency nodes from the min-heap.
  2. Merge them under a new parent node whose frequency is the sum of the two.
  3. Push that parent back into the heap.

Each merge shrinks the heap by one. When a single node remains, it's the root of the finished tree. Read each leaf's path from the root — left = 0, right = 1 — and that string of bits is the symbol's code.

Why greedy-by-lowest-frequency is optimal

The depth of a leaf in the tree is the length of its code. Cost = the sum over all symbols of (frequency × depth). We want to minimize that.

The two rarest symbols should sit deepest — they're used least, so paying a long code for them costs the fewest total bits. By always merging the two smallest frequencies first, we push them down early; every later merge sits above them, so they end up at the greatest depth. Exchange-argument proof sketch: in any optimal tree, you can swap the two least-frequent symbols down to the deepest level without increasing cost, which is exactly what this greedy choice produces. So the local choice is provably globally optimal.

// Build the optimal prefix-code tree with a min-heap of nodes.
#include <queue>
#include <vector>

struct Node {
    int freq; char ch;
    Node *left = nullptr, *right = nullptr;
};
// min-heap: smallest freq on top
struct Cmp { bool operator()(Node* a, Node* b) { return a->freq > b->freq; } };

Node* buildHuffman(std::vector<std::pair<char,int>>& freqs) {
    std::priority_queue<Node*, std::vector<Node*>, Cmp> pq;
    for (auto& [c, f] : freqs) pq.push(new Node{f, c});

    while (pq.size() > 1) {
        Node* a = pq.top(); pq.pop();   // two smallest
        Node* b = pq.top(); pq.pop();
        Node* parent = new Node{a->freq + b->freq, '\0'};
        parent->left = a; parent->right = b;
        pq.push(parent);                // push merged node back
    }
    return pq.top();                    // the root
}

// Walk the tree: left = '0', right = '1' to read each leaf's code.

import java.util.*;

class Node {
    int freq; char ch;
    Node left, right;
    Node(int f, char c) { freq = f; ch = c; }
}

Node buildHuffman(char[] chars, int[] freqs) {
    // min-heap ordered by frequency
    PriorityQueue<Node> pq = new PriorityQueue<>((a, b) -> a.freq - b.freq);
    for (int i = 0; i < chars.length; i++)
        pq.offer(new Node(freqs[i], chars[i]));

    while (pq.size() > 1) {
        Node a = pq.poll();             // two smallest
        Node b = pq.poll();
        Node parent = new Node(a.freq + b.freq, '\0');
        parent.left = a; parent.right = b;
        pq.offer(parent);               // push merged node back
    }
    return pq.poll();                   // the root
}

# Build the optimal prefix-code tree with a min-heap of nodes.
import heapq

def build_huffman(freqs):           # freqs: list of (char, frequency)
    # heap entries: [freq, tiebreak, node]; node = (char, left, right)
    heap = [[f, i, (c, None, None)] for i, (c, f) in enumerate(freqs)]
    heapq.heapify(heap)
    counter = len(heap)

    while len(heap) > 1:
        f1, _, n1 = heapq.heappop(heap)   # two smallest
        f2, _, n2 = heapq.heappop(heap)
        merged = (None, n1, n2)            # internal node
        heapq.heappush(heap, [f1 + f2, counter, merged])
        counter += 1                       # push merged node back

    return heap[0][2]                  # the root node

# Walk the tree: left = '0', right = '1' to read each leaf's code.

Building the tree does n - 1 merges, each a couple of O(log n) heap operations, so the whole thing is O(n log n).

2. Connect Ropes / Minimum Cost to Connect Sticks

You have several ropes of given lengths and want to tie them all into one. Joining two ropes costs the sum of their lengths, and the joined rope keeps that combined length for the next join. Minimize the total cost.

This is Huffman's twin. A rope's length gets re-paid every time it's part of a later join — exactly like a symbol's frequency getting re-paid at every level of depth. So you want short ropes joined first (paid many times while small) and long ropes joined last. A min-heap gives you the two shortest every step.

  • Pop the two smallest lengths.
  • Add their sum to the running cost.
  • Push the sum back into the heap.

Stop when one rope remains. (The Huffman playground below uses the very same loop — watch the running merge-cost; that number is the connect-ropes answer.)

#include <queue>
#include <vector>

int connectRopes(std::vector<int>& ropes) {
    // min-heap of lengths
    std::priority_queue<int, std::vector<int>, std::greater<>> pq(ropes.begin(), ropes.end());
    int cost = 0;
    while (pq.size() > 1) {
        int a = pq.top(); pq.pop();   // two shortest
        int b = pq.top(); pq.pop();
        cost += a + b;                // pay to join
        pq.push(a + b);               // joined rope goes back
    }
    return cost;
}

import java.util.PriorityQueue;

int connectRopes(int[] ropes) {
    PriorityQueue<Integer> pq = new PriorityQueue<>();  // min-heap
    for (int r : ropes) pq.offer(r);
    int cost = 0;
    while (pq.size() > 1) {
        int a = pq.poll();            // two shortest
        int b = pq.poll();
        cost += a + b;                // pay to join
        pq.offer(a + b);              // joined rope goes back
    }
    return cost;
}

import heapq

def connect_ropes(ropes):

        b = heapq.heappop(ropes)
        cost += a + b                 # pay to join
        heapq.heappush(ropes, a + b)  # joined rope goes back
    return cost

3. Task Scheduler — CPU Scheduling by Priority

The CPU must run a list of tasks, but identical tasks need a cooldown of n intervals between runs (think: a core needs to cool off before repeating the same heavy job). Tasks of different kinds can run back-to-back. Minimize total time, including any forced idle slots.

Intuition: the task with the most remaining copies is the bottleneck — it's the one most likely to force idle gaps later. So at each tick, run the most-pending task. That calls for a max-heap keyed by remaining count.

The cooldown is the subtle part. When you run a task, you don't immediately put it back — it must wait. Pop the most-pending task, decrement its count, and place it in a cooldown queue stamped with the time it becomes available again. Each tick, anything whose cooldown has expired goes back into the max-heap. If the heap is empty but the cooldown queue isn't, the CPU sits idle for that tick.

#include <queue>
#include <unordered_map>

int leastInterval(std::vector<char>& tasks, int n) {
    std::unordered_map<char, int> cnt;
    for (char t : tasks) cnt[t]++;

    std::priority_queue<int> pq;          // max-heap of remaining counts
    for (auto& [c, k] : cnt) pq.push(k);

    std::queue<std::pair<int,int>> cool; // {remaining, readyTime}
    int time = 0;
    while (!pq.empty() || !cool.empty()) {
        time++;
        if (!cool.empty() && cool.front().second == time) {
            pq.push(cool.front().first);     // cooldown expired -> re-push
            cool.pop();
        }
        if (!pq.empty()) {
            int remaining = pq.top() - 1; pq.pop();
            if (remaining > 0) cool.push({remaining, time + n + 1});
        }
        // else: heap empty but tasks cooling -> this tick is idle
    }
    return time;
}

import java.util.*;

int leastInterval(char[] tasks, int n) {
    int[] cnt = new int[26];
    for (char t : tasks) cnt[t - 'A']++;

    // max-heap of remaining counts
    PriorityQueue<Integer> pq = new PriorityQueue<>(Collections.reverseOrder());
    for (int c : cnt) if (c > 0) pq.offer(c);

    Queue<int[]> cool = new LinkedList<>(); // {remaining, readyTime}
    int time = 0;
    while (!pq.isEmpty() || !cool.isEmpty()) {
        time++;
        if (!cool.isEmpty() && cool.peek()[1] == time)
            pq.offer(cool.poll()[0]);          // cooldown done -> re-push
        if (!pq.isEmpty()) {
            int remaining = pq.poll() - 1;
            if (remaining > 0) cool.offer(new int[]{remaining, time + n + 1});
        }
    }
    return time;
}

import heapq
from collections import Counter, deque

def least_interval(tasks, n):
    # Python heapq is a min-heap, so store NEGATIVE counts for a max-heap
    heap = [-c for c in Counter(tasks).values()]
    heapq.heapify(heap)
    cool = deque()                    # (remaining_neg, ready_time)
    time = 0
    while heap or cool:
        time += 1
        if cool and cool[0][1] == time:
            heapq.heappush(heap, cool.popleft()[0])  # re-push after cooldown
        if heap:
            remaining = heapq.heappop(heap) + 1   # +1 because counts are negative
            if remaining < 0:
                cool.append((remaining, time + n + 1))
        # else: idle tick
    return time

With a heap this is O(n log A) where A is the number of distinct tasks (at most 26 letters, so effectively O(n)). A pure counting formula can solve it in O(n) without a heap, but the heap version makes the "most-pending-first" intuition explicit.

4. Reorganize String

Rearrange a string so that no two adjacent characters are the same — for example "aab" → "aba". Intuition: the character with the most copies left is the one most in danger of being forced next to itself, so always place it next. A max-heap keyed by count hands you that character.

The trick: after placing a character, you can't reuse it on the very next slot. So hold the just-used character aside for one step (decrement its count first), and only push it back into the heap on the following iteration — once a different character has been placed in between.

It's infeasible when the most frequent character appears more than (n + 1) / 2 times — there simply aren't enough gaps to separate all its copies, and the heap will run dry while that character still has leftovers.

#include <queue>
#include <string>

std::string reorganize(std::string s) {
    int cnt[26] = {0};
    for (char c : s) cnt[c - 'a']++;
    // max-heap of {count, char}
    std::priority_queue<std::pair<int,char>> pq;
    for (int i = 0; i < 26; i++)
        if (cnt[i]) pq.push({cnt[i], char('a' + i)});

    std::string res;
    std::pair<int,char> hold = {0, '\0'};   // just-used char on hold
    while (!pq.empty()) {
        auto [c, ch] = pq.top(); pq.pop();    // most frequent
        res += ch;
        if (hold.first > 0) pq.push(hold);    // release previous after a gap
        hold = {c - 1, ch};                  // hold this one for one step
    }
    return res.size() == s.size() ? res : "";  // "" if impossible
}

import java.util.*;

String reorganize(String s) {
    int[] cnt = new int[26];
    for (char c : s.toCharArray()) cnt[c - 'a']++;
    // max-heap by count
    PriorityQueue<int[]> pq = new PriorityQueue<>((a, b) -> b[0] - a[0]);
    for (int i = 0; i < 26; i++)
        if (cnt[i] > 0) pq.offer(new int[]{cnt[i], i});

    StringBuilder sb = new StringBuilder();
    int[] hold = null;                       // just-used char on hold
    while (!pq.isEmpty()) {
        int[] cur = pq.poll();              // most frequent
        sb.append((char)('a' + cur[1]));
        if (hold != null && hold[0] > 0) pq.offer(hold);
        cur[0]--;
        hold = cur;                       // release on next round
    }
    return sb.length() == s.length() ? sb.toString() : "";
}

import heapq
from collections import Counter

def reorganize(s):
    # max-heap via negative counts
    heap = [(-c, ch) for ch, c in Counter(s).items()]
    heapq.heapify(heap)
    res, hold = [], None                  # hold = just-used char
    while heap:
        c, ch = heapq.heappop(heap)       # most frequent (c is negative)
        res.append(ch)
        if hold and hold[0] < 0:
            heapq.heappush(heap, hold)    # release after a gap
        hold = (c + 1, ch)               # used one copy, hold for one step
    return "".join(res) if len(res) == len(s) else ""

5. Meeting Rooms II

Given a list of meetings with start and end times, how many rooms do you need so none overlap? The answer is the maximum number of meetings happening at the same time.

Intuition: process meetings in order of start time. Keep a min-heap of end times — one entry per room currently in use, with the soonest-to-free room on top. For each new meeting: if the earliest-ending room frees up before this meeting starts, reuse it (pop, then push the new end time). Otherwise, you need a fresh room (just push). The heap's size at any moment is the number of rooms in use; its peak size is the answer.

#include <queue>
#include <vector>
#include <algorithm>

int minMeetingRooms(std::vector<std::vector<int>>& m) {
    std::sort(m.begin(), m.end());        // by start time
    // min-heap of end times (soonest-free room on top)
    std::priority_queue<int, std::vector<int>, std::greater<>> ends;
    for (auto& mt : m) {
        if (!ends.empty() && ends.top() <= mt[0])
            ends.pop();                   // a room freed up -> reuse it
        ends.push(mt[1]);                 // occupy a room with this end time
    }
    return ends.size();                   // peak rooms in use
}

import java.util.*;

int minMeetingRooms(int[][] m) {
    Arrays.sort(m, (a, b) -> a[0] - b[0]);   // by start time
    // min-heap of end times
    PriorityQueue<Integer> ends = new PriorityQueue<>();
    for (int[] mt : m) {
        if (!ends.isEmpty() && ends.peek() <= mt[0])
            ends.poll();                  // a room freed up -> reuse it
        ends.offer(mt[1]);                // occupy a room
    }
    return ends.size();                   // peak rooms in use
}

import heapq

def min_meeting_rooms(meetings):
    meetings.sort()                       # by start time
    ends = []                             # min-heap of end times
    for start, end in meetings:
        if ends and ends[0] <= start:
            heapq.heappop(ends)           # a room freed up -> reuse it
        heapq.heappush(ends, end)         # occupy a room
    return len(ends)                      # peak rooms in use

The Same Engine: Dijkstra, Prim, IPO

Once you see the pattern, you'll spot it everywhere. These three live in full in the Graphs module, but they're driven by the exact same greedy-pop-the-best heap engine:

  • Dijkstra (shortest paths) — a min-heap repeatedly pops the unvisited node with the smallest known distance, relaxes its neighbors, and pushes updated distances back.
  • Prim (minimum spanning tree) — a min-heap repeatedly pops the cheapest edge crossing out of the tree built so far, then pushes the new frontier edges.
  • IPO / Maximize Capital — two heaps: a min-heap by capital reveals which projects you can afford, and a max-heap by profit picks the most profitable affordable project each round.

Greedy + heap only works when the local choice is provably optimal.
  • Huffman and Connect Ropes are provably optimal (exchange argument) — taking the two smallest now never hurts. But many greedy attempts are not optimal; do not assume "grab the best now" works without proving it or recognizing a known pattern (e.g. coin change with arbitrary denominations breaks greedy).
  • Task Scheduler bug: the classic mistake is forgetting to re-push a task into the max-heap after its cooldown expires. Drop that step and counts vanish, the heap empties early, and you report too short a time. Always stamp the cooldown and re-insert when it's up.

Pattern to remember: when a problem repeatedly needs the smallest, largest, most-frequent, cheapest, or soonest item next, a heap is almost always the engine. Decide your greedy rule first (what makes one choice "best"), then let the heap deliver that choice in O(log n) and re-balance itself for the next round.
