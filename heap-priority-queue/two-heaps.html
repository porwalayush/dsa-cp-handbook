Two Heaps: Median of a Data Stream

Imagine numbers arriving one at a time — a live feed of sensor readings, stock ticks, response times. After each new number, someone asks: "What's the median of everything we've seen so far?"

The median is the middle value: half the numbers are below it, half above. The obvious approach — keep a list, sort it, grab the middle — costs O(n log n) every single time you're asked. On a fast stream, that's hopelessly slow.

There's a far better way, and it's one of the most elegant tricks in all of data structures: two heaps.

The Insight: Split the Data in Half

Here's the whole idea. Instead of keeping one sorted list, keep the numbers in two piles, split right down the middle:

  • A max-heap holds the lower half — all the smaller numbers. Its top is the largest of the small ones.
  • A min-heap holds the upper half — all the bigger numbers. Its top is the smallest of the big ones.

Picture the sorted data laid out left to right. The max-heap owns the left side, the min-heap owns the right side, and they meet in the middle. The two tops are exactly the two numbers sitting on either side of that boundary.

And the median? It lives right at that boundary:

  • If both heaps are the same size, the median is the average of the two tops (the count is even — the middle falls between two values).
  • If one heap has one extra element, the median is that heap's top (the count is odd — there's a single true middle).

So once the heaps are arranged correctly, reading the median is just peeking at one or two tops. That's O(1). The clever part is keeping the two piles arranged correctly as new numbers stream in.

The Balancing Rule

Every time a number arrives, we have to maintain two invariants:

  • Invariant 1 (ordering): every element in the max-heap is ≤ every element in the min-heap. The lower half really is lower than the upper half.
  • Invariant 2 (balance): the two heaps differ in size by at most 1. (We'll let the max-heap be the one allowed to hold the extra element.)

A clean recipe that maintains both, no matter what value arrives:

  1. Push the new number x onto the max-heap.
  2. Move the max-heap's top over to the min-heap. (This guarantees Invariant 1: whatever was the biggest of the lower half gets a chance to float up to the upper half.)
  3. If the min-heap is now larger than the max-heap, move the min-heap's top back to the max-heap. (This restores Invariant 2 and keeps the max-heap as the heap that holds any extra element.)

After these three steps the heaps are always balanced within 1, and the max-heap is never smaller than the min-heap. It's a tidy little dance: push, hand one across, hand one back if needed.

The whole trick in one sentence: the median always sits at the boundary between the two halves. Keep the lower half in a max-heap and the upper half in a min-heap, keep their sizes balanced within 1, and the median is always sitting right there at the tops — ready to read in O(1).

The MedianFinder Class

Two methods do all the work: addNum(x) runs the balancing recipe in O(log n), and findMedian() peeks the tops in O(1). The only real subtlety is heap direction, which differs by language — read the comments carefully.

// C++: priority_queue is a MAX-heap by default.
// For the min-heap we pass greater<> to flip the order.
#include <queue>

class MedianFinder {
    std::priority_queue<int> lo;                                  // max-heap: lower half
    std::priority_queue<int, std::vector<int>, std::greater<>> hi; // min-heap: upper half
public:
    void addNum(int x) {
        lo.push(x);              // 1. push onto max-heap
        hi.push(lo.top());       // 2. hand its top across to min-heap
        lo.pop();
        if (hi.size() > lo.size()) { // 3. hand one back if min-heap is bigger
            lo.push(hi.top());
            hi.pop();
        }
    }
    double findMedian() {
        if (lo.size() > hi.size()) return lo.top();        // odd count
        return (lo.top() + hi.top()) / 2.0;               // even count
    }
};

// Java: PriorityQueue is a MIN-heap by default.
// For the max-heap we pass Collections.reverseOrder().
import java.util.*;

class MedianFinder {
    private PriorityQueue<Integer> lo = new PriorityQueue<>(Collections.reverseOrder()); // max-heap
    private PriorityQueue<Integer> hi = new PriorityQueue<>();                       // min-heap

    public void addNum(int x) {
        lo.offer(x);                 // 1. push onto max-heap
        hi.offer(lo.poll());         // 2. hand its top across to min-heap
        if (hi.size() > lo.size()) {  // 3. hand one back if min-heap is bigger
            lo.offer(hi.poll());
        }
    }
    public double findMedian() {
        if (lo.size() > hi.size()) return lo.peek();      // odd count
        return (lo.peek() + hi.peek()) / 2.0;             // even count
    }
}

# Python: heapq is ALWAYS a min-heap.
# To fake a max-heap, store NEGATED values and negate back when reading.
import heapq

class MedianFinder:
    def __init__(self):
        self.lo = []   # max-heap (store negatives): lower half
        self.hi = []   # min-heap: upper half

    def addNum(self, x):
        heapq.heappush(self.lo, -x)                  # 1. push onto max-heap
        heapq.heappush(self.hi, -heapq.heappop(self.lo)) # 2. hand top across
        if len(self.hi) > len(self.lo):              # 3. hand one back if needed
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def findMedian(self):
        if len(self.lo) > len(self.hi):
            return -self.lo[0]                       # odd count
        return (-self.lo[0] + self.hi[0]) / 2.0      # even count
The #1 mistake: getting heap direction wrong. Each language defaults differently — C++ priority_queue is a max-heap (use greater<> for the min-heap); Java PriorityQueue is a min-heap (use reverseOrder() for the max-heap); Python heapq is a min-heap (negate values to fake the max-heap). Get a direction backwards and the tops stop meaning what you think. Two more traps: you must rebalance after every insert — skip a rebalance and the tops no longer represent the true halves; and watch the even/odd median formula — average the two tops only when the sizes are equal.

Complexity at a Glance

addNum does a constant number of heap pushes and pops, each O(log n). findMedian just peeks the tops, which is instant. That's the payoff over re-sorting: queries are free.

OperationTime Complexity
addNum (insert + rebalance)O(log n)
findMedian (peek tops)O(1)
Space (store every number)O(n)

The Harder Cousin: Sliding Window Median

A tougher variant asks for the median of only the last k numbers as a window slides across the stream — so elements must leave the structure, not just arrive. The same two-heaps idea still works, but plain heaps can't remove an arbitrary middle element cheaply. The usual fixes are lazy deletion (mark a leaving element and discard it only once it bubbles to a top) or a balanced multiset / ordered structure that supports O(log k) removal. Same boundary intuition, more bookkeeping.

When to Reach for Two Heaps

The signal

  • "Running or streaming median."
  • Repeated "find the middle" queries as data arrives.
  • You need the boundary between a lower and upper half.

The shape of the fix

  • Max-heap for the lower half, min-heap for the upper half.
  • Rebalance to keep sizes within 1 every insert.
  • Read the median off the tops in O(1).
Pattern to remember: whenever a problem keeps asking for the middle of a growing set of numbers, think "two heaps meeting at the boundary." Balance the halves and the answer is always sitting at the tops.
