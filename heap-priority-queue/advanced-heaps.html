Advanced Heap Variants

The binary heap from the earlier lessons is a workhorse: simple, cache-friendly, and fast enough for almost everything. But it has a few sharp edges. It can't lower the priority of something already inside it without help, and it can't merge two heaps without dumping everything out and rebuilding.

This page is a tour of the variants that fix those edges. Think of it like upgrading from a reliable sedan to specialized vehicles — most days the sedan is the right choice, but sometimes you genuinely need the truck. We'll meet d-ary heaps (a shape tweak that speeds up some operations), indexed heaps (so you can decrease a key in place), and mergeable heaps (so two heaps can become one cheaply).

It's a touch more theory-flavored than the basics, but we'll keep every idea grounded in why you'd reach for it — usually while writing a Dijkstra or a Prim's MST.

D-ary Heaps: Wider, Shorter Trees

A binary heap gives every node up to two children. A d-ary heap simply lets each node have up to d children — say 3, or 4, or 8. Nothing else changes: same array layout, same "parent beats its children" rule.

The intuition is about tree shape. Picture pouring the same number of marbles into two funnels. A narrow funnel (d = 2) stacks them tall; a wide funnel (d = 4) spreads them into a short, bushy pile. More children per node means fewer levels: the height drops from about log₂n to about logdn.

That shorter height is a direct win for any operation that walks upward, because it has fewer levels to climb:

  • insert and decrease-key sift a value up toward the root — now only logdn steps. Faster.
  • extract-min sifts a value down, and at each level it must compare against all d children to find the smallest. So each level costs more: O(d) work over logdn levels = O(d·logdn). Slower.

So d is a tuning knob, a trade between cheaper sift-ups and pricier sift-downs. This matters for Dijkstra, which does a lot of decrease-key relative to extract-min on dense graphs. In practice a 4-ary heap is often measurably faster than a binary one there — and it's friendlier to the CPU cache too, since siblings sit next to each other in memory.

The index math (0-indexed)

The pointer-free array trick still works, you just scale by d instead of 2. For a node at index i:

  • children live at indices d*i + 1, d*i + 2, …, d*i + d
  • parent is at (i - 1) / d (integer / floor division)

When d = 2 these collapse back to the familiar 2i+1, 2i+2, (i-1)/2. Here are the helpers plus sift-up / sift-down parameterized by d.

// A d-ary MIN-heap stored in a flat vector. d children per node.
int D = 4;                                  // try 4-ary for Dijkstra
std::vector<int> h;

int parent(int i) { return (i - 1) / D; }
int child (int i, int k) { return D * i + 1 + k; } // k = 0..D-1

void siftUp(int i) {                       // O(log_d n)
    while (i > 0 && h[i] < h[parent(i)]) {
        std::swap(h[i], h[parent(i)]);
        i = parent(i);
    }
}

void siftDown(int i) {                     // O(d * log_d n)
    int n = h.size();
    while (true) {
        int smallest = i;
        for (int k = 0; k < D; ++k) {       // scan all d children
            int c = child(i, k);
            if (c < n && h[c] < h[smallest]) smallest = c;
        }
        if (smallest == i) break;
        std::swap(h[i], h[smallest]);
        i = smallest;
    }
}

// A d-ary MIN-heap stored in an int array. d children per node.
int D = 4;                                  // try 4-ary for Dijkstra
int[] h; int n;                            // n = current size

int parent(int i) { return (i - 1) / D; }
int child (int i, int k) { return D * i + 1 + k; } // k = 0..D-1

        int t = h[i]; h[i] = h[parent(i)]; h[parent(i)] = t;
        i = parent(i);
    }
}

# A d-ary MIN-heap stored in a list. d children per node.
D = 4                                     # try 4-ary for Dijkstra
h = []

def parent(i): return (i - 1) // D
def child(i, k): return D * i + 1 + k    # k = 0..D-1

def sift_up(i):                          # O(log_d n)
    while i > 0 and h[i] < h[parent(i)]:
        h[i], h[parent(i)] = h[parent(i)], h[i]
        i = parent(i)

def sift_down(i):                        # O(d * log_d n)
    n = len(h)
    while True:
        smallest = i
        for k in range(D):                 # scan all d children
            c = child(i, k)
            if c < n and h[c] < h[smallest]:
                smallest = c
        if smallest == i:
            break
        h[i], h[smallest] = h[smallest], h[i]
        i = smallest
Why 4-ary, not 8-ary? Push d too high and the per-level scan of d children in siftDown starts to dominate. There's a sweet spot — empirically around d = 4 for Dijkstra-style workloads on modern hardware. It's a tuning parameter, not a magic number: measure on your own data if it's a hot path.

Decrease-Key & the Indexed Heap

Here's the operation a plain library priority queue quietly leaves out, and the one graph algorithms desperately want.

Picture Dijkstra. Each vertex sits in the heap keyed by its best-known tentative distance. When you relax an edge and find a cheaper route to a vertex that's already in the heap, you want to lower its key — that's decrease-key. The vertex should bubble up toward the front because it just got more urgent.

The snag: a binary heap can sift a value up in O(log n) only if you can first find it. And a heap has no fast way to locate an arbitrary element — searching is O(n). So decrease-key naively becomes O(n), wrecking the whole algorithm.

Fix #1 — the indexed heap (position map)

The classic fix is to keep a side table: a position map pos[] that records, for each element (e.g. each vertex id), where it currently lives in the heap array. Maintain it on every swap. Now "find vertex v" is an O(1) lookup, and decrease-key is back to a clean O(log n): look up its slot, lower the value, sift up.

This bundle — a heap that also tracks element positions and supports decrease-key — is called an indexed priority queue. The only real cost is discipline: every swap must also update pos[], or the map silently rots.

Fix #2 — lazy deletion (what most people actually do)

The pragmatic alternative skips decrease-key entirely. Instead of editing the entry already in the heap, you just push a new entry with the smaller key and leave the old one to rot. The old entry is now stale.

When you pop, you check: is this the current best for its vertex? If a vertex is already settled, or the popped key is larger than the best you've recorded, the entry is stale — discard it and pop again. Each vertex gets settled exactly once, the first (smallest) time it surfaces.

It's "lazy" because you never clean up the garbage proactively; you just ignore it on the way out. The heap may hold up to O(E) entries instead of O(V), but it needs no position map and works with any off-the-shelf PQ. For competitive programming this is the default.

Indexed heap (decrease-key)

  • Edits the existing entry in place; sifts it up.
  • Heap holds at most one entry per element → O(V) size.
  • Needs a pos[] map, kept in sync on every swap.
  • More code, more bug surface, but tight memory.
  • The "textbook" Dijkstra / Prim implementation.

Lazy deletion (push duplicates)

  • Pushes a new smaller entry; skips stale ones on pop.
  • Heap can grow to O(E) entries (duplicates linger).
  • No position map — works with any standard PQ.
  • Less code; the go-to for contests and most real code.
  • Same O(E log V) asymptotics for Dijkstra.

Code: indexed sift-up vs the lazy-deletion pop

First, the heart of an indexed heap — a sift-up that keeps pos[] in sync (full in all three languages). Then the lazy-deletion pattern, which needs no map at all.

// Indexed binary MIN-heap. heap[] holds element ids; key[id] is its priority.
// pos[id] = where id currently sits in heap[], or -1 if absent.
std::vector<int> heap;          // heap[slot] = id
std::vector<int> pos;           // pos[id]  = slot
std::vector<long> key;          // key[id]  = priority

void swapNodes(int a, int b) {
    std::swap(heap[a], heap[b]);
    pos[heap[a]] = a;            // keep the map in sync!
    pos[heap[b]] = b;
}

void siftUp(int i) {
    while (i > 0) {
        int p = (i - 1) / 2;
        if (key[heap[i]] >= key[heap[p]]) break;
        swapNodes(i, p);
        i = p;
    }
}

// LOWER an element's key, then float it up. O(log n) thanks to pos[].
void decreaseKey(int id, long newKey) {
    if (newKey >= key[id]) return;   // only ever decreases
    key[id] = newKey;
    siftUp(pos[id]);                // O(1) to find it, then sift
}

        i = p

        return                   # only ever decreases

    sift_up(pos[id])             # O(1) to find it, then sift

And the lazy-deletion pattern — the one most Dijkstra solutions actually ship. No pos[], no decrease-key; just push duplicates and skip stale pops (Python heapq shown; the same shape works in C++/Java):

import heapq
# Dijkstra with LAZY deletion. No decrease-key, no pos map.
def dijkstra(adj, src):
    dist = [float('inf')] * len(adj)
    dist[src] = 0
    pq = [(0, src)]                 # (key, vertex)
    while pq:
        d, u = heapq.heappop(pq)
        if d > dist[u]:             # STALE: a better d was already settled
            continue                 # skip it and pop again
        for v, w in adj[u]:
            if dist[u] + w < dist[v]:
                dist[v] = dist[u] + w
                heapq.heappush(pq, (dist[v], v))  # push duplicate
    return dist

// Dijkstra with LAZY deletion. min-heap via greater<>. No pos map.
using P = std::pair<long, int>;   // (key, vertex)
std::vector<long> dist(N, LLONG_MAX);
std::priority_queue<P, std::vector<P>, std::greater<P>> pq;
dist[src] = 0;
pq.push({0, src});
while (!pq.empty()) {
    auto [d, u] = pq.top(); pq.pop();
    if (d > dist[u]) continue;        // STALE: skip and pop again
    for (auto [v, w] : adj[u]) {
        if (dist[u] + w < dist[v]) {
            dist[v] = dist[u] + w;
            pq.push({dist[v], v});       // push duplicate
        }
    }
}

Your standard-library priority queue does NOT do this for you. C++'s std::priority_queue, Java's PriorityQueue, and Python's heapq support push and pop only — no efficient decrease-key and no efficient meld. Don't assume they do. If you need decrease-key, build an indexed heap with a pos[] map, or (far more often) just use lazy deletion. If you need to merge two queues cheaply, reach for a mergeable heap below.

Mergeable (Meldable) Heaps

Sometimes you have two heaps and want to fuse them into one. A binary heap is terrible at this: there's no clever shortcut, so you concatenate the arrays and re-heapify — O(n). A whole family of structures exists precisely to make meld (also called union) cheap, usually O(log n) or even O(1).

They earn it by giving up the rigid "complete tree in an array" shape. These heaps are pointer-based trees with looser structure, which is exactly what lets two of them snap together quickly.

* = amortized. Binomial insert is O(1) amortized, O(log n) worst case. Pairing-heap bounds are amortized and partly conjectured to be even better in practice. Leftist gives worst-case O(log n); skew gives amortized O(log n) with simpler code.

Leftist & skew heaps: meld is the whole game

In these two, every operation is built on one primitive: meld. Insert? Meld a single-node heap in. Extract-min? Remove the root, then meld its two subtrees. So if meld is O(log n), everything is.

The meld itself is a tidy recursion. To merge two min-heaps, take the one with the smaller root as the new top; its left child stays, and you recursively meld its right subtree with the other whole heap. The trick that keeps things balanced lives in what you do after that recursive merge:

  • Leftist heap: track each node's "null-path length" and swap children so the shorter spine is always on the right. This guarantees the right spine is O(log n) — a worst-case bound.
  • Skew heap: skip the bookkeeping entirely — just unconditionally swap children after every merge. No stored metadata, dead simple, and it still achieves O(log n) amortized. It's the leftist heap's lazy cousin.

Because a skew heap is so short to write, it's the one worth seeing in full. The entire data structure is essentially this one function.

// Skew heap: a min-heap where meld() swaps children each merge.
// insert / extractMin are both just thin wrappers over meld().
struct Node { int val; Node *l = nullptr, *r = nullptr; };

Node* meld(Node* a, Node* b) {
    if (!a) return b;
    if (!b) return a;
    if (b->val < a->val) std::swap(a, b);   // a = smaller root
    a->r = meld(a->r, b);                   // recurse on right
    std::swap(a->l, a->r);                  // the skew step!
    return a;
}

Node* insert(Node* h, int v) { return meld(h, new Node{v}); }
Node* extractMin(Node* h) { return meld(h->l, h->r); }  // h->val is the min

// Skew heap: a min-heap where meld() swaps children each merge.
// insert / extractMin are both just thin wrappers over meld().
static class Node {
    int val; Node l, r;
    Node(int v) { val = v; }
}

static Node meld(Node a, Node b) {
    if (a == null) return b;
    if (b == null) return a;
    if (b.val < a.val) { Node t = a; a = b; b = t; }  // a = smaller
    a.r = meld(a.r, b);                    // recurse on right
    Node t = a.l; a.l = a.r; a.r = t;      // the skew step!
    return a;
}

static Node insert(Node h, int v) { return meld(h, new Node(v)); }
static Node extractMin(Node h) { return meld(h.l, h.r); } // h.val is min

# Skew heap: a min-heap where meld() swaps children each merge.
# insert / extract_min are both just thin wrappers over meld().
class Node:
    def __init__(self, val):
        self.val = val
        self.l = self.r = None

def meld(a, b):
    if a is None: return b
    if b is None: return a
    if b.val < a.val:
        a, b = b, a                  # a = smaller root
    a.r = meld(a.r, b)               # recurse on right
    a.l, a.r = a.r, a.l              # the skew step!
    return a

def insert(h, v): return meld(h, Node(v))
def extract_min(h): return meld(h.l, h.r)   # h.val is the min

Binomial & Fibonacci: the theory end

The remaining two are worth knowing about more than implementing.

A binomial heap is a collection of specially-shaped trees (binomial trees), merged the way you add binary numbers — carrying trees of equal size together. That gives O(log n) meld and a neat O(1)-amortized insert.

The Fibonacci heap is the theoretical champion: O(1) amortized insert, decrease-key, and meld, with O(log n) amortized extract-min. That O(1) decrease-key is what famously drops Dijkstra to O(E + V log V). But it's notorious for large constant factors and fiddly lazy-restructuring bookkeeping — so it's rarely worth implementing. A well-tuned binary or 4-ary heap usually beats it in wall-clock time.

Which one do I actually use?
  • For competitive programming, a plain binary heap with lazy deletion handles Dijkstra and Prim's just fine — reach for it first.
  • Tuning a hot Dijkstra on dense graphs? Try a 4-ary heap.
  • Genuinely need fast meld? A pairing heap or skew heap is simple and fast in practice.
  • Fibonacci heaps are mostly of theoretical interest — beautiful bounds, but the constants make them lose in real benchmarks.
