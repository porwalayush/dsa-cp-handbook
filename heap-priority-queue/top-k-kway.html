Top-K & K-Way Merge

Two patterns cover a surprising share of every heap problem you'll meet. Once you internalize them, a whole category of interview questions collapses into "oh, that's just a heap of size K" or "oh, that's a K-way merge."

The big idea underneath both: a heap lets you keep just the handful of items you actually care about — and cheaply swap out the worst one as better candidates arrive. You never have to sort the whole pile.

The Core Trick: A Heap of Size K

Suppose you have a million numbers and you only want the 10 largest. The lazy instinct is to sort all million (O(n log n)) and read off the top 10. That works, but it's wasteful — you sorted 999,990 numbers you didn't care about.

Here's the better way. Keep a small container that holds only K items. Walk through the numbers one at a time. For each new number, drop it in; if the container now holds more than K, throw away the worst one. When you reach the end, the K items left standing are exactly the K best.

Because the container never grows past K, each insert-and-evict is O(log k), not O(log n). Over n items that's O(n log k) — and you never hold more than K things in memory. For "top 10 out of a million," that's a heap of size 10 versus sorting a million.

1. Kth Largest Element

"Find the Kth largest number in this array." The trick above gives it to you directly: keep a heap of size K, and the answer falls out the bottom.

But here's the part that trips everyone up the first time. To find the K largest elements, you use a MIN-heap. Read that again — a min-heap to track the largest. Why?

The min-heap's top is its smallest element. If you keep a min-heap of size K, its top is the smallest of the K biggest you've seen so far — that is, the weakest survivor, the one on the bubble. When a new number arrives, you ask: "Is it bigger than the weakest survivor?" If yes, the weakest gets evicted (you pop the top) and the newcomer joins. If no, the newcomer can't make the cut and you skip it. After processing everything, the heap holds the K largest — and its top is the Kth largest, since the top is the smallest of those K.

// Kth largest: keep a MIN-heap of size k.
#include <queue>
#include <vector>

int findKthLargest(std::vector<int>& nums, int k) {
    // greater<> makes this a MIN-heap (smallest on top)
    std::priority_queue<int, std::vector<int>, std::greater<>> pq;
    for (int x : nums) {
        pq.push(x);
        if ((int)pq.size() > k) pq.pop();  // evict the weakest survivor
    }
    return pq.top();  // smallest of the k largest = Kth largest
}

// Java: PriorityQueue is a MIN-heap by default — exactly what we want.
int findKthLargest(int[] nums, int k) {
    PriorityQueue<Integer> pq = new PriorityQueue<>();
    for (int x : nums) {
        pq.offer(x);
        if (pq.size() > k) pq.poll();  // drop the smallest survivor
    }
    return pq.peek();  // the Kth largest
}

# Python: heapq is a MIN-heap. heappushpop pushes then pops the min in one step.
import heapq

def find_kth_largest(nums, k):
    heap = []
    for x in nums:
        heapq.heappush(heap, x)
        if len(heap) > k:
            heapq.heappop(heap)      # evict weakest survivor
    return heap[0]               # min of the k largest = Kth largest

# Shortcut: heapq.nlargest(k, nums)[-1] does the same thing.

2. Top K Frequent Elements

"Which K values show up most often?" Same machinery, one extra step in front. First count how many times each value appears (a hash map: value → count). Now you have a pile of (value, count) pairs, and you want the K with the biggest counts — which is exactly the Kth-largest pattern again, just keyed by count instead of by the value itself.

Keep a size-K min-heap ordered by frequency. Push each (count, value) pair; whenever the heap exceeds K, pop the least-frequent. The K most frequent survive. That's O(n) to count plus O(n log k) to filter.

// Top K frequent: count, then a size-k MIN-heap keyed by frequency.
std::vector<int> topKFrequent(std::vector<int>& nums, int k) {
    std::unordered_map<int, int> cnt;
    for (int x : nums) cnt[x]++;

    // min-heap of {frequency, value}; smallest frequency on top
    std::priority_queue<std::pair<int,int>,
        std::vector<std::pair<int,int>>, std::greater<>> pq;
    for (auto& [val, f] : cnt) {
        pq.push({f, val});
        if ((int)pq.size() > k) pq.pop();  // drop least frequent
    }
    std::vector<int> res;
    while (!pq.empty()) { res.push_back(pq.top().second); pq.pop(); }
    return res;
}

// Java: count into a map, then a size-k min-heap by frequency.
int[] topKFrequent(int[] nums, int k) {
    Map<Integer,Integer> cnt = new HashMap<>();
    for (int x : nums) cnt.merge(x, 1, Integer::sum);

    // order by frequency ascending -> min-frequency on top
    PriorityQueue<Integer> pq =
        new PriorityQueue<>((a, b) -> cnt.get(a) - cnt.get(b));
    for (int val : cnt.keySet()) {
        pq.offer(val);
        if (pq.size() > k) pq.poll();  // drop least frequent
    }
    int[] res = new int[k];
    for (int i = 0; i < k; i++) res[i] = pq.poll();
    return res;
}

# Python: Counter + a size-k min-heap of (freq, value).
import heapq
from collections import Counter

def top_k_frequent(nums, k):
    cnt = Counter(nums)
    heap = []
    for val, f in cnt.items():
        heapq.heappush(heap, (f, val))     # keyed by frequency
        if len(heap) > k:
            heapq.heappop(heap)            # drop least frequent
    return [val for f, val in heap]

# Shortcut: [v for v, _ in Counter(nums).most_common(k)]
Even faster — bucket sort, O(n). For Top K frequent, frequencies can't exceed n, so you can drop counts into n+1 buckets indexed by frequency, then read buckets from the high end until you've collected K values. No heap, no log factor — just O(n). The heap version is still the go-to in interviews because it generalizes to any "top K by some key" problem.

3. K Closest Points to Origin

"Return the K points nearest to (0, 0)." Notice this is the opposite direction from Kth-largest: now we want the K smallest distances. So we flip the heap.

Use a size-K MAX-heap keyed by distance. The max-heap's top is the farthest point among the K we're keeping — the weakest survivor in a "smaller is better" contest. When a new point arrives that's closer than the current farthest, evict the farthest and admit the newcomer. The K closest remain. (We can compare squared distance and skip the square root — it doesn't change the ordering.)

// K closest: size-k MAX-heap keyed by squared distance.
std::vector<std::vector<int>> kClosest(
        std::vector<std::vector<int>>& pts, int k) {
    // default priority_queue is a MAX-heap: largest distance on top
    std::priority_queue<std::pair<int, int>> pq;  // {dist2, index}
    for (int i = 0; i < (int)pts.size(); i++) {
        int d = pts[i][0]*pts[i][0] + pts[i][1]*pts[i][1];
        pq.push({d, i});
        if ((int)pq.size() > k) pq.pop();  // evict the farthest
    }
    std::vector<std::vector<int>> res;
    while (!pq.empty()) { res.push_back(pts[pq.top().second]); pq.pop(); }
    return res;
}

// Java: max-heap via reverse comparator on squared distance.
int[][] kClosest(int[][] pts, int k) {
    PriorityQueue<int[]> pq = new PriorityQueue<>(
        (a, b) -> (b[0]*b[0]+b[1]*b[1]) - (a[0]*a[0]+a[1]*a[1]));
    for (int[] p : pts) {
        pq.offer(p);
        if (pq.size() > k) pq.poll();  // evict the farthest
    }
    return pq.toArray(new int[0][]);
}

# Python heapq is a MIN-heap, so negate the distance to fake a MAX-heap.
import heapq

def k_closest(points, k):
    heap = []  # entries: (-dist2, x, y)
    for x, y in points:
        heapq.heappush(heap, (-(x*x + y*y), x, y))
        if len(heap) > k:
            heapq.heappop(heap)            # pops most-negative = farthest
    return [[x, y] for _, x, y in heap]

# Shortcut: heapq.nsmallest(k, points, key=lambda p: p[0]**2 + p[1]**2)

4. Merge K Sorted Lists (K-Way Merge)

"You have K already-sorted lists. Merge them into one sorted list." If K were 2, you'd just zip the two together with two pointers. With K lists, the question each step is: of all the current front-of-list items, which is the smallest? A min-heap answers that in O(log k).

Here's the recipe. Seed the heap with the first element of each list, and tag each entry with which list it came from (and its index in that list). Then loop: pop the smallest, append it to the output, and push the next element from that same list — the successor of the one you just took. The heap always holds at most K items (one candidate per list), so you never look at more than K things to find the next smallest.

This is exactly how external merge sort stitches together sorted chunks too big to fit in memory: each chunk is a "list," and the heap picks the global next-smallest one record at a time. Total work is O(N log k) for N elements across K lists.

// K-way merge of sorted arrays. Heap entry: {value, listIdx, elemIdx}.
std::vector<int> mergeKSorted(std::vector<std::vector<int>>& lists) {
    using Node = std::tuple<int, int, int>;  // (val, list, idx)
    std::priority_queue<Node, std::vector<Node>, std::greater<>> pq;

    for (int i = 0; i < (int)lists.size(); i++)
        if (!lists[i].empty()) pq.push({lists[i][0], i, 0});

    std::vector<int> out;
    while (!pq.empty()) {
        auto [val, li, idx] = pq.top(); pq.pop();
        out.push_back(val);
        if (idx + 1 < (int)lists[li].size())   // successor from SAME list
            pq.push({lists[li][idx + 1], li, idx + 1});
    }
    return out;
}

// K-way merge. Heap entry: {value, listIdx, elemIdx}.
int[] mergeKSorted(int[][] lists) {
    PriorityQueue<int[]> pq =
        new PriorityQueue<>((a, b) -> a[0] - b[0]);  // by value
    int total = 0;
    for (int i = 0; i < lists.length; i++) {
        total += lists[i].length;
        if (lists[i].length > 0) pq.offer(new int[]{lists[i][0], i, 0});
    }
    int[] out = new int[total];
    int w = 0;
    while (!pq.isEmpty()) {
        int[] n = pq.poll();          // n = {val, listIdx, elemIdx}
        out[w++] = n[0];
        int li = n[1], idx = n[2];
        if (idx + 1 < lists[li].length)  // successor from SAME list
            pq.offer(new int[]{lists[li][idx + 1], li, idx + 1});
    }
    return out;
}

# K-way merge. heapq compares tuples lexicographically -> sorts by value.
import heapq

def merge_k_sorted(lists):
    heap = []
    for i, lst in enumerate(lists):
        if lst:
            heapq.heappush(heap, (lst[0], i, 0))  # (val, listIdx, idx)

    out = []
    while heap:
        val, li, idx = heapq.heappop(heap)          # smallest head
        out.append(val)
        if idx + 1 < len(lists[li]):              # successor from SAME list
            heapq.heappush(heap, (lists[li][idx + 1], li, idx + 1))
    return out

Complexity at a Glance

Every pattern on this page rides the same engine — a heap that never grows past K — so the costs rhyme. The win over a full sort is the log k instead of log n factor.

ProblemTimeHeap Space
Kth largest elementO(n log k)O(k)
Top K frequent elementsO(n log k)O(k)
K closest points to originO(n log k)O(k)
Merge K sorted lists (K-way)O(N log k)O(k)

N is the total number of elements across all K lists. A full sort would be O(n log n) — the heap-of-size-K trick shaves the log factor down to log k whenever k << n.

The one idea to remember: a heap of size K hands you the K best in O(n log k) instead of O(n log n) — and you never hold more than K items at once. Pick the heap direction by what you want to throw away:
  • Want the K largest? Use a MIN-heap — its top is the weakest survivor, the first to evict when something better arrives.
  • Want the K smallest / closest? Use a MAX-heap — its top is the farthest/biggest, the one to drop.
The two classic traps:
  • Inverting the heap direction. It feels backwards, and it's the #1 confusion: use a MIN-heap to keep the K LARGEST, and a MAX-heap to keep the K SMALLEST. The heap's top is always the candidate you're ready to evict, not the answer's headline.
  • K-way merge: push the successor from the SAME list. After you pop a value, the next candidate must come from the same list you just pulled from — not the next list. Mix that up and your output stops being sorted.
