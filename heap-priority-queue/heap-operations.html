Insert, Extract & Heapify

You already know the shape: a heap is a complete binary tree packed into a flat array, where every parent is smaller than (or equal to) its children. We'll use a min-heap throughout, so the smallest value always sits at the root.

But a data structure is only as good as the things you can do with it. There are really just two moving parts here — sift-up and sift-down — and once you've seen an element bubble up or sink down, every heap operation becomes obvious. Let's watch them happen.

Quick reminder of the index math (0-indexed) — we lean on it constantly below:

  • parent(i) = (i - 1) / 2 (integer division)
  • left(i) = 2i + 1  ·  right(i) = 2i + 2

Insert: drop it at the end, then bubble up

Imagine a new person joining a line where everyone is ordered shortest-at-front. The fair, lazy thing to do is put them at the very back first — then let them step forward past anyone taller than them, one swap at a time, until they're in the right spot.

That's exactly insert. We put the new value in the next free array slot (the end). This keeps the tree complete — no gaps. But it probably violates the heap property, so we sift it up: while the new element is smaller than its parent, swap them. Stop when it's ≥ its parent, or when it reaches the root.

Because the tree's height is log n, the worst case is one swap per level — O(log n).

Why "the end"? The end of the array is always the next available leaf in level-order — the only spot that keeps the tree complete. Sift-up then fixes the ordering without ever breaking the shape.

Extract-min: the root is the answer — now refill the hole

The whole reason we built a min-heap is to grab the smallest element fast. It's sitting right there at the root. So extract-min returns the root — but now there's a hole at the top.

The trick: take the last element in the array and move it up to the root. (Why the last? Because removing the last slot is the only deletion that keeps the tree complete.) That last element is almost certainly too big to be the minimum, so we sift it down: while it's bigger than its smaller child, swap it with that smaller child. Stop when it's ≤ both children, or when it becomes a leaf.

Again, one swap per level at most — O(log n).

Peek: just look at the root — O(1)

No bubbling, no swapping. The minimum is always at index 0. Reading it is a single array lookup, so peek is O(1). (This is exactly why heaps power priority queues — the "best" item is always one glance away.)

Two classic sift-down bugs:
  • For a min-heap you must compare against the smaller child. If you blindly swap with the left child, you can push a value below something even smaller and silently break the heap. Always pick min(left, right) first, then compare.
  • On extract, move the last element to the root — don't shift every element down by one. Shifting is O(n) and destroys the heap shape. The move-last-then-sift-down trick is what keeps it O(log n).

Build-heap: turn a raw array into a heap in O(n)

Say you're handed an unordered array and told "make this a heap." The naive way is to insert all n elements one at a time — that's n sift-ups, costing O(n log n).

There's a better way. Notice that leaves are already valid heaps (a single node trivially satisfies the heap property). So we only need to fix the internal nodes. We start at the last non-leaf node — index n/2 - 1 — and call sift-down on each node walking backward down to index 0. By the time we reach a node, both of its subtrees are already heaps, so a single sift-down fixes it.

The pleasant surprise: this is O(n), not O(n log n). Here's the intuition. Most of the nodes live near the bottom of the tree, and a node near the bottom can only sift down a tiny distance. Only the handful of nodes near the top can sift far. When you add up "number of nodes at each height × how far each can fall," the sum converges to a constant times n — so bottom-up heapify beats inserting n times. Same goal, strictly better cost.

The Code

Everything below is built from two helpers — siftUp and siftDown. insert, extractMin, and buildHeap are just thin wrappers around them. All 0-indexed.

#include <vector>
#include <utility>   // std::swap
std::vector<int> h;          // our min-heap, 0-indexed

// bubble element at i up while it's smaller than its parent
void siftUp(int i) {
    while (i > 0) {
        int parent = (i - 1) / 2;
        if (h[i] >= h[parent]) break;   // in place -> stop
        std::swap(h[i], h[parent]);
        i = parent;
    }
}

// sink element at i down while it's bigger than its SMALLER child
void siftDown(int i) {
    int n = h.size();
    while (true) {
        int left = 2*i + 1, right = 2*i + 2, smallest = i;
        if (left  < n && h[left]  < h[smallest]) smallest = left;
        if (right < n && h[right] < h[smallest]) smallest = right;
        if (smallest == i) break;          // already ≤ both children
        std::swap(h[i], h[smallest]);
        i = smallest;
    }
}

void insert(int x) {
    h.push_back(x);              // drop at the end (keeps tree complete)
    siftUp(h.size() - 1);        // bubble it up to its spot
}

int extractMin() {
    int root = h[0];               // the answer
    h[0] = h.back();              // move LAST element to the root
    h.pop_back();
    if (!h.empty()) siftDown(0);  // sink it down
    return root;
}

int peek() { return h[0]; }      // O(1)

void buildHeap() {              // O(n) bottom-up heapify
    for (int i = h.size()/2 - 1; i >= 0; --i)
        siftDown(i);             // last non-leaf down to root
}

int[] h;          // backing array
int n;            // current heap size (number of valid elements)

void swap(int a, int b) { int t = h[a]; h[a] = h[b]; h[b] = t; }

// bubble element at i up while it's smaller than its parent
void siftUp(int i) {
    while (i > 0) {
        int parent = (i - 1) / 2;
        if (h[i] >= h[parent]) break;   // in place -> stop
        swap(i, parent);
        i = parent;
    }
}

// sink element at i down while it's bigger than its SMALLER child
void siftDown(int i) {
    while (true) {
        int left = 2*i + 1, right = 2*i + 2, smallest = i;
        if (left  < n && h[left]  < h[smallest]) smallest = left;
        if (right < n && h[right] < h[smallest]) smallest = right;
        if (smallest == i) break;          // already ≤ both children
        swap(i, smallest);
        i = smallest;
    }
}

void insert(int x) {
    h[n++] = x;                  // drop at the end (keeps tree complete)
    siftUp(n - 1);              // bubble it up to its spot
}

int extractMin() {
    int root = h[0];               // the answer
    h[0] = h[--n];               // move LAST element to the root
    if (n > 0) siftDown(0);       // sink it down
    return root;
}

int peek() { return h[0]; }      // O(1)

void buildHeap() {              // O(n) bottom-up heapify
    for (int i = n/2 - 1; i >= 0; i--)
        siftDown(i);             // last non-leaf down to root
}

h = []          # our min-heap, a plain list (0-indexed)

# bubble element at i up while it's smaller than its parent
def sift_up(i):
    while i > 0:
        parent = (i - 1) // 2
        if h[i] >= h[parent]:          # in place -> stop
            break
        h[i], h[parent] = h[parent], h[i]
        i = parent

# sink element at i down while it's bigger than its SMALLER child
def sift_down(i):
    n = len(h)
    while True:
        left, right, smallest = 2*i + 1, 2*i + 2, i
        if left  < n and h[left]  < h[smallest]: smallest = left
        if right < n and h[right] < h[smallest]: smallest = right
        if smallest == i:             # already ≤ both children
            break
        h[i], h[smallest] = h[smallest], h[i]
        i = smallest

def insert(x):
    h.append(x)                   # drop at the end (keeps tree complete)
    sift_up(len(h) - 1)           # bubble it up to its spot

def extract_min():
    root = h[0]                    # the answer
    last = h.pop()                # take LAST element off
    if h:                         # if anything remains...
        h[0] = last                # ...move it to the root
        sift_down(0)              # and sink it down
    return root

def peek():
    return h[0]                   # O(1)

def build_heap():                # O(n) bottom-up heapify
    for i in range(len(h)//2 - 1, -1, -1):
        sift_down(i)              # last non-leaf down to root
You rarely write this by hand. Every language ships a heap: C++ std::priority_queue (and std::make_heap for O(n) build), Java PriorityQueue, Python heapq (with heapq.heapify for the O(n) build). Learn the mechanics here, reach for the library in practice — the next lesson covers exactly that.

Complexity at a Glance

Every operation that reorders the heap costs at most one swap per level, and the tree is only log n tall. Peek touches nothing, and build-heap is the surprise winner.

OperationTime Complexity
insert (append + sift-up)O(log n)
extract-min (remove root + sift-down)O(log n)
peek (read root)O(1)
heapify-down (sift-down one node)O(log n)
build-heap (heapify whole array)O(n)

Build-heap is O(n) even though it calls an O(log n) routine n/2 times — because most nodes are near the bottom and barely move. The naive "insert n times" approach is O(n log n).

Pattern to remember: insert = append, then sift up. Extract = save the root, move the last element to the top, then sift down. Two helpers, and you've got the entire heap.
