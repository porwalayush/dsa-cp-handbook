Heap Sort

Imagine a pile of numbered tickets and one simple rule: always pluck out the biggest ticket, set it aside in its final spot, and repeat. Do that until the pile is empty and — without ever comparing the small tickets to each other directly — you've sorted everything.

That's the whole idea of heap sort. The trick is making "find the biggest" fast. A plain pile would force you to scan everything each round. So instead we arrange the numbers into a max-heap — a structure whose root is always the largest element. Now grabbing the biggest is instant, and fixing the heap afterward only costs O(log n).

Heap sort runs entirely in place inside the original array. The same array is part heap, part sorted result — and the boundary between them slides as we go.

The Big Picture: Two Phases

Heap sort happens in two clean phases. First we organize the chaos into a heap; then we drain the heap one max at a time into the sorted tail.

Phase 1 — Build a max-heap (O(n))

We turn the raw array into a max-heap, in place. The clever part: we don't insert items one by one. Instead we start at the last non-leaf node (index n/2 - 1) and sift-down each node, walking backwards to index 0.

Why start there? Every node past n/2 - 1 is a leaf — a heap of size one, already valid. By fixing parents from the bottom up, each sift-down can assume its children are already proper heaps. This bottom-up build costs only O(n), not O(n log n).

Phase 2 — Sort by draining the heap

Now the largest element sits at the root, index 0. Here's the loop, for end = n-1 down to 1:

  • Swap a[0] with a[end]. The biggest element is now parked at the back — its final sorted position.
  • Shrink the heap by one (pretend a[end] no longer exists; it's "sorted" now).
  • Sift down the new root within the shrunken heap, so the next-biggest bubbles up to index 0.

Each pass locks one more element into place at the back. The sorted region grows from right to left until the whole array is in order — ascending.

The key insight: the array does double duty. The front is a shrinking max-heap; the back is a growing, already-sorted tail. The biggest value in the heap always swaps into the slot just before the sorted tail — so the tail stays sorted, and we never need extra memory.

Why O(n log n)?

Add up the work in the two phases:

  • Build phase: O(n) — a tighter bound than it looks, because most nodes are near the bottom and sift down only a level or two.
  • Sort phase: n - 1 extractions, each doing one sift-down of cost O(log n) → O(n log n).

The sort phase dominates, so the total is O(n log n). And crucially, this holds in the best, average, AND worst case — there's no input that can make heap sort go quadratic.

Heap Sort in Code

Here's the full algorithm in each language: a siftDown helper plus the two-phase heapSort. All max-heap, 0-indexed, sorting ascending. Notice that siftDown takes the current heap size as a parameter — that's how the sort phase keeps the locked tail untouched.

// Max-heap heap sort, 0-indexed, sorts ascending.
// Sift a[i] down within a heap of size n.
void siftDown(int a[], int n, int i) {
    while (true) {
        int largest = i;
        int l = 2 * i + 1, r = 2 * i + 2;
        if (l < n && a[l] > a[largest]) largest = l;
        if (r < n && a[r] > a[largest]) largest = r;
        if (largest == i) break;     // heap property restored
        std::swap(a[i], a[largest]);
        i = largest;                  // follow the swapped child down
    }
}

void heapSort(int a[], int n) {
    // Phase 1: build max-heap from last non-leaf down to root.
    for (int i = n / 2 - 1; i >= 0; --i)
        siftDown(a, n, i);

    // Phase 2: repeatedly move the max to the back.
    for (int end = n - 1; end > 0; --end) {
        std::swap(a[0], a[end]);     // max -> its final spot
        siftDown(a, end, 0);       // sift within SHRUNK heap (size = end)
    }
}

// Max-heap heap sort, 0-indexed, sorts ascending.
static void siftDown(int[] a, int n, int i) {
    while (true) {
        int largest = i;
        int l = 2 * i + 1, r = 2 * i + 2;
        if (l < n && a[l] > a[largest]) largest = l;
        if (r < n && a[r] > a[largest]) largest = r;
        if (largest == i) break;
        int t = a[i]; a[i] = a[largest]; a[largest] = t;
        i = largest;
    }
}

static void heapSort(int[] a) {
    int n = a.length;
    // Phase 1: build max-heap.
    for (int i = n / 2 - 1; i >= 0; i--)
        siftDown(a, n, i);

    // Phase 2: drain the heap into the sorted tail.
    for (int end = n - 1; end > 0; end--) {
        int t = a[0]; a[0] = a[end]; a[end] = t;
        siftDown(a, end, 0);   // heap size shrinks to 'end'
    }
}

# Max-heap heap sort, 0-indexed, sorts ascending.
def sift_down(a, n, i):
    while True:
        largest = i
        l, r = 2 * i + 1, 2 * i + 2
        if l < n and a[l] > a[largest]: largest = l
        if r < n and a[r] > a[largest]: largest = r
        if largest == i: break
        a[i], a[largest] = a[largest], a[i]
        i = largest

def heap_sort(a):
    n = len(a)
    # Phase 1: build max-heap from last non-leaf to root.
    for i in range(n // 2 - 1, -1, -1):
        sift_down(a, n, i)

    # Phase 2: move the max to the back, then re-sift.
    for end in range(n - 1, 0, -1):
        a[0], a[end] = a[end], a[0]
        sift_down(a, end, 0)   # sift within shrunk heap of size 'end'
Two gotchas. First, heap sort is NOT stable — equal elements can be reordered, because swapping the root to the back jumbles their original order. Second, in the sort phase you MUST sift down within the shrinking heap size (end), not the full array length. If you pass n after swapping, the sift-down reaches back into the sorted tail, drags those elements back into the heap, and ruins everything.

Heap Sort vs. Its Rivals

Heap sort, merge sort, and quicksort are the three classic O(n log n) comparison sorts. They make different trade-offs around memory, stability, and worst-case behavior.

Heap Sort

  • Time: O(n log n) all cases
  • Space: O(1) in-place
  • Not stable
  • Guaranteed worst case, but poor cache locality → often slower in practice

Merge Sort

  • Time: O(n log n) all cases
  • Space: O(n) extra
  • Stable
  • Great for linked lists & external sorting

Quick Sort

  • Time: avg O(n log n), worst O(n²)
  • Space: O(log n) in-place
  • Not stable
  • Usually fastest in practice (cache-friendly)

The headline: heap sort is the only one of the three that gives you both a guaranteed O(n log n) worst case and O(1) extra space. Quicksort can degrade to O(n²); merge sort needs O(n) scratch memory. Heap sort gives up the constant-factor speed crown but never blows up.
