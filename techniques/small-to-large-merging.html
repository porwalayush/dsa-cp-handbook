Small-to-Large Merging

When you repeatedly merge sets together, always pour the smaller set into the larger one. This one rule drops the total cost from O(n²) to O(n log n) — because each element can only move into a set at least twice its size, and that can happen at most log n times.

The Key Insight

  • If you merge small → large, an element moves only when its new home is ≥ 2× the size of its old one.
  • An element's set size can double at most log₂ n times before reaching n.
  • So across the whole process, each element is moved ≤ log n times → O(n log n) total moves.

Why Direction Matters

   Merging set A (size a) and B (size b):

   WRONG: always move B into A → could move a huge B every time → O(n²)

   RIGHT: move the SMALLER into the LARGER
          → each moved element lands in a set ≥ 2× its old size
          → ≤ log n moves per element → O(n log n)

   The merged result is the same; only the WORK differs.

Code: Small-to-Large (set merging)

// Merge set b into set a, keeping the larger as the target
void mergeInto(set<int>& a, set<int>& b) {
    if (a.size() < b.size()) swap(a, b);  // ensure a is larger
    for (int x : b) a.insert(x);          // pour small into large
    b.clear();
}

// DSU on tree (small-to-large) sketch:
// dfs returns the set of values in a subtree; merge children
set<int> dfs(int u, int parent) {
    set<int> cur = { value[u] };
    for (int v : adj[u]) if (v != parent) {
        set<int> child = dfs(v, u);
        if (cur.size() < child.size()) swap(cur, child);
        for (int x : child) cur.insert(x);  // small into large
    }
    // answer[u] = something about cur (e.g. cur.size())
    return cur;
}
// Merge set b into set a, keeping the larger as the target
Set<Integer> mergeInto(Set<Integer> a, Set<Integer> b) {
    if (a.size() < b.size()) { Set<Integer> t = a; a = b; b = t; }
    a.addAll(b);   // pour small into large
    return a;
}

// DSU on tree (small-to-large) sketch
Set<Integer> dfs(int u, int parent) {
    Set<Integer> cur = new HashSet<>();
    cur.add(value[u]);
    for (int v : adj[u]) if (v != parent) {
        Set<Integer> child = dfs(v, u);
        if (cur.size() < child.size()) { Set<Integer> t = cur; cur = child; child = t; }
        cur.addAll(child);  // small into large
    }
    return cur;
}
# Merge set b into set a, keeping the larger as the target
def merge_into(a, b):
    if len(a) < len(b):
        a, b = b, a          # ensure a is larger
    a |= b                   # pour small into large
    return a

# DSU on tree (small-to-large)
def dfs(u, parent):
    cur = {value[u]}
    for v in adj[u]:
        if v != parent:
            child = dfs(v, u)
            if len(cur) < len(child):
                cur, child = child, cur
            cur |= child     # small into large
    # answer[u] = f(cur)
    return cur

The Famous Use: DSU on Tree

   "For every node, answer a query about its whole subtree."
   (e.g. count distinct colors, most frequent value, etc.)

   Naive: recompute each subtree → O(n²).

   Small-to-large: each node returns its subtree's multiset;
   a parent merges children by pouring small into large.
   → O(n log n) or O(n log² n) with a balanced structure.

Where It Shows Up

DSU on Tree — subtree queries (distinct count, dominant color).

Merging components — when union-find isn't enough and you need the actual element sets.

Auxiliary maps/sets per node — frequency maps merged up a tree.

Offline connectivity — combining groups while answering queries.

Common Pitfalls

1. Merging large into small: the whole speedup is lost — always check sizes and swap.

2. Swapping references, not copying: in C++ use swap (O(1)); never deep-copy the big set.

3. Forgetting to clear the emptied set if you reuse storage.

4. Heavy per-element cost: if each insert is O(log n) (a balanced set), total is O(n log² n) — fine, but know your factor.
