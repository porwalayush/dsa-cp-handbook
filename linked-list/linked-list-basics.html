Linked Lists: The Basics

Imagine a treasure hunt. You start with one clue. That clue holds a value and tells you where to find the next clue. Follow it, and the next clue points to the one after that — until a clue says "the hunt ends here." That's a linked list: a chain of little boxes, where each box knows only about the next box.

Unlike an array — where everything sits in one continuous block of memory and you can jump straight to slot #5 — a linked list scatters its boxes anywhere in memory and stitches them together with pointers (arrows). The trade-off is the whole point of the topic, so let's build the picture first.

What is a Node?

A linked list is made of nodes. Every node carries two things:

  • data — the value it stores (a number, a string, anything).
  • next — a pointer/reference to the following node, or null if it's the last one.

The list itself is just a single reference called the head — the first node. Lose the head and you lose the whole list, so we guard it carefully.

🎮 Build Your Own List
Add and remove nodes and watch the chain rewire. The red node is the head; the chain always ends at null.

Here's the node defined in code. Notice how small it is — that's the entire building block:

// One link in the chain
struct Node {
    int data;
    Node* next;
    Node(int d) : data(d), next(nullptr) {}
};

// Build:  7 -> 3 -> 9 -> null
Node* head = new Node(7);
head->next = new Node(3);
head->next->next = new Node(9);

// One link in the chain
class Node {
    int data;
    Node next;
    Node(int d) { data = d; next = null; }
}

// Build:  7 -> 3 -> 9 -> null
Node head = new Node(7);
head.next = new Node(3);
head.next.next = new Node(9);

# One link in the chain
class Node:
    def __init__(self, data):
        self.data = data
        self.next = None

# Build:  7 -> 3 -> 9 -> None
head = Node(7)
head.next = Node(3)
head.next.next = Node(9)

Linked List vs. Array

This is the question interviewers love. The honest answer: it depends on what you do most.

Array wins at…

  • Random access — arr[5] is instant, O(1).
  • Cache locality — contiguous memory is fast to scan.
  • Less memory — no pointer per element.

Linked list wins at…

  • Insert/delete at the front — O(1), no shifting.
  • Growing — no resize/copy, just allocate a node.
  • Splicing — rewire a few pointers, move nothing.

The golden rule: use a linked list when you do lots of insertions/deletions at the ends or middle and rarely need to jump to a random index. Otherwise, an array (or dynamic array / vector) is usually faster and simpler.
The #1 beginner mistake: overwriting head or a node's next before saving the rest of the chain. Once you drop the only pointer to a node, everything after it is lost (and leaks memory in C++). Whenever you rewire, ask: "do I still have a handle on every node I need?"
