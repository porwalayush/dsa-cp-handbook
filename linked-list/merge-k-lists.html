Merge K Sorted Lists

Imagine k separate checkout lines at a store, and within each line the people are already ordered shortest-to-tallest. You want one single line, sorted by height. You don't re-sort everyone — you just keep glancing at the front person of every line and wave the shortest of them forward, over and over. That's the whole problem.

Formally: you're given k linked lists, each already sorted ascending. Merge them into one sorted list. This builds directly on the two-list zipper merge from Merging & Sorting Lists — here we just have more lists to juggle.

1. The problem & why the naive way is slow

The obvious idea: take the zipper merge you already know, and fold the lists in one at a time. Merge list 1 with list 2, then merge that result with list 3, then with list 4, and so on.

It works, but think about how much each node gets re-touched. Say every list has the same length and there are N nodes total. When you merge in the last list, you walk over the entire accumulated result again — almost all N nodes. The round before that, nearly N again. You repeat this k times, so the early nodes get copied over and over.

That gives O(N × k) time. With many lists that k factor hurts. The two better approaches below both knock it down to O(N log k) by being smart about which comparison to do next.

// Naive: fold lists in one at a time. O(N * k).
Node* mergeKNaive(vector<Node*>& lists) {
    Node* acc = nullptr;
    for (Node* l : lists)
        acc = mergeTwo(acc, l);   // re-walks acc every time
    return acc;
}

// Naive: fold lists in one at a time. O(N * k).
Node mergeKNaive(Node[] lists) {
    Node acc = null;
    for (Node l : lists)
        acc = mergeTwo(acc, l);   // re-walks acc every time
    return acc;
}

# Naive: fold lists in one at a time. O(N * k).
def merge_k_naive(lists):
    acc = None
    for l in lists:
        acc = merge_two(acc, l)   # re-walks acc every time
    return acc
Where does the waste come from? A node that lives in the first list gets copied during merge #1, then walked again in merge #2, again in merge #3... up to k times. The two approaches below make sure each node is touched only about log k times.

2. Min-heap (priority queue) approach

Back to the checkout-line analogy: you don't need to look at everybody. You only ever care about the front person of each line — the global shortest must be one of them. A min-heap is exactly the gadget that holds a handful of items and instantly hands you the smallest.

The recipe:

  1. Push the head node of each list into the heap (so it holds at most k nodes — one candidate per list).
  2. Pop the smallest. Append it to your result list.
  3. If that node had a next, push next — that list's new front candidate.
  4. Repeat until the heap is empty.

Each of the N nodes is pushed and popped once. Heap operations cost log(size), and the heap never holds more than k items, so each op is log k. Total: O(N log k) time, O(k) extra space for the heap.

The one subtlety in each language is telling the heap how to compare nodes — we want the smallest data on top (a min-heap):

// Comparator: priority_queue is a MAX-heap by default,
// so we invert ( > ) to get the smallest on top.
struct Cmp {
    bool operator()(Node* a, Node* b) const { return a->data > b->data; }
};

Node* mergeKLists(vector<Node*>& lists) {
    priority_queue<Node*, vector<Node*>, Cmp> pq;
    for (Node* l : lists) if (l) pq.push(l);   // seed heads

    Node dummy(0); Node* tail = &dummy;
    while (!pq.empty()) {
        Node* smallest = pq.top(); pq.pop();      // global front
        tail->next = smallest; tail = tail->next;
        if (smallest->next) pq.push(smallest->next);  // pull next
    }
    tail->next = nullptr;
    return dummy.next;
}

// PriorityQueue is a MIN-heap; pass a comparator on data.
Node mergeKLists(Node[] lists) {
    PriorityQueue<Node> pq =
        new PriorityQueue<>((a, b) -> a.data - b.data);
    for (Node l : lists) if (l != null) pq.offer(l);  // seed heads

    Node dummy = new Node(0), tail = dummy;
    while (!pq.isEmpty()) {
        Node smallest = pq.poll();        // global front
        tail.next = smallest; tail = tail.next;
        if (smallest.next != null)
            pq.offer(smallest.next);       // pull next
    }
    return dummy.next;
}

import heapq

# heapq is a min-heap but can't compare Node objects directly.
# Push tuples (data, tiebreaker, node) so ties never compare nodes.
def merge_k_lists(lists):
    heap = []
    for idx, node in enumerate(lists):
        if node:
            heapq.heappush(heap, (node.data, idx, node))  # seed heads

    dummy = Node(0); tail = dummy
    while heap:
        _, idx, smallest = heapq.heappop(heap)   # global front
        tail.next = smallest; tail = tail.next
        if smallest.next:
            heapq.heappush(heap, (smallest.next.data, idx, smallest.next))
    return dummy.next
Python gotcha: heapq compares whole tuples, and if two nodes have equal data it would try to compare the Node objects themselves — which raises TypeError. Add a unique tiebreaker (here the list index) as the middle element so the comparison never reaches the node.

3. Divide-and-conquer pairwise merging

Different intuition, same payoff. Instead of one heap, run a tournament bracket: pair the lists up and merge each pair, which halves how many lists you have. Then pair up the survivors and merge again. Keep halving until one list remains.

This is exactly merge sort's merge phase, applied at the list level rather than the element level. Each "round" passes over all N nodes once with the trusty two-list zipper, and because you halve k each round there are only log k rounds. So N work per round × log k rounds = O(N log k).

It all rests on the two-sorted-list merge (restated from the merge-sort page) — the mergeTwo zipper with a dummy node:

// The zipper merge (same as the two-list version).
Node* mergeTwo(Node* a, Node* b) {
    Node dummy(0); Node* tail = &dummy;
    while (a && b) {
        if (a->data <= b->data) { tail->next = a; a = a->next; }
        else                   { tail->next = b; b = b->next; }
        tail = tail->next;
    }
    tail->next = a ? a : b;
    return dummy.next;
}

// Pairwise: halve the list count each pass. O(N log k).
Node* mergeKLists(vector<Node*>& lists) {
    if (lists.empty()) return nullptr;
    int n = lists.size();
    while (n > 1) {
        int k = (n + 1) / 2;             // pairs this round
        for (int i = 0; i < n / 2; i++)
            lists[i] = mergeTwo(lists[i], lists[i + k]);
        n = k;
    }
    return lists[0];
}

// The zipper merge (same as the two-list version).
Node mergeTwo(Node a, Node b) {
    Node dummy = new Node(0), tail = dummy;
    while (a != null && b != null) {
        if (a.data <= b.data) { tail.next = a; a = a.next; }
        else                 { tail.next = b; b = b.next; }
        tail = tail.next;
    }
    tail.next = (a != null) ? a : b;
    return dummy.next;
}

// Pairwise: halve the list count each pass. O(N log k).
Node mergeKLists(Node[] lists) {
    if (lists.length == 0) return null;
    int n = lists.length;
    while (n > 1) {
        int k = (n + 1) / 2;             // pairs this round
        for (int i = 0; i < n / 2; i++)
            lists[i] = mergeTwo(lists[i], lists[i + k]);
        n = k;
    }
    return lists[0];
}

# The zipper merge (same as the two-list version).
def merge_two(a, b):
    dummy = Node(0); tail = dummy
    while a and b:
        if a.data <= b.data:
            tail.next, a = a, a.next
        else:
            tail.next, b = b, b.next
        tail = tail.next
    tail.next = a if a else b
    return dummy.next

# Pairwise: halve the list count each pass. O(N log k).
def merge_k_lists(lists):
    if not lists:
        return None
    n = len(lists)
    while n > 1:
        k = (n + 1) // 2              # pairs this round
        for i in range(n // 2):
            lists[i] = merge_two(lists[i], lists[i + k])
        n = k
    return lists[0]
Why halving is the key. The naive fold merges list[0] with everything, so the front nodes are re-walked k times. Pairwise merging guarantees every node participates in only log k merges — same total nodes per round, but far fewer rounds.

4. Comparing the two approaches

Both hit O(N log k) time. They differ in space and feel:

AspectMin-heapDivide & conquer
TimeO(N log k)O(N log k)
Extra spaceO(k) for the heapO(1) iterative / O(log k) recursive
Streaming?Yes — emits nodes one at a timeNo — needs all lists up front
Mental model"Always pop the global smallest front""Tournament bracket of zipper merges"
Best whenLists arrive as a stream; k is largeAll lists known; want minimal extra memory

Rule of thumb: reach for the heap when lists are produced lazily or k is huge, and for divide-and-conquer when you already hold all the lists and want the leanest memory footprint. Pick whichever you can write bug-free under pressure — the asymptotics are identical.
