Classic Linked List Problems

These are the four interview problems that show up again and again. None of them need fancy data structures — they all come down to walking the chain cleverly with one or two pointers. Once you internalize the tricks here (slow/fast, reverse-in-place, length offset), most "hard" list problems start to look familiar.

Every example uses a simple node: a val (or data) field and a next pointer. We never index into the list — we only follow next. That constraint is exactly what makes these tricks necessary.

1. Palindrome Linked List

A palindrome reads the same forwards and backwards, like the word "racecar." For an array that's trivial — compare the ends and walk inward. But a singly linked list only goes forward, so you can't read it backwards directly.

The trick is to fold the list in half. Imagine folding a strip of paper at the middle so the second half lies on top of the first, reversed. Now the last element sits over the first, the second-to-last over the second, and so on. If every overlapping pair matches, it's a palindrome.

Concretely, four steps:

  1. Find the middle with the slow/fast pointer trick: fast moves two steps for every one of slow, so when fast reaches the end, slow sits at the middle.
  2. Reverse the second half in place (the classic three-pointer reversal).
  3. Compare the first half and the reversed second half value by value.
  4. (Optional) Restore the list by reversing the second half back, so you don't leave the input mutated.

bool isPalindrome(Node* head) {
    if (!head || !head->next) return true;
    // 1) find middle: slow ends at start of 2nd half
    Node *slow = head, *fast = head;
    while (fast && fast->next) {
        slow = slow->next;
        fast = fast->next->next;
    }
    // 2) reverse the second half
    Node *prev = nullptr, *curr = slow;
    while (curr) {
        Node* nxt = curr->next;
        curr->next = prev;
        prev = curr;
        curr = nxt;
    }
    // 3) compare halves (prev = head of reversed 2nd half)
    Node *p1 = head, *p2 = prev;
    bool ok = true;
    while (p2) {
        if (p1->val != p2->val) { ok = false; break; }
        p1 = p1->next;
        p2 = p2->next;
    }
    // 4) optional: reverse 2nd half back to restore list
    return ok;
}

boolean isPalindrome(Node head) {
    if (head == null || head.next == null) return true;
    // 1) find middle
    Node slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    // 2) reverse the second half
    Node prev = null, curr = slow;
    while (curr != null) {
        Node nxt = curr.next;
        curr.next = prev;
        prev = curr;
        curr = nxt;
    }
    // 3) compare halves
    Node p1 = head, p2 = prev;
    while (p2 != null) {
        if (p1.val != p2.val) return false;
        p1 = p1.next;
        p2 = p2.next;
    }
    // 4) optional: reverse back to restore
    return true;
}

def is_palindrome(head):
    if not head or not head.next:
        return True
    # 1) find middle
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
    # 2) reverse the second half
    prev, curr = None, slow
    while curr:
        nxt = curr.next
        curr.next = prev
        prev = curr
        curr = nxt
    # 3) compare halves
    p1, p2 = head, prev
    while p2:
        if p1.val != p2.val:
            return False
        p1 = p1.next
        p2 = p2.next
    # 4) optional: reverse back to restore
    return True
Why slow/fast lands on the middle: for every single step slow takes, fast takes two — so slow covers exactly half the distance. With an odd length the middle node belongs to neither half; the comparison loop stops when the (shorter) reversed half runs out, so that lone center is harmlessly ignored.

2. Remove Duplicates

"Remove duplicates" has two very different difficulties depending on one word: is the list sorted or not?

Sorted list — skip equal neighbors

If the list is sorted, every duplicate sits right next to its twin (like a sorted attendance sheet where repeated names are adjacent). So you just walk the list and whenever curr.val == curr.next.val, unlink the neighbor by pointing curr.next one node further. One pass, no extra memory.

// Sorted: remove duplicates, O(n) time, O(1) space
Node* dedupSorted(Node* head) {
    Node* curr = head;
    while (curr && curr->next) {
        if (curr->val == curr->next->val) {
            Node* dup = curr->next;
            curr->next = dup->next;  // skip the twin
            delete dup;
        } else {
            curr = curr->next;
        }
    }
    return head;
}

// Sorted: remove duplicates, O(n) time, O(1) space
Node dedupSorted(Node head) {
    Node curr = head;

            curr.next = curr.next.next;  // skip the twin
        } else {
            curr = curr.next;
        }
    }
    return head;
}

# Sorted: remove duplicates, O(n) time, O(1) space
def dedup_sorted(head):
    curr = head
    while curr and curr.next:
        if curr.val == curr.next.val:
            curr.next = curr.next.next   # skip the twin
        else:
            curr = curr.next
    return head

Unsorted list — remember what you've seen

If the list isn't sorted, twins can be anywhere, so adjacency no longer helps. Instead you keep a hash set of values you've already seen — like ticking names off a guest list as people arrive. When you reach a value already in the set, unlink it; otherwise add it and move on. This costs O(n) extra space but stays O(n) time.

// Unsorted: O(n) time, O(n) space via hash set
Node* dedupUnsorted(Node* head) {
    if (!head) return head;
    unordered_set<int> seen{head->val};
    Node* prev = head;
    while (prev->next) {
        if (seen.count(prev->next->val)) {
            Node* dup = prev->next;
            prev->next = dup->next;   // unlink seen value
            delete dup;
        } else {
            seen.insert(prev->next->val);
            prev = prev->next;
        }
    }
    return head;
}

// Unsorted: O(n) time, O(n) space via hash set
Node dedupUnsorted(Node head) {
    if (head == null) return head;
    Set<Integer> seen = new HashSet<>();
    seen.add(head.val);
    Node prev = head;
    while (prev.next != null) {
        if (seen.contains(prev.next.val)) {
            prev.next = prev.next.next;  // unlink seen value
        } else {
            seen.add(prev.next.val);
            prev = prev.next;
        }
    }
    return head;
}

# Unsorted: O(n) time, O(n) space via hash set
def dedup_unsorted(head):
    if not head:
        return head
    seen = {head.val}
    prev = head
    while prev.next:
        if prev.next.val in seen:
            prev.next = prev.next.next   # unlink seen value
        else:
            seen.add(prev.next.val)
            prev = prev.next
    return head
No hash set allowed? You can dedupe an unsorted list in O(1) space with two nested pointers: for each node, scan the rest of the list and unlink any later copies. That drops memory to O(1) but costs O(n²) time — a classic space-vs-time trade-off.

3. Add Two Numbers

Picture two numbers written as linked lists, one digit per node, with the least significant digit first. So 2 → 4 → 3 is the number 342, and 5 → 6 → 4 is 465. Storing digits in reverse is actually convenient: it lets you add the way you do by hand — start from the ones column and carry left.

Walk both lists together. At each step add the two digits plus any incoming carry, write down sum % 10 as the new digit, and pass sum / 10 as the carry into the next column. Don't forget a final carry (e.g. 9 + 9 needs one more node).

// Digits stored least-significant-first.
Node* addTwoNumbers(Node* a, Node* b) {
    Node dummy(0);
    Node* tail = &dummy;
    int carry = 0;
    while (a || b || carry) {
        int sum = carry;
        if (a) { sum += a->val; a = a->next; }
        if (b) { sum += b->val; b = b->next; }
        carry = sum / 10;
        tail->next = new Node(sum % 10);
        tail = tail->next;
    }
    return dummy.next;
}

// Digits stored least-significant-first.
Node addTwoNumbers(Node a, Node b) {
    Node dummy = new Node(0);
    Node tail = dummy;
    int carry = 0;
    while (a != null || b != null || carry != 0) {
        int sum = carry;
        if (a != null) { sum += a.val; a = a.next; }
        if (b != null) { sum += b.val; b = b.next; }
        carry = sum / 10;
        tail.next = new Node(sum % 10);
        tail = tail.next;
    }
    return dummy.next;
}

# Digits stored least-significant-first.
def add_two_numbers(a, b):
    dummy = Node(0)
    tail = dummy
    carry = 0
    while a or b or carry:
        s = carry
        if a:
            s += a.val; a = a.next
        if b:
            s += b.val; b = b.next
        carry, digit = divmod(s, 10)
        tail.next = Node(digit)
        tail = tail.next
    return dummy.next
Forward-order variant: if the most significant digit comes first (e.g. 3 → 4 → 2 = 342) you can't add left-to-right because carries flow the other way. The cleanest fix is to push both lists onto two stacks, then pop them together — popping gives you the digits least-significant-first, so you're back to the easy case. Build the result by prepending each new digit to the front. (Reversing both inputs, adding, then reversing the result works too.)

4. Intersection of Two Lists

Two linked lists "intersect" when, at some point, they merge and share the rest of their nodes — like two roads that join and continue as one. We want the node where they first become the same. The catch: they may have different lengths before the join, so the i-th nodes don't line up.

The length-difference trick

Measure both lengths, then advance the pointer on the longer list by the difference. Now both pointers are equidistant from the end, so walking them in lockstep makes them arrive at the shared node at the same time.

The "switch heads" trick

An even slicker version needs no length math. Walk pA down list A and pB down list B one step at a time. Whenever a pointer hits the end, redirect it to the other list's head. Each pointer then travels lenA + lenB nodes total, so after at most that many steps they line up — meeting at the intersection node, or both reaching null together if the lists never intersect.

// "Switch heads" — O(m+n) time, O(1) space.
Node* getIntersection(Node* a, Node* b) {
    if (!a || !b) return nullptr;
    Node *pA = a, *pB = b;
    while (pA != pB) {
        pA = pA ? pA->next : b;   // hop to other head at end
        pB = pB ? pB->next : a;
    }
    return pA;  // meeting point, or nullptr if none
}

// "Switch heads" — O(m+n) time, O(1) space.
Node getIntersection(Node a, Node b) {
    if (a == null || b == null) return null;
    Node pA = a, pB = b;
    while (pA != pB) {
        pA = (pA == null) ? b : pA.next;
        pB = (pB == null) ? a : pB.next;
    }
    return pA;  // meeting point, or null if none
}

# "Switch heads" — O(m+n) time, O(1) space.
def get_intersection(a, b):
    if not a or not b:
        return None
    pA, pB = a, b
    while pA is not pB:
        pA = b if pA is None else pA.next
        pB = a if pB is None else pB.next
    return pA  # meeting point, or None if none
Compare nodes, not values. Intersection means the two lists share the same physical node (same address / reference), not just equal values. Use == on references (C++/Java) or is (Python) — comparing .val would give false matches.
