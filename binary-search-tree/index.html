BST Problem Patterns

How to recognize the right approach

1

Need sorted order?

Inorder traversal of a BST is sorted. Validation, kth element, closest value, and range sums all lean on this.

2

Searching by value?

Compare with the node and go left or right — halving the search space each step gives O(h). This is the core BST advantage.

3

Worried about skew?

A naive BST degrades to O(n) on sorted input. Use a self-balancing BST (AVL / Red-Black) to guarantee O(log n).

4

Rank / kth queries?

Augment nodes with subtree sizes. Rank, select, and order-statistics become O(h) using the size field.

5

Floor / ceil / neighbors?

Track the best candidate while walking down. Successor/predecessor, floor, and ceil are all single root-to-leaf descents.

6

Dynamic ordered set?

When you need insert/delete + ordered queries, a balanced BST (TreeMap/std::set) beats sorting on every update.
