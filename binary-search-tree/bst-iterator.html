BST Iterator (Inorder On-Demand)

Think of a BST Iterator as a “next” button bolted onto a binary search tree. Each press hands you the next value in sorted order — smallest first — and nothing more. You never ask for the whole sorted list; you just keep pressing the button and the next number appears.

The clever part is how it remembers where to resume. Behind the button sits a small stack that bookmarks exactly which nodes are still waiting their turn. Press the button, it pops the next answer, updates its bookmarks, and pauses again. This makes each press cost O(1) on average and the bookmarks take only O(h) memory, where h is the tree height — far cheaper than dumping every value into a list up front.

1. The Problem

Design a class that, given the root of a BST, supports two operations:

  • next() — return the next smallest key (the next value in sorted/inorder order).
  • hasNext() — return true if there is still a value left to hand out.

We want next() / hasNext() to be fast (O(1) amortized) and to use only O(h) memory. The whole point is to avoid precomputing and storing the entire sorted sequence: that wastes memory on big trees and does work the caller may never use if they stop early (say, after grabbing the 3rd-smallest value).

2. Naive Approach: Flatten First

The simplest idea: in the constructor, do a full inorder traversal, dump every key into a list, and have next() read from the front while hasNext() checks whether the list is empty.

class BSTIteratorNaive:
    def __init__(self, root):
        self.vals = []
        self._inorder(root)        # O(n) time AND O(n) space up front
        self.i = 0

    def _inorder(self, node):
        if node:
            self._inorder(node.left)
            self.vals.append(node.val)
            self._inorder(node.right)

    def next(self):
        v = self.vals[self.i]; self.i += 1; return v

    def hasNext(self):
        return self.i < len(self.vals)

This works, and each next() really is O(1) — but it pays O(n) memory no matter how far you iterate. If you only need the first few keys, or the tree is enormous, the rest of that list is wasted. The controlled approach below defers the work so memory stays O(h) and subtrees you never reach are never even visited.

3. Controlled (Stack-Based) Inorder Iterator

Here is the intuition before the rules: a normal recursive inorder traversal uses the program’s call stack to remember “which node do I come back to after finishing this left subtree?” We are going to keep that exact same stack by hand, so we can pause and resume the traversal between button presses.

The invariant: the stack always holds the chain of nodes whose key has not yet been returned but is next in line, with the smallest pending key on top.

  • Constructor: push the entire left spine from the root (root, root.left, root.left.left, …). The deepest left node lands on top — it is the global minimum, the very first answer.
  • next(): pop the top node cur (its key is the answer). Before returning, push the left spine of cur.right — because everything in the right subtree comes after cur but before cur’s ancestors.
  • hasNext(): just check whether the stack is non-empty.
Why the left spine? In sorted order you always want the smallest thing you can still reach. From any node, the smallest reachable key is found by walking left as far as possible. Pushing the left spine puts that smallest node on top, with each ancestor stacked beneath it — waiting for their turn once everything to their left is done.

struct TreeNode {
    int val;
    TreeNode *left, *right;
    TreeNode(int v) : val(v), left(nullptr), right(nullptr) {}
};

class BSTIterator {
    stack<TreeNode*> st;

    // Push the whole left spine starting at node.
    void pushLeft(TreeNode* node) {
        while (node) {
            st.push(node);
            node = node->left;
        }
    }

public:
    BSTIterator(TreeNode* root) {
        pushLeft(root);                // O(h) work, O(h) stack
    }

    // Return next-smallest key.
    int next() {
        TreeNode* cur = st.top();
        st.pop();
        pushLeft(cur->right);          // expose right subtree's left spine
        return cur->val;
    }

    bool hasNext() {
        return !st.empty();
    }
};

class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int v) { val = v; }
}

class BSTIterator {
    private Deque<TreeNode> stack = new ArrayDeque<>();

    public BSTIterator(TreeNode root) {
        pushLeft(root);                // O(h) work, O(h) stack
    }

    // Push the whole left spine starting at node.
    private void pushLeft(TreeNode node) {
        while (node != null) {
            stack.push(node);
            node = node.left;
        }
    }

    // Return next-smallest key.
    public int next() {
        TreeNode cur = stack.pop();
        pushLeft(cur.right);           // expose right subtree's left spine
        return cur.val;
    }

    public boolean hasNext() {
        return !stack.isEmpty();
    }
}

class TreeNode:
    def __init__(self, val=0, left=None, right=None):
        self.val = val
        self.left = left
        self.right = right

class BSTIterator:
    def __init__(self, root):
        self.stack = []
        self._push_left(root)          # O(h) work, O(h) stack

    def _push_left(self, node):
        # Push the whole left spine starting at node.
        while node:
            self.stack.append(node)
            node = node.left

    def next(self):
        # Return next-smallest key.
        cur = self.stack.pop()
        self._push_left(cur.right)     # expose right subtree's left spine
        return cur.val

    def hasNext(self):
        return len(self.stack) > 0

4. Amortized Analysis

A single next() can cost O(h) in the worst case — if the popped node has a long left spine in its right subtree, one pushLeft shoves many nodes on at once. So per-call worst case is not O(1).

But zoom out to the whole traversal. Every node is pushed exactly once and popped exactly once across the entire run of next() calls. With n nodes that is O(n) total work spread over n calls.

Total stack operations over a complete iteration:
    pushes = n   (each node pushed once)
    pops   = n   (each node popped once)
    -------------------------------------
    total  = 2n  operations for n calls to next()

Amortized cost per next() = 2n / n = O(1)
Single-call worst case    = O(h)   (one big pushLeft)

This is the classic aggregate / amortized argument: individual calls vary, but the average over the whole run is constant. hasNext() is always O(1) since it just checks emptiness.

5. Variant: Bidirectional Iterator (next + prev)

Some problems want both next() (successor) and prev() (predecessor) from a movable cursor. A clean design keeps two stacks: a forward stack holding the left spine ahead of the cursor (as above), and a backward stack holding the right spine behind it (built symmetrically with a pushRight helper that walks down right children). Calling next() advances the cursor and records the visited node so prev() can rewind; prev() mirrors the logic using the right spine.

An alternative, simpler when the tree is static, is to materialize the sorted array once and keep an integer index that next() increments and prev() decrements — trading O(n) memory for trivial O(1) worst-case moves in both directions. Choose based on whether memory or worst-case latency matters more.

6. Use Cases

Merge / compare two BSTs in sorted order

Run two iterators in lockstep and always advance the one with the smaller current key — exactly like merging two sorted lists, but each stream uses only O(h) memory.

def merge_sorted(rootA, rootB):
    # Yield all keys of two BSTs in global sorted order.
    a, b = BSTIterator(rootA), BSTIterator(rootB)
    va = a.next() if a.hasNext() else None
    vb = b.next() if b.hasNext() else None
    out = []
    while va is not None and vb is not None:
        if va <= vb:
            out.append(va); va = a.next() if a.hasNext() else None
        else:
            out.append(vb); vb = b.next() if b.hasNext() else None
    while va is not None:
        out.append(va); va = a.next() if a.hasNext() else None
    while vb is not None:
        out.append(vb); vb = b.next() if b.hasNext() else None
    return out

k-th smallest element

Just call next() exactly k times. Because work is deferred, you only touch the part of the tree needed to surface the first k keys — ideal when k << n.

def kth_smallest(root, k):
    it = BSTIterator(root)
    for _ in range(k - 1):
        it.next()                  # skip the first k-1 keys
    return it.next()              # the k-th smallest

7. Complexity

For a balanced tree h = O(log n), so peak memory is logarithmic; for a degenerate (linked-list-shaped) tree h = O(n), matching the naive approach in the worst case but still never doing wasted work past where you stop.

Mental model — the stack is the inorder frontier:
  • At any moment the stack holds exactly the ancestors whose keys are still pending — nodes we have descended into (via left edges) but not yet returned.
  • The top is always the smallest unreturned key; the rest are its ancestors waiting their turn.
  • Popping a node and pushing its right child’s left spine advances this frontier by one step — precisely the “in-progress” state of a paused inorder traversal.
