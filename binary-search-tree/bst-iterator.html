BST Iterator (Inorder On-Demand)

A BST Iterator walks a binary search tree in sorted order, producing one key at a time through next() and hasNext(). The trick is to do this lazily — without materializing the entire inorder traversal up front — achieving O(1) amortized time per next() and only O(h) extra memory, where h is the tree height.

1. The Problem

Design a class that, given the root of a BST, supports two operations:

  • next() — return the next smallest key in the BST (the next value in inorder order).
  • hasNext() — return true if a next key exists, false otherwise.

The goal is O(1) amortized time for next() / hasNext() and O(h) memory. Crucially, we want to avoid precomputing and storing the whole inorder sequence: that wastes memory on large trees and does work the caller may never need if they stop iterating early.

Inorder of a BST = keys in ascending sorted order.

           20
          /  \
         9    25
        / \
       5   12
          /  \
         11   14

Inorder sequence: 5, 9, 11, 12, 14, 20, 25
                  ^ next() must yield these in this order.

2. Naive Approach: Flatten First

The simplest idea: run a full inorder traversal in the constructor, store every key in a list (or queue), and have next() pop from the front and hasNext() check whether the list is exhausted.

class BSTIteratorNaive:
    def __init__(self, root):
        self.vals = []
        self._inorder(root)        # O(n) time AND O(n) space up front
        self.i = 0

    def _inorder(self, node):
        if node:
            self._inorder(node.left)
            self.vals.append(node.val)
            self._inorder(node.right)

    def next(self):
        v = self.vals[self.i]; self.i += 1; return v

    def hasNext(self):
        return self.i < len(self.vals)

This is correct and each next() is genuinely O(1), but it pays O(n) memory regardless of how far you iterate. If you only need the first few keys (e.g. the 3rd smallest), or the tree is huge, you waste the rest. The controlled approach below defers work so memory stays O(h) and unused subtrees are never visited until needed.

3. Controlled (Stack-Based) Inorder Iterator

We simulate the recursion stack of an inorder traversal explicitly. The invariant: the stack holds the chain of nodes whose key has not yet been returned but is next in line, with the smallest pending key on top.

  • Constructor: push the entire left spine from the root (root, root.left, root.left.left, ...). The deepest left node sits on top — it is the global minimum.
  • next(): pop the top node cur (its key is the answer). Before returning, push the left spine of cur.right, because everything in the right subtree comes after cur but before cur's ancestors.
  • hasNext(): return whether the stack is non-empty.
Helper: pushLeft(node) -> while node: stack.push(node); node = node.left

Tree:
           20
          /  \
         9    25
        / \
       5   12
          /  \
         11   14

Constructor: pushLeft(20)  -> push 20, 9, 5
Stack (top -> bottom): [5, 9, 20]

Walk through the next() calls. Top of stack is on the left:

Stack state (top is leftmost element):

start            : [5, 9, 20]
next() -> 5      : pop 5; 5.right = null -> nothing pushed
                   [9, 20]
next() -> 9      : pop 9; pushLeft(12) -> push 12, 11
                   [11, 12, 20]
next() -> 11     : pop 11; 11.right = null
                   [12, 20]

                   [14, 20]

                   [20]

                   [25]

struct TreeNode {
    int val;
    TreeNode *left, *right;
    TreeNode(int v) : val(v), left(nullptr), right(nullptr) {}
};

class BSTIterator {
    stack<TreeNode*> st;

    // Push the whole left spine starting at node.
    void pushLeft(TreeNode* node) {
        while (node) {
            st.push(node);
            node = node->left;
        }
    }

public:
    BSTIterator(TreeNode* root) {
        pushLeft(root);                // O(h) work, O(h) stack
    }

    // Return next-smallest key.
    int next() {
        TreeNode* cur = st.top();
        st.pop();
        pushLeft(cur->right);          // expose right subtree's left spine
        return cur->val;
    }

    bool hasNext() {
        return !st.empty();
    }
};

class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int v) { val = v; }
}

class BSTIterator {
    private Deque<TreeNode> stack = new ArrayDeque<>();

    public BSTIterator(TreeNode root) {
        pushLeft(root);                // O(h) work, O(h) stack
    }

    // Push the whole left spine starting at node.
    private void pushLeft(TreeNode node) {
        while (node != null) {
            stack.push(node);
            node = node.left;
        }
    }

    // Return next-smallest key.
    public int next() {
        TreeNode cur = stack.pop();
        pushLeft(cur.right);           // expose right subtree's left spine
        return cur.val;
    }

    public boolean hasNext() {
        return !stack.isEmpty();
    }
}

class TreeNode:
    def __init__(self, val=0, left=None, right=None):
        self.val = val
        self.left = left
        self.right = right

class BSTIterator:
    def __init__(self, root):
        self.stack = []
        self._push_left(root)          # O(h) work, O(h) stack

    def _push_left(self, node):
        # Push the whole left spine starting at node.
        while node:
            self.stack.append(node)
            node = node.left

    def next(self):
        # Return next-smallest key.
        cur = self.stack.pop()
        self._push_left(cur.right)     # expose right subtree's left spine
        return cur.val

    def hasNext(self):
        return len(self.stack) > 0

4. Amortized Analysis

A single next() can cost O(h) in the worst case — for example, when the popped node has a long left spine in its right subtree, pushLeft pushes many nodes at once. So per-call worst case is not O(1).

But look at the total work across a full traversal. Every node is pushed onto the stack exactly once and popped exactly once over the entire sequence of next() calls. There are n nodes, so the total push + pop work is O(n) spread across n calls to next().

Total stack operations over a complete iteration:
    pushes = n   (each node pushed once)
    pops   = n   (each node popped once)
    -------------------------------------
    total  = 2n  operations for n calls to next()

Amortized cost per next() = 2n / n = O(1)
Single-call worst case    = O(h)   (one big pushLeft)

This is the classic aggregate / amortized argument: even though individual calls vary, the average over the whole run is constant. hasNext() is always true O(1) since it just checks emptiness.

5. Variant: Bidirectional Iterator (next + prev)

Some problems want both next() (successor) and prev() (predecessor) from a movable cursor. A clean design keeps two stacks: a forward stack holding the left spine ahead of the cursor (as above), and a backward stack holding the right spine behind it (built symmetrically with a pushRight helper that walks down right children). Calling next() advances the cursor and records the visited node so prev() can rewind; prev() mirrors the logic using the right spine.

An alternative, simpler when the tree is static, is to materialize the sorted array once and keep an integer index that next() increments and prev() decrements — trading O(n) memory for trivial O(1) worst-case moves in both directions. Choose based on whether memory or worst-case latency matters more.

6. Use Cases

Merge / compare two BSTs in sorted order

Run two iterators in lockstep and always advance the one with the smaller current key — exactly like merging two sorted lists, but each stream uses only O(h) memory.

def merge_sorted(rootA, rootB):
    # Yield all keys of two BSTs in global sorted order.
    a, b = BSTIterator(rootA), BSTIterator(rootB)
    va = a.next() if a.hasNext() else None
    vb = b.next() if b.hasNext() else None
    out = []
    while va is not None and vb is not None:
        if va <= vb:
            out.append(va); va = a.next() if a.hasNext() else None
        else:
            out.append(vb); vb = b.next() if b.hasNext() else None
    while va is not None:
        out.append(va); va = a.next() if a.hasNext() else None
    while vb is not None:
        out.append(vb); vb = b.next() if b.hasNext() else None
    return out

k-th smallest element

Just call next() exactly k times. Because work is deferred, you only touch the part of the tree needed to surface the first k keys — ideal when k << n.

def kth_smallest(root, k):
    it = BSTIterator(root)
    for _ in range(k - 1):
        it.next()                  # skip the first k-1 keys
    return it.next()              # the k-th smallest

7. Complexity

OperationTimeMemory
ConstructorO(h) — pushes only the left spineO(h)
next()O(1) amortized (O(h) worst case)-
hasNext()O(1)-
Full traversal of n keysO(n) totalO(h)

For a balanced tree h = O(log n), so peak memory is logarithmic; for a degenerate (linked-list-shaped) tree h = O(n), matching the naive approach in the worst case but still never doing wasted work past where you stop.

Mental model — the stack is the inorder frontier:
  • At any moment the stack holds exactly the ancestors whose keys are still pending — nodes we have descended into (via left edges) but not yet returned.
  • The top is always the smallest unreturned key; the rest are its ancestors waiting their turn.
  • Popping a node and pushing its right child's left spine advances this frontier by one step — precisely the "in-progress" state of a paused inorder traversal.
