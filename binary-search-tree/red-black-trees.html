Red-Black Trees

Think of traffic rules. A city doesn't keep every street perfectly equal in length — that would be exhausting to enforce. Instead it posts a few simple rules (stop here, no two of these in a row, every route must pass the same number of checkpoints) and trusts that following them keeps traffic flowing "good enough" everywhere. A red-black tree works the same way: instead of obsessively keeping both sides of every node equal (the strict AVL approach), it paints each node red or black and enforces a handful of coloring rules.

That extra bit of color is all it takes. Follow the rules and the tree can never get more than twice as tall on one side as another — which is enough to keep search, insert, and delete at a fast O(log n), while doing far less rebalancing work than AVL on every update.

Motivation: Why Another Balanced BST?

We already have AVL trees, which are also self-balancing. So why do red-black trees exist and why are they so widely deployed? The answer is a different point on the balance/maintenance tradeoff.

  • Looser balance. AVL trees keep the heights of every node's two subtrees within 1 of each other — very strict. Red-black trees only guarantee the longest path is at most twice the shortest. They can be slightly taller than AVL trees.
  • Fewer rotations on updates. Because the balance criterion is relaxed, an insertion needs at most 2 rotations and a deletion at most 3 rotations to restore the invariants. AVL deletions can cascade rotations all the way up the tree.
  • The library workhorse. Red-black trees back C++ std::map / std::set / std::multimap, Java's TreeMap / TreeSet, and many OS kernels (the Linux completely-fair scheduler, the Linux process address-space tree, and various memory managers). When updates and lookups are roughly mixed, the cheaper rebalancing wins.

Rule of thumb: lookup-heavy workloads slightly favor AVL (shorter trees), while general-purpose / update-mixed workloads favor red-black (cheaper writes). Both are O(log n).

The Five Properties (the traffic rules)

A red-black tree is just a normal BST that always obeys these five rules. Here's each one in plain English, with why it exists:

  1. Color: every node is painted red or black. (The one extra bit we pay for.)
  2. Root is black. The very top is always black — a clean starting point, and it can never cause trouble.
  3. Empty slots are black. Every "missing child" (a NIL leaf) counts as black. Think of these as the black endpoints every path eventually reaches.
  4. No two reds in a row. A red node may not have a red child. Intuition: red means "free, doesn't add height" — but if reds could stack, one branch could pile up reds and grow much longer than another. Banning red-on-red caps that.
  5. Equal black-height. From any node, every path down to a NIL leaf passes through the same number of black nodes. This is the real balancer: it forces all routes to share the same "black skeleton," so no path can be wildly longer than the rest.

Rules 4 and 5 are the workhorses: equal black-height keeps the black skeleton balanced, and "no two reds in a row" stops the red nodes from stretching any single path too far. Together they guarantee the longest path is at most twice the shortest.

One note on NIL leaves: they're conceptual stand-ins for "nothing here." Real code uses a single shared black sentinel (or just treats null as black). The colored nodes in the visualizer below are the real key-carrying nodes; black NILs hang invisibly off every empty slot.

Why These Properties Imply Balance

The magic is property 4 (no two reds in a row) combined with property 5 (equal black-height). Together they bound how lopsided the tree can be.

  • Consider the shortest possible root-to-NIL path: it could be all black nodes. If its length is bh (the black-height), it has bh black nodes.
  • Consider the longest possible root-to-NIL path. Property 5 says it also has exactly bh black nodes. Property 4 says reds cannot be adjacent, so reds can make up at most every other node. The longest path therefore alternates black-red-black-red... and has at most 2 · bh nodes.

So the longest path is at most twice the shortest path. That is the core balance guarantee.

Proof sketch of O(log n) height:

1. A subtree rooted at node x with black-height bh(x)
   contains at least  2^bh(x) - 1  internal nodes.
   (Induction: a node has two subtrees each with
    black-height >= bh(x) - 1, so
    size >= 1 + 2*(2^(bh-1) - 1) = 2^bh - 1.)

2. By property 4, at least half the nodes on any
   root-to-NIL path are black, so for a tree of
   height h:   bh(root) >= h / 2.

3. Combine:   n >= 2^bh - 1 >= 2^(h/2) - 1
   =>  n + 1 >= 2^(h/2)
   =>  h <= 2 * log2(n + 1)

Therefore  h = O(log n), and every operation that
walks a single root-to-leaf path is O(log n).

Rotations

Rotations are the structural primitive shared with AVL trees: they locally rearrange three nodes to change height while preserving the BST in-order property. A left rotation around x makes x's right child y the new subtree root; a right rotation is its mirror. Colors are not changed by the rotation itself — the fix-up code recolors separately.

        leftRotate(x)                rightRotate(y)
       ------------->                <-------------

         x                                 y
        / \                               / \
       a   y           <=>               x   c
          / \                           / \
         b   c                         a   b

In-order sequence stays  a x b y c  in both shapes,
so the BST ordering is preserved. Only the subtree b
changes parent (from y to x, or x to y).

struct Node {
    int key;
    bool red;              // true = red, false = black
    Node *left, *right, *parent;
};

Node* NIL;               // shared black sentinel (red = false)
Node* root;

// Rotate the subtree rooted at x to the left.
void leftRotate(Node* x) {
    Node* y = x->right;          // y becomes new subtree root
    x->right = y->left;          // move y's left subtree (b) under x
    if (y->left != NIL) y->left->parent = x;
    y->parent = x->parent;       // link y to x's old parent
    if (x->parent == NIL)        root = y;
    else if (x == x->parent->left) x->parent->left  = y;
    else                         x->parent->right = y;
    y->left = x;                 // put x on y's left
    x->parent = y;
}

// Mirror image of leftRotate.
void rightRotate(Node* y) {
    Node* x = y->left;
    y->left = x->right;
    if (x->right != NIL) x->right->parent = y;
    x->parent = y->parent;
    if (y->parent == NIL)        root = x;
    else if (y == y->parent->left) y->parent->left  = x;
    else                         y->parent->right = x;
    x->right = y;
    y->parent = x;
}

    int key;
    boolean red;          // true = red, false = black
    Node left, right, parent;
}

    else if (x == x.parent.left) x.parent.left  = y;
    else                        x.parent.right = y;
    y.left = x;                  // put x on y's left
    x.parent = y;
}

// Mirror image of leftRotate.
void rightRotate(Node y) {
    Node x = y.left;
    y.left = x.right;
    if (x.right != NIL) x.right.parent = y;
    x.parent = y.parent;
    if (y.parent == NIL)        root = x;
    else if (y == y.parent.left) y.parent.left  = x;
    else                        y.parent.right = x;
    x.right = y;
    y.parent = x;
}

Insertion

Insertion proceeds in two phases:

  1. BST insert. Walk down like an ordinary BST and attach the new node as a leaf. Color it red and give it two black NIL children. Coloring it red is the safe default — a red node never changes any path's black-height, so property 5 is automatically preserved. The only invariant we might break is property 4 (a red node with a red parent).
  2. Fix-up. If the new node's parent is also red, we have a "red-red" violation. We repair it by walking up, looking at the new node's uncle (the sibling of its parent). Three cases handle every situation; each is mirrored for the left/right symmetry.

Below, z is the current node we are fixing, p its parent, g its grandparent, and u the uncle. We assume the parent is a left child of the grandparent; the right-child case is the exact mirror (swap left/right rotations).

Case 1: Uncle is RED — recolor and move up

If the uncle is red, we can push blackness down: recolor p and u black and g red. This fixes the local red-red conflict without changing any path's black-height. But g is now red and may conflict with its parent, so we set z = g and repeat.

Case 1 (uncle u is RED):

        g[B]                          g[R]   <- new z, recurse up
       /    \                        /    \
     p[R]   u[R]      recolor      p[B]   u[B]
     /                 ------>      /
   z[R]                          z[R]

p and u become black, g becomes red. Black-height of
every path through g is unchanged (we removed one red
above z but added one at g's level on each side).

Case 2: Uncle is BLACK, z is the "inner" child — rotate to set up Case 3

If the uncle is black and z is a right child (the "inner" grandchild, bending toward the uncle side), we first left-rotate around p. This straightens the zig-zag into a zig-zig, transforming the situation into Case 3. After the rotation we relabel so the lower node is again z.

Case 2 (u BLACK, z is inner/right child of left-child p):

        g[B]                        g[B]
       /    \                      /    \
     p[R]   u[B]    leftRotate   z[R]   u[B]
        \             (p)        /
       z[R]                    p[R]   <- relabel this as z, go to Case 3

The zig-zag (p left, z right) becomes a straight line.
No colors change here; we only restructure.

Case 3: Uncle is BLACK, z is the "outer" child — recolor and rotate grandparent

Now z is a left child of the left-child p (a straight line). Recolor p black and g red, then right-rotate around g. The black p rises to the top of the subtree, both children below it are balanced, and the red-red violation is resolved permanently — the loop terminates.

Case 3 (u BLACK, z is outer/left child of left-child p):

        g[B]                       p[B]
       /    \      recolor +      /    \
     p[R]   u[B]   rightRotate  z[R]   g[R]
     /              (g)                   \
   z[R]                                   u[B]

p -> black, g -> red, then rotate. The subtree root is
now black p, so its parent (whatever it was) sees no
red-red conflict. Done.

The fix-up loop runs while the parent is red. Case 1 may iterate up the tree (O(log n) recolorings), but Cases 2 and 3 each happen at most once and then terminate — hence at most 2 rotations per insert. Finally we force the root black (property 2), which never violates anything.

// Standard BST insert, then fix-up. NIL is the black sentinel.
void insert(int key) {
    Node* z = new Node{key, true, NIL, NIL, NIL}; // new node is RED
    Node* y = NIL;
    Node* x = root;
    while (x != NIL) {            // ordinary BST descent
        y = x;
        x = (key < x->key) ? x->left : x->right;
    }
    z->parent = y;
    if (y == NIL)            root = z;
    else if (key < y->key)    y->left  = z;
    else                     y->right = z;
    insertFixup(z);
}

void insertFixup(Node* z) {
    while (z->parent->red) {                 // while parent is RED
        Node* g = z->parent->parent;
        if (z->parent == g->left) {          // parent is a LEFT child
            Node* u = g->right;             // uncle
            if (u->red) {                   // --- Case 1: uncle RED ---
                z->parent->red = false;
                u->red = false;
                g->red = true;
                z = g;                     // move up and repeat
            } else {
                if (z == z->parent->right) { // --- Case 2: inner child ---
                    z = z->parent;
                    leftRotate(z);         // becomes Case 3
                }
                z->parent->red = false;     // --- Case 3: outer child ---
                g->red = true;
                rightRotate(g);
            }
        } else {                           // MIRROR: parent is a RIGHT child
            Node* u = g->left;
            if (u->red) {                   // Case 1
                z->parent->red = false;
                u->red = false;
                g->red = true;
                z = g;
            } else {
                if (z == z->parent->left) {  // Case 2 (mirror)
                    z = z->parent;
                    rightRotate(z);
                }
                z->parent->red = false;     // Case 3 (mirror)
                g->red = true;
                leftRotate(g);
            }
        }
    }
    root->red = false;                      // property 2: root is black
}

# NIL is the shared black sentinel. Each Node has
# key, red (bool), left, right, parent.

def insert(self, key):
    z = Node(key)
    z.red = True                      # new node is RED
    z.left = z.right = self.NIL
    y, x = self.NIL, self.root
    while x is not self.NIL:           # BST descent
        y = x
        x = x.left if key < x.key else x.right
    z.parent = y
    if y is self.NIL:
        self.root = z
    elif key < y.key:
        y.left = z
    else:
        y.right = z
    self._insert_fixup(z)

def _insert_fixup(self, z):
    while z.parent.red:               # while parent is RED
        g = z.parent.parent
        if z.parent is g.left:        # parent is LEFT child
            u = g.right
            if u.red:                # Case 1: uncle RED
                z.parent.red = False
                u.red = False
                g.red = True
                z = g
            else:
                if z is z.parent.right:  # Case 2: inner child
                    z = z.parent
                    self._left_rotate(z)
                z.parent.red = False     # Case 3: outer child
                g.red = True
                self._right_rotate(g)
        else:                         # MIRROR: parent is RIGHT child
            u = g.left
            if u.red:                # Case 1
                z.parent.red = False
                u.red = False
                g.red = True
                z = g
            else:
                if z is z.parent.left:   # Case 2 (mirror)
                    z = z.parent
                    self._right_rotate(z)
                z.parent.red = False     # Case 3 (mirror)
                g.red = True
                self._left_rotate(g)
    self.root.red = False            # root is black

Deletion

Deletion is the genuinely hard part of red-black trees. The BST removal itself is standard (splice out a node, or replace it with its in-order successor), but the tricky bit is what happens when the node that is physically removed — or the one that takes its place — was black. Removing a black node shortens the black-height of every path through it, violating property 5.

The classic fix introduces the idea of a "double black" node: the spot left behind is treated as carrying an extra unit of blackness that must be pushed away or absorbed. The fix-up loop examines the sibling w of the double-black node x and applies rotations/recolorings to discharge the extra black, walking up the tree until it is resolved or reaches the root (where the extra black is simply discarded).

CaseSibling w & its childrenAction
Case 1w is REDRecolor w black and parent red, rotate parent toward x. This converts to one of the cases below with a black sibling.
Case 2w BLACK, both of w's children BLACKRecolor w red, move the extra black up to the parent (set x = parent and repeat). If parent was red it absorbs it and we stop.
Case 3w BLACK, w's near child RED, far child BLACKRecolor and rotate w to turn the near-red into a far-red, converting to Case 4.
Case 4w BLACK, w's far child REDRecolor (w takes parent's color, parent and far child become black), rotate parent toward x. The extra black is absorbed; the loop terminates.

Each case has a left/right mirror, exactly like insertion. Only Cases 1, 3, and 4 perform rotations, and the loop performs at most 3 rotations total before terminating. Case 2 is the only one that iterates upward, and it does no rotation, so deletion is still O(log n).

Interview reality check:
  • Almost no interview asks you to code red-black deletion from scratch — it is long and error-prone. Knowing the double-black concept and the four sibling cases at a conceptual level is plenty.
  • Insertion fix-up (the three cases above) is fair game and worth being able to reason through.
  • Most of all, interviewers want you to know which container to reach for and what it guarantees, not to reimplement the balancing.

Choose AVL when

  • Lookups vastly outnumber updates
  • You want the shortest possible tree
  • Search latency is the bottleneck

Choose Red-Black when

  • Inserts and deletes are frequent
  • You want predictable, bounded rebalancing
  • You are using or building a general-purpose ordered map/set
Practical takeaway:
  • You will almost never implement a red-black tree by hand in production — reach for std::map / std::set in C++ or TreeMap / TreeSet in Java, which are red-black trees under the hood.
  • For interviews, internalize the five properties, the three insert cases, and the balance argument (height ≤ 2·log₂(n+1)). That is the high-value 20%.
  • Compare the tradeoffs against AVL trees, and see where ordered maps and sets fit in BST applications.
