Binary Search Tree: Basics & Operations

Think of the guess-the-number game. A friend picks a number; you guess, and they tell you "higher" or "lower." Each answer lets you throw away half the remaining numbers. A Binary Search Tree (BST) bakes that exact game into its shape: smaller values live to the left, bigger values to the right, so finding anything is just a series of "higher or lower?" hops.

It's also like a tidy filing system: at every drawer, papers that sort before it go in the left cabinet, papers that sort after it go in the right. Because everything is pre-sorted by position, search, insert, and delete all cost time proportional to the tree's height (how many hops from top to bottom). A tidy, balanced tree gives O(log n); a lopsided one that's become a long chain degrades to O(n).

1. The BST Ordering Invariant

An invariant is just a promise the tree always keeps. The BST's one promise (its ordering invariant) is this: for every node x,

  • Everything in x's left subtree is smaller than x — "go left for smaller."
  • Everything in x's right subtree is bigger than x — "go right for bigger."

That's the whole rule. We assume distinct keys here (no repeats). One handy payoff: if you read the tree inorder (visit left, then the node, then right), the keys come out perfectly sorted, smallest to largest.

Common mistake: The invariant is about entire subtrees, not just the immediate children.
  • It is NOT enough that left.key < x.key < right.key for each node.
  • Every descendant on the left must be smaller than x, and every descendant on the right larger. A node deep in the left subtree that exceeds an ancestor breaks the tree even if each local parent/child pair looks fine.
Duplicate-handling policy: Real BSTs must decide what to do with equal keys. Common policies: (a) disallow duplicates (insert is a no-op if the key exists) — the policy used on this page; (b) store a count per node; or (c) consistently send equal keys to one side (e.g. <= goes left). Pick one and apply it everywhere — mixing policies corrupts the invariant.

2. Node Definition

Each node stores a key plus pointers/references to its left and right children (null when absent).

struct TreeNode {
    int key;
    TreeNode* left;
    TreeNode* right;
    TreeNode(int k) : key(k), left(nullptr), right(nullptr) {}
};

class TreeNode {
    int key;
    TreeNode left, right;
    TreeNode(int k) {
        key = k;
        left = right = null;
    }
}

class TreeNode:
    def __init__(self, key):
        self.key = key
        self.left = None
        self.right = None

3. Search

Searching is the guess-the-number game. Compare your target to the current node: if it's smaller, go left; if it's bigger, go right; if it matches, you found it. If you walk off the bottom of the tree (hit a null spot), the value isn't there. Each step drops one level, so the cost is O(h), where h is the height. The live playground above runs exactly this loop — try it.

search(13) in the tree from section 1:

                 8        13 > 8  -> go right
               /   \
              3      10     13 > 10 -> go right
            /   \      \
           1     6      14   13 < 14 -> go left
                / \     /
               4   7   13   13 == 13 -> FOUND

Path: 8 -> 10 -> 14 -> 13   (4 comparisons, = height of path)

// Iterative search: returns the node or nullptr
TreeNode* searchIterative(TreeNode* root, int key) {
    while (root != nullptr && root->key != key) {
        if (key < root->key) root = root->left;
        else                 root = root->right;
    }
    return root;  // nullptr if not found
}

// Recursive search
TreeNode* searchRecursive(TreeNode* root, int key) {
    if (root == nullptr || root->key == key) return root;
    if (key < root->key) return searchRecursive(root->left, key);
    return searchRecursive(root->right, key);
}

def search_recursive(root, key):
    if root is None or root.key == key:
        return root
    if key < root.key:
        return search_recursive(root.left, key)
    return search_recursive(root.right, key)

4. Insert

Insertion follows the same descent as search. We walk down until we reach an empty (null) spot where the key belongs, then attach a new leaf there. With the no-duplicates policy, hitting an equal key is a no-op. Cost is O(h).

insert(5) into:                After attaching new leaf at the
                               empty left child of 6:
        8                            8
      /   \                        /   \
     3     10        ==>          3     10
   /   \                        /   \
  1     6                      1     6
       / \                          / \
      4   7     5 > 3, 5 > 4,      4   7
                5 < 6 -> left      \
                of 6 is empty       5   <-- new leaf

// Recursive insert: returns the (possibly new) subtree root
TreeNode* insert(TreeNode* root, int key) {
    if (root == nullptr) return new TreeNode(key);
    if (key < root->key)      root->left  = insert(root->left, key);
    else if (key > root->key) root->right = insert(root->right, key);
    // key == root->key: duplicate, ignore (no-op policy)
    return root;
}

// Iterative insert
TreeNode* insertIterative(TreeNode* root, int key) {
    TreeNode* node = new TreeNode(key);
    if (root == nullptr) return node;
    TreeNode* cur = root;
    while (true) {
        if (key == cur->key) { delete node; return root; }  // duplicate
        if (key < cur->key) {
            if (cur->left == nullptr) { cur->left = node; return root; }
            cur = cur->left;
        } else {
            if (cur->right == nullptr) { cur->right = node; return root; }
            cur = cur->right;
        }
    }
}

            return root  # duplicate
        if key < cur.key:
            if cur.left is None:
                cur.left = node
                return root
            cur = cur.left
        else:
            if cur.right is None:
                cur.right = node
                return root
            cur = cur.right

5. Find Min / Find Max

Because of the ordering invariant, the minimum key is the leftmost node (keep going left) and the maximum is the rightmost node (keep going right). Both take O(h).

        8
      /   \
     3     10
   /   \      \
  1     6      14
 ^leftmost     ^... 14 then no right child
 = min = 1            = max = 14

TreeNode* findMin(TreeNode* root) {
    while (root != nullptr && root->left != nullptr)
        root = root->left;
    return root;
}

TreeNode* findMax(TreeNode* root) {
    while (root != nullptr && root->right != nullptr)
        root = root->right;
    return root;
}

TreeNode findMin(TreeNode root) {
    while (root != null && root.left != null)
        root = root.left;
    return root;
}

TreeNode findMax(TreeNode root) {
    while (root != null && root.right != null)
        root = root.right;
    return root;
}

def find_min(root):
    while root is not None and root.left is not None:
        root = root.left
    return root

def find_max(root):
    while root is not None and root.right is not None:
        root = root.right
    return root

6. Delete

Deletion is the trickiest operation. After locating the node, there are three cases:

  • (a) Leaf (no children): simply remove it — replace it with null.
  • (b) One child: splice the node out — replace it with its single child.
  • (c) Two children: replace the node's key with its inorder successor (the minimum of the right subtree) or inorder predecessor (max of the left subtree), then recursively delete that successor/predecessor from the subtree. The successor has at most one child, so it reduces to case (a) or (b).
Case (a) - delete a LEAF: delete(4)

        8                 8
      /   \             /   \
     3     10   ==>    3     10
   /   \             /   \
  1     6           1     6
       /                            (4's parent now points to null)
      4   <-- remove

Case (b) - ONE child: delete(6)   (6 has only a left child, 4)

        8                 8
      /   \             /   \
     3     10   ==>    3     10
   /   \             /   \
  1     6           1     4    <-- 6 replaced by its child 4
       /
      4

Case (c) - TWO children: delete(3)

        8                          8
      /   \                      /   \
     3     10                   4     10     <-- 3's key overwritten
   /   \      \      ==>      /   \      \         with successor 4
  1     6      14            1     6      14
       / \     /                   \     /
      4   7   13                    7   13
   ^successor = min of                ^ old successor node (4)
    right subtree {6,4,7} = 4           was a leaf -> removed (case a)

Steps: 1) succ = findMin(3.right) = 4
       2) copy succ.key (4) into the node being deleted
       3) delete 4 from the right subtree (now a leaf / one-child case)

// Recursive delete: returns the new root of this subtree
TreeNode* deleteNode(TreeNode* root, int key) {
    if (root == nullptr) return nullptr;  // not found

    if (key < root->key) {
        root->left = deleteNode(root->left, key);
    } else if (key > root->key) {
        root->right = deleteNode(root->right, key);
    } else {
        // found the node to delete
        if (root->left == nullptr) {          // case a/b: no left child
            TreeNode* r = root->right;
            delete root;
            return r;
        } else if (root->right == nullptr) {  // case b: no right child
            TreeNode* l = root->left;
            delete root;
            return l;
        }
        // case c: two children -> inorder successor
        TreeNode* succ = root->right;
        while (succ->left != nullptr) succ = succ->left;
        root->key = succ->key;                         // copy successor key up
        root->right = deleteNode(root->right, succ->key); // remove successor
    }
    return root;
}

// Recursive delete: returns the new root of this subtree
TreeNode deleteNode(TreeNode root, int key) {
    if (root == null) return null;  // not found

    if (key < root.key) {
        root.left = deleteNode(root.left, key);
    } else if (key > root.key) {
        root.right = deleteNode(root.right, key);
    } else {
        // found the node to delete
        if (root.left == null)  return root.right;  // case a/b
        if (root.right == null) return root.left;   // case b

        // case c: two children -> inorder successor
        TreeNode succ = root.right;
        while (succ.left != null) succ = succ.left;
        root.key = succ.key;                          // copy successor key up
        root.right = deleteNode(root.right, succ.key); // remove successor
    }
    return root;
}

# Recursive delete: returns the new root of this subtree
def delete_node(root, key):
    if root is None:
        return None  # not found

    if key < root.key:
        root.left = delete_node(root.left, key)
    elif key > root.key:
        root.right = delete_node(root.right, key)
    else:
        # found the node to delete
        if root.left is None:
            return root.right   # case a/b
        if root.right is None:
            return root.left    # case b

        # case c: two children -> inorder successor
        succ = root.right
        while succ.left is not None:
            succ = succ.left
        root.key = succ.key                            # copy successor key up
        root.right = delete_node(root.right, succ.key) # remove successor
    return root
Why the successor works: the inorder successor is the smallest key larger than the deleted key, so substituting it preserves the ordering invariant. It lives in the right subtree and has no left child (it is that subtree's minimum), so deleting it only ever hits case (a) or (b) — no infinite recursion. Using the inorder predecessor (max of the left subtree) is equally valid.

7. Why Height Matters

Every operation costs O(h). If the tree is balanced, h = O(log n) and everything is fast. But a plain BST has no self-balancing: inserting already-sorted data produces a degenerate tree — effectively a linked list — where h = n and operations degrade to O(n).

Inserting 1,2,3,4,5 in sorted order into a plain BST:

  1
   \
    2
     \
      3            Height = n - 1
       \           search/insert/delete = O(n)
        4          (no better than a linked list)
         \
          5

Compare to a balanced arrangement of the same keys:

        3
      /   \
     2     4       Height = O(log n)
    /       \      all operations = O(log n)
   1         5
This is the motivation for self-balancing trees. Structures like AVL trees and Red-Black trees perform rotations during insert/delete to guarantee h = O(log n) in the worst case. See the dedicated AVL and Red-Black pages for how rotations keep the height bounded.

8. Complexity Summary

For a BST holding n keys, every core operation costs O(h):

OperationBalanced (h = log n)Worst case / skewed (h = n)
SearchO(log n)O(n)
InsertO(log n)O(n)
DeleteO(log n)O(n)
Find min / maxO(log n)O(n)
SpaceO(n)O(n)

The takeaway: a BST is only as good as it is balanced. For guaranteed O(log n) behavior on adversarial or sorted input, reach for a self-balancing variant.
