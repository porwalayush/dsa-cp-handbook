Largest BST Subtree

Imagine you've inherited a messy family org chart — a binary tree where the numbers were filled in by hand and nobody enforced any ordering rule. Somewhere inside that mess there are pockets that accidentally obey the binary-search-tree rule perfectly. Your job is to find the biggest such pocket: the largest chunk of the tree that, taken whole (a node together with all its descendants), is a valid BST. We report its size — the number of nodes in it.

The key word is subtree: you can't cherry-pick a few nodes here and there. A subtree means "pick a node, and you must take everything hanging beneath it." The whole pocket has to satisfy the BST rule from top to bottom.

The BST rule, one more time: in a valid binary search tree every node's key is greater than every key in its left subtree and less than every key in its right subtree. Not just its immediate children — its entire left and right families. That "entire family" part is exactly what makes this problem trickier than it first looks.

1. The Problem

You're handed the root of an arbitrary binary tree. It is not promised to be a BST — values can be in any arrangement at all. Return the number of nodes in the largest subtree that happens to be a valid BST.

Here's a tree to anchor the idea. Notice the whole thing is not a BST (the right side is broken), but there's a clean BST hiding on the left.

              10
             /  \
            5    15
           / \     \
          1   8     7      <-- 7 is the RIGHT child of 15, but 7 < 15!

Walk the candidates:
  subtree at 8  : single node          -> BST, size 1
  subtree at 1  : single node          -> BST, size 1
  subtree at 7  : single node          -> BST, size 1
  subtree at 5  : 1 < 5 < 8             -> BST, size 3   <== largest
  subtree at 15 : right child 7 < 15    -> NOT a BST
  subtree at 10 : right subtree broken  -> NOT a BST

Answer: largest BST subtree has size 3 (the subtree rooted at 5).

Even though node 10 looks orderly next to its own two children (5 < 10 < 15), it fails: its right subtree contains a 7 that violates the rule, so the subtree rooted at 10 is not a BST. The biggest valid pocket is the subtree rooted at 5, with 3 nodes.

Edge cases up front (they fall out naturally below):
  • An empty subtree (a null pointer) counts as a valid BST of size 0. This is the clean base case the whole algorithm leans on.
  • A single node with no children is always a valid BST of size 1.
  • Duplicates: a strict BST forbids equal keys. We use strict comparisons (node.val > left.max and node.val < right.min), so a duplicate value across the boundary makes that subtree invalid. (If your problem variant allows duplicates on one side, relax the matching comparison to ≥ or ≤.)

2. The Naive O(n²) Approach

The most direct idea: for every node, ask "is the subtree rooted here a valid BST?" If yes, count its nodes and update a running best. Two helpers, both standard: one checks BST-ness with the (low, high) bounds trick, another counts nodes.

The mental picture: you stand at each node and re-scan its entire family from scratch to answer the question. Then you move to the next node and re-scan its family from scratch again — even though those families overlap massively.

For EACH node:
     isBST(node)?  -- walks the whole subtree    O(size of subtree)
     if valid: size(node) -- walks it AGAIN        O(size of subtree)
     best = max(best, size)

Re-checking overlapping subtrees over and over:
                10            check 10's subtree (all 6 nodes)
               /  \
              5    15         then re-check 5's subtree (3 nodes)
             / \     \        then re-check 15's subtree...
            1   8     7       then 1, 8, 7 individually...

Same nodes get visited again and again from different ancestors.
Why it's wasteful: the BST-check at a node visits every node beneath it, and we run that check at every node. On a skewed (chain-like) tree that's n + (n-1) + (n-2) + ... = O(n²) total work. We keep re-deriving the same facts about lower subtrees that we could have computed once and reused. That redundancy is the signal to switch to a bottom-up approach. 
struct Node { int val; Node *left, *right; };

// Standard bounds check: is the whole subtree a BST?
bool isBST(Node* n, long long low, long long high) {
    if (!n) return true;                 // empty subtree is a BST
    if (n->val <= low || n->val >= high) return false;
    return isBST(n->left, low, n->val) &&
           isBST(n->right, n->val, high);
}

int countNodes(Node* n) {
    return n ? 1 + countNodes(n->left) + countNodes(n->right) : 0;
}

// For every node, re-check + re-count its subtree: O(n^2).
int largestBSTSubtree(Node* root) {
    if (!root) return 0;
    int best = 0;
    if (isBST(root, LLONG_MIN, LLONG_MAX))
        best = countNodes(root);
    return max({ best,
                 largestBSTSubtree(root->left),
                 largestBSTSubtree(root->right) });
}

class Node { int val; Node left, right; }

// Standard bounds check: is the whole subtree a BST?
boolean isBST(Node n, long low, long high) {
    if (n == null) return true;        // empty subtree is a BST
    if (n.val <= low || n.val >= high) return false;
    return isBST(n.left, low, n.val) &&
           isBST(n.right, n.val, high);
}

int countNodes(Node n) {
    return n == null ? 0 : 1 + countNodes(n.left) + countNodes(n.right);
}

// For every node, re-check + re-count its subtree: O(n^2).
int largestBSTSubtree(Node root) {
    if (root == null) return 0;
    int best = 0;
    if (isBST(root, Long.MIN_VALUE, Long.MAX_VALUE))
        best = countNodes(root);
    return Math.max(best,
           Math.max(largestBSTSubtree(root.left),
                    largestBSTSubtree(root.right)));
}

class Node:
    def __init__(self, val):
        self.val = val
        self.left = self.right = None

# Standard bounds check: is the whole subtree a BST?
def is_bst(n, low, high):
    if n is None:
        return True                  # empty subtree is a BST
    if not (low < n.val < high):
        return False
    return is_bst(n.left, low, n.val) and is_bst(n.right, n.val, high)

def count_nodes(n):
    return 0 if n is None else 1 + count_nodes(n.left) + count_nodes(n.right)

# For every node, re-check + re-count its subtree: O(n^2).
def largest_bst_subtree(root):
    if root is None:
        return 0
    best = 0
    if is_bst(root, float('-inf'), float('inf')):
        best = count_nodes(root)
    return max(best,
               largest_bst_subtree(root.left),
               largest_bst_subtree(root.right))

3. The Optimal O(n) Postorder Approach

Here's the insight that kills the redundancy: a subtree's BST-ness depends entirely on small summaries of its two children. So compute those summaries once, from the bottom up, and let each node hand a tidy report card up to its parent. This is the textbook "return a struct from each node" pattern — process children first (postorder), combine, pass the result upward.

Each node returns a four-field summary about its own subtree:

  • isBST — is this whole subtree a valid BST?
  • size — how many nodes are in this subtree?
  • min — the smallest key anywhere in this subtree.
  • max — the largest key anywhere in this subtree.

A node combines its children's reports into its own. The subtree at a node is a BST if and only if all three of these hold:

  1. the left child's subtree is a BST, and
  2. the right child's subtree is a BST, and
  3. node.val > left.max and node.val < right.min — the node sits strictly above everything on its left and strictly below everything on its right.

If it qualifies, its size is left.size + right.size + 1, its min is the left subtree's min (or the node itself), and its max is the right subtree's max. We keep a single global best and update it with every BST we confirm. The empty-subtree base case returns {isBST: true, size: 0, min: +∞, max: -∞} — those flipped infinities are deliberate so the comparison node.val > left.max automatically passes when there is no left child.

Postorder: children report first, then the node combines.
Summary = {isBST, size, min, max}

              10
             /  \
            5    15
           / \     \
          1   8     7

  leaf 1 : {T, 1, 1, 1}     leaf 8 : {T, 1, 8, 8}
  node 5 : left.max=1 < 5 < right.min=8  -> BST
           {T, size 3, min 1, max 8}            <== best so far = 3
  leaf 7 : {T, 1, 7, 7}
  node 15: needs val < right.min(7)?  15 < 7 is FALSE
           -> NOT a BST  {F, -, -, -}
  node 10: right child not a BST -> NOT a BST  {F, -, -, -}

  Largest BST subtree size = 3
Why null returns min=+∞, max=-∞: for a node with no left child, the test node.val > left.max becomes node.val > -∞, which is always true — exactly what we want (nothing on the left to violate). Symmetrically node.val < right.min becomes node.val < +∞ for a missing right child. The sentinels make the leaf and single-child cases need no special handling.
Short-circuit, don't crash: once either child reports isBST = false, the current node is immediately not a BST — you must not read a non-existent ordering from a broken child. Return a "not a BST" summary right away and let the global best stay as it is. The size field of a non-BST report is never trusted by an ancestor. 
struct Info {
    bool isBST;
    int size;
    long long mn, mx;   // min and max key in this subtree
};

int best = 0;

Info solve(Node* node) {
    if (!node)                            // empty subtree: BST of size 0
        return { true, 0, LLONG_MAX, LLONG_MIN };  // flipped sentinels

    Info L = solve(node->left);
    Info R = solve(node->right);

    // BST iff both children are BSTs and node fits strictly between them
    if (L.isBST && R.isBST &&
        node->val > L.mx && node->val < R.mn) {
        int size = L.size + R.size + 1;
        best = max(best, size);
        return { true, size,
                 min((long long)node->val, L.mn),
                 max((long long)node->val, R.mx) };
    }
    // Not a BST: size/min/max won't be trusted by an ancestor.
    return { false, 0, 0, 0 };
}

int largestBSTSubtree(Node* root) {
    best = 0;
    solve(root);
    return best;
}

class Info {
    boolean isBST;
    int size;
    long mn, mx;        // min and max key in this subtree
    Info(boolean b, int s, long mn, long mx) {
        isBST = b; size = s; this.mn = mn; this.mx = mx;
    }
}

int best = 0;

Info solve(Node node) {
    if (node == null)                    // empty subtree: BST of size 0
        return new Info(true, 0, Long.MAX_VALUE, Long.MIN_VALUE);

    Info L = solve(node.left);
    Info R = solve(node.right);

    // BST iff both children are BSTs and node fits strictly between them
    if (L.isBST && R.isBST &&
        node.val > L.mx && node.val < R.mn) {
        int size = L.size + R.size + 1;
        best = Math.max(best, size);
        return new Info(true, size,
            Math.min(node.val, L.mn),
            Math.max(node.val, R.mx));
    }
    // Not a BST: size/min/max won't be trusted by an ancestor.
    return new Info(false, 0, 0, 0);
}

int largestBSTSubtree(Node root) {
    best = 0;
    solve(root);
    return best;
}

import math

def largest_bst_subtree(root):
    best = [0]        # list cell so the closure can mutate it

    # returns (isBST, size, mn, mx) for the subtree at `node`
    def solve(node):
        if node is None:                       # empty: BST of size 0
            return (True, 0, math.inf, -math.inf)  # flipped sentinels

        l_bst, l_sz, l_mn, l_mx = solve(node.left)
        r_bst, r_sz, r_mn, r_mx = solve(node.right)

        # BST iff both children are BSTs and node fits strictly between
        if l_bst and r_bst and l_mx < node.val < r_mn:
            size = l_sz + r_sz + 1
            best[0] = max(best[0], size)
            return (True, size, min(node.val, l_mn), max(node.val, r_mx))

        # Not a BST: these fields won't be trusted by an ancestor.
        return (False, 0, 0, 0)

    solve(root)
    return best[0]

5. Complexity Analysis

Each node hands one small constant-size report up to its parent, and is visited exactly once.

ApproachTimeSpace
Naive (re-check each subtree)O(n²) worst case (skewed tree)O(h) recursion stack
Optimal postorder (return a struct)O(n)O(h) recursion stack

Here n is the number of nodes and h is the tree height — O(log n) for a balanced tree, O(n) for a skewed one. The postorder method touches every node once and does O(1) work per node, so it is linear in time; the only space cost is the recursion stack proportional to the height.

The takeaway pattern: whenever a tree question asks about every subtree at once — largest BST subtree, diameter, balanced-tree check, max path sum — reach for the postorder "return a struct from each node" technique. Compute children first, fold their summaries into the parent's summary, update a global answer along the way. One pass, O(n), no re-scanning. Internalize this and a whole family of "hard" tree problems collapses into the same shape.
