AVL Trees (Self-Balancing BST)

An AVL tree (Adelson-Velsky and Landis, 1962) is a binary search tree that keeps itself balanced by guaranteeing that, for every node, the heights of its two child subtrees differ by at most one. This bound is restored after every insertion and deletion using rotations, giving worst-case O(log n) search, insert, and delete.

Motivation: Why Balance?

A plain BST has O(log n) operations only when it stays bushy. Feed it already-sorted data and each new key becomes the right child of the previous one. The tree collapses into a linked list and every operation degrades to O(n).

Insert 1, 2, 3, 4, 5 into a plain BST (no balancing):

    1
     \
      2
       \
        3              height = 4, n = 5
         \             search(5) visits all 5 nodes -> O(n)
          4
           \
            5

The same 5 keys in an AVL tree stay balanced:

        2
       / \
      1   4
         / \
        3   5          height = 2, search(5) -> O(log n)

AVL trees prevent this degeneration by enforcing a strict height invariant on every node, so the tree height stays O(log n) no matter what order the keys arrive in.

Balance Factor

For any node, define its balance factor as:

balanceFactor(node) = height(node.left) - height(node.right)

An AVL tree requires balanceFactor(node) to be in {-1, 0, +1} for every node. A factor outside this range means the node is too left-heavy (> +1) or too right-heavy (< -1) and must be fixed by a rotation. By convention an empty subtree has height -1 and a single leaf has height 0. Each node caches its own height so the balance factor can be computed in O(1).

A balanced AVL tree.  Notation:  key (h=height, bf=balanceFactor)

                30 (h=2, bf=0)
               /            \
       20 (h=1, bf=+1)      40 (h=1, bf=-1)
       /                       \
  10 (h=0, bf=0)            50 (h=0, bf=0)

height(empty) = -1,  height(leaf) = 0
bf(20) = h(10) - h(empty) = 0 - (-1) = +1   (ok)
bf(40) = h(empty) - h(50) = -1 - 0  = -1    (ok)
bf(30) = h(20) - h(40)    = 1 - 1   =  0    (ok)

Every |bf| <= 1, so the tree is a valid AVL tree.

Rotations: The Core Primitive

A rotation locally rearranges three pointers to change the height of a subtree while preserving the BST in-order ordering. There are two mirror-image rotations.

Right Rotation (around y)

Used when the left subtree is too tall. The left child x moves up to become the new root of the subtree; y becomes x's right child; x's former right subtree T2 is re-parented as y's left subtree.

        rightRotate(y)

         y                         x
        / \                       / \
       x   T3      ===>         T1   y
      / \                           / \
    T1   T2                       T2   T3

Subtree that moves: T2 (was x.right, becomes y.left)
Ordering preserved: T1 < x < T2 < y < T3  in both shapes.

Left Rotation (around x)

The mirror case: the right subtree is too tall. The right child y moves up; x becomes y's left child; y's former left subtree T2 is re-parented as x's right subtree.

        leftRotate(x)

       x                           y
      / \                         / \
    T1   y          ===>         x   T3
        / \                     / \
      T2   T3                 T1   T2

Subtree that moves: T2 (was y.left, becomes x.right)
Ordering preserved: T1 < x < T2 < y < T3  in both shapes.
Critical: after rewiring pointers you MUST recompute heights, and you must do it in the right order — the lower node (the one that moved down) first, then the new subtree root. Both rotations are O(1): a fixed number of pointer assignments and two height updates. 
struct Node {
    int key, height;
    Node *left, *right;
    Node(int k) : key(k), height(0), left(nullptr), right(nullptr) {}
};

int height(Node* n) { return n ? n->height : -1; }

void updateHeight(Node* n) {
    n->height = 1 + max(height(n->left), height(n->right));
}

int getBalance(Node* n) {
    return n ? height(n->left) - height(n->right) : 0;
}

// Right rotation around y, returns the new subtree root (x)
Node* rightRotate(Node* y) {
    Node* x  = y->left;
    Node* T2 = x->right;
    x->right = y;          // y moves down to the right
    y->left  = T2;         // T2 re-parented under y
    updateHeight(y);      // update lower node first
    updateHeight(x);      // then the new root
    return x;
}

// Left rotation around x, returns the new subtree root (y)
Node* leftRotate(Node* x) {
    Node* y  = x->right;
    Node* T2 = y->left;
    y->left  = x;          // x moves down to the left
    x->right = T2;         // T2 re-parented under x
    updateHeight(x);      // update lower node first
    updateHeight(y);      // then the new root
    return y;
}

class Node:
    def __init__(self, key):
        self.key = key
        self.height = 0
        self.left = None
        self.right = None

def height(n):
    return n.height if n else -1

def update_height(n):
    n.height = 1 + max(height(n.left), height(n.right))

def get_balance(n):
    return height(n.left) - height(n.right) if n else 0

# Right rotation around y, returns the new subtree root (x)
def right_rotate(y):
    x  = y.left
    T2 = x.right
    x.right = y          # y moves down to the right
    y.left  = T2         # T2 re-parented under y
    update_height(y)     # update lower node first
    update_height(x)     # then the new root
    return x

# Left rotation around x, returns the new subtree root (y)
def left_rotate(x):
    y  = x.right
    T2 = y.left
    y.left  = x          # x moves down to the left
    x.right = T2         # T2 re-parented under x
    update_height(x)     # update lower node first
    update_height(y)     # then the new root
    return y

The Four Imbalance Cases

When a node z becomes unbalanced (|balanceFactor(z)| > 1), exactly one of four shapes applies, decided by the sign of z's balance factor and the relevant child's balance factor. The fix is one or two rotations.

Case LL (Left-Left) — single right rotation

Trigger: balance(z) > 1 and balance(z.left) >= 0 (inserted into the left subtree of the left child).

        z (bf=+2)                 y
       / \                       / \
      y   T4    rightRotate(z)  x   z
     / \         ========>     / \ / \
    x   T3                    T1 T2 T3 T4
   / \
  T1  T2

Fix:  z = rightRotate(z)

Case RR (Right-Right) — single left rotation

Trigger: balance(z) < -1 and balance(z.right) <= 0 (inserted into the right subtree of the right child).

    z (bf=-2)                       y
   / \                             / \
  T1  y         leftRotate(z)     z   x
     / \         ========>       / \ / \
    T2  x                       T1 T2 T3 T4
       / \
      T3  T4

Fix:  z = leftRotate(z)

Case LR (Left-Right) — left rotate child, then right rotate node

Trigger: balance(z) > 1 and balance(z.left) < 0 (inserted into the right subtree of the left child). First a left rotation on the child turns it into the LL shape, then a right rotation on z.

      z (bf=+2)              z                      x
     / \                    / \                    / \
    y   T4  leftRotate(y)  x   T4  rightRotate(z) y   z
   / \      =========>    / \      ==========>   / \ / \
  T1  x                  y   T3                 T1 T2 T3 T4
     / \                / \
    T2  T3            T1  T2

Fix:  z.left = leftRotate(z.left);  z = rightRotate(z)

Case RL (Right-Left) — right rotate child, then left rotate node

Trigger: balance(z) < -1 and balance(z.right) > 0 (inserted into the left subtree of the right child). First a right rotation on the child turns it into the RR shape, then a left rotation on z.

   z (bf=-2)                  z                          x
  / \                        / \                        / \
 T1  y     rightRotate(y)   T1  x     leftRotate(z)     z   y
    / \     =========>          / \    =========>      / \ / \
   x   T4                      T2  y                  T1 T2 T3 T4
  / \                             / \
 T2  T3                          T3  T4

Fix:  z.right = rightRotate(z.right);  z = leftRotate(z)
Decision logic in one place (let bf = balance(z)):
  • bf > 1 && balance(z.left) >= 0 → LL: rightRotate(z)
  • bf > 1 && balance(z.left) < 0 → LR: z.left = leftRotate(z.left); rightRotate(z)
  • bf < -1 && balance(z.right) <= 0 → RR: leftRotate(z)
  • bf < -1 && balance(z.right) > 0 → RL: z.right = rightRotate(z.right); leftRotate(z)

Insertion

Insertion is a normal recursive BST insert followed by a "fix-up" on the way back up the recursion: update the height of the current node, compute its balance factor, and if it is out of range apply the matching rotation. Because a single insertion lengthens at most one path, at most one rotation (single or double) is ever needed, and it happens at the lowest unbalanced ancestor.

Node* insert(Node* node, int key) {
    // 1. Standard BST insert
    if (!node) return new Node(key);
    if (key < node->key)
        node->left  = insert(node->left, key);
    else if (key > node->key)
        node->right = insert(node->right, key);
    else
        return node;  // duplicates not allowed

    // 2. Update height of this ancestor
    updateHeight(node);

    // 3. Compute balance and rebalance if needed
    int bf = getBalance(node);

    // LL
    if (bf > 1 && getBalance(node->left) >= 0)
        return rightRotate(node);
    // LR
    if (bf > 1 && getBalance(node->left) < 0) {
        node->left = leftRotate(node->left);
        return rightRotate(node);
    }
    // RR
    if (bf < -1 && getBalance(node->right) <= 0)
        return leftRotate(node);
    // RL
    if (bf < -1 && getBalance(node->right) > 0) {
        node->right = rightRotate(node->right);
        return leftRotate(node);
    }

    return node;  // already balanced
}

Deletion

Deletion is a standard BST delete (replace a two-child node by its in-order successor, then delete that successor) followed by the same rebalancing logic on the way up. The key difference from insertion: a deletion can shorten a subtree, which may cascade and require rotations at multiple ancestors all the way up to the root — whereas insertion needs at most one.

One subtlety: when rebalancing after a delete, the off-balance child may itself have a balance factor of 0. The comparisons above (>= 0 for the left child, <= 0 for the right child) handle this correctly, so the exact same case logic works for both insert and delete.

Node* minValueNode(Node* n) {
    while (n->left) n = n->left;
    return n;
}

// Shared rebalance step (identical to insert's fix-up)
Node* rebalance(Node* node) {
    updateHeight(node);
    int bf = getBalance(node);
    if (bf > 1 && getBalance(node->left) >= 0)
        return rightRotate(node);                 // LL
    if (bf > 1 && getBalance(node->left) < 0) {     // LR
        node->left = leftRotate(node->left);
        return rightRotate(node);
    }
    if (bf < -1 && getBalance(node->right) <= 0)
        return leftRotate(node);                  // RR
    if (bf < -1 && getBalance(node->right) > 0) { // RL
        node->right = rightRotate(node->right);
        return leftRotate(node);
    }
    return node;
}

Node* deleteNode(Node* node, int key) {
    if (!node) return node;

    // 1. Standard BST delete
    if (key < node->key)
        node->left  = deleteNode(node->left, key);
    else if (key > node->key)
        node->right = deleteNode(node->right, key);
    else {
        if (!node->left || !node->right) {        // 0 or 1 child
            Node* child = node->left ? node->left : node->right;
            delete node;
            return child;                         // may be nullptr
        }
        // 2 children: replace with in-order successor
        Node* succ = minValueNode(node->right);
        node->key = succ->key;
        node->right = deleteNode(node->right, succ->key);
    }

    // 2. Rebalance on the way up (may rotate at many levels)
    return rebalance(node);
}

def min_value_node(n):
    while n.left:
        n = n.left
    return n

# Shared rebalance step (identical to insert's fix-up)
def rebalance(node):
    update_height(node)
    bf = get_balance(node)
    if bf > 1 and get_balance(node.left) >= 0:      # LL
        return right_rotate(node)
    if bf > 1 and get_balance(node.left) < 0:       # LR
        node.left = left_rotate(node.left)
        return right_rotate(node)
    if bf < -1 and get_balance(node.right) <= 0:    # RR
        return left_rotate(node)
    if bf < -1 and get_balance(node.right) > 0:     # RL
        node.right = right_rotate(node.right)
        return left_rotate(node)
    return node

def delete_node(node, key):
    if node is None:
        return node

    # 1. Standard BST delete
    if key < node.key:
        node.left  = delete_node(node.left, key)
    elif key > node.key:
        node.right = delete_node(node.right, key)
    else:
        if node.left is None or node.right is None:
            return node.left if node.left else node.right
        succ = min_value_node(node.right)     # in-order successor
        node.key = succ.key
        node.right = delete_node(node.right, succ.key)

    # 2. Rebalance on the way up (may rotate at many levels)
    return rebalance(node)

Height Bound and Fibonacci

How tall can an AVL tree of n nodes get? Ask the opposite question: what is the minimum number of nodes N(h) in an AVL tree of height h? Such a "worst-case sparse" tree has a root, one subtree of height h-1, and the other of height h-2 (the smallest difference still allowed):

N(h) = 1 + N(h-1) + N(h-2),   N(0) = 1,  N(1) = 2

h :  0   1   2   3    4    5    6
N :  1   2   4   7   12   20   33

Compare Fibonacci:  F: 1, 1, 2, 3, 5, 8, 13, 21, 34, ...
   N(h) = F(h + 3) - 1     (the recurrence is Fibonacci shifted)

Since N(h) grows like the Fibonacci numbers (which grow as φ^h with the golden ratio φ ≈ 1.618), inverting the relation gives the height bound:

h < 1.4405 · log₂(n + 2) - 0.3277  →  h = O(log n)

So an AVL tree is never more than about 44% taller than a perfectly balanced tree, which is what guarantees O(log n) operations.

Complexity

OperationTime (worst case)Notes
SearchO(log n)Height is bounded by ~1.44 log₂n
InsertO(log n)BST insert + at most one rotation
DeleteO(log n)BST delete + up to O(log n) rotations
Single rotationO(1)Fixed pointer rewiring + 2 height updates
SpaceO(n)One extra int (height) per node
Most common bug: forgetting to update node heights after a rotation, or updating them in the wrong order. In each rotation, recompute the height of the node that moved down first, then the new subtree root — otherwise the parent's height is computed from a stale child height and every later balance factor is wrong.

AVL vs Red-Black Trees

AVL Tree

  • More rigidly balanced (height ≤ ~1.44 log₂n)
  • Faster lookups — shallower trees
  • May do more rotations per update
  • Best for read-heavy workloads

Red-Black Tree

  • Looser balance (height ≤ 2 log₂(n+1))
  • Slightly slower lookups — taller trees
  • At most O(1) rotations per insert/delete (amortized fewer)
  • Best for write-heavy workloads
Rule of thumb: choose AVL when lookups dominate and you want the flattest possible tree; choose a Red-Black tree when inserts and deletes are frequent and you want to minimize restructuring work. Both guarantee O(log n) for all operations; the difference is in the constant factors. Most standard library ordered maps (e.g. C++ std::map, Java TreeMap) use red-black trees for exactly this reason.
