Order Statistics in a BST

Imagine all the keys in your tree lined up in sorted order, like runners at a finish line. Order statistics are the questions you ask about that lineup: "who came in 4th?" (the kth smallest — called select) and "how many runners are ahead of this person?" (how many keys are smaller — called rank). A binary search tree already keeps its keys in sorted order through its in-order walk, so it's a natural place to answer these. This page starts with a plain O(n) walk and works up to an augmented BST that jumps straight to the answer in O(h) time.

1. Kth Smallest via In-order Traversal

An in-order traversal of a BST (left → root → right) visits keys in ascending order — it's literally counting from the smallest key upward. So finding the kth smallest is just counting nodes as you walk: the kth one you visit is your answer. And you can stop the moment your counter hits k — no need to walk the rest. An iterative stack version makes that early exit clean and avoids recursion overhead.

BST:
            8
          /   \
         3      10
        / \       \
       1   6       14
          / \      /
         4   7    13

In-order sequence:  1  3  4  6  7  8  10  13  14
visit count (k):    1  2  3  4  5  6   7   8   9

kthSmallest(k = 4):
  push 8, push 3, push 1
  pop 1  -> count 1                (right of 1 is null)
  pop 3  -> count 2 -> push 6, push 4
  pop 4  -> count 3                (right of 4 is null)
  pop 6  -> count 4  == k  =>  return 6   (early exit)

struct Node { int key; Node *left, *right; };

// kth smallest (1-indexed) via iterative in-order with early exit
int kthSmallest(Node* root, int k) {
    stack<Node*> st;
    Node* cur = root;
    while (cur != nullptr || !st.empty()) {
        // go as far left as possible
        while (cur != nullptr) { st.push(cur); cur = cur->left; }
        cur = st.top(); st.pop();
        if (--k == 0) return cur->key;  // early exit
        cur = cur->right;
    }
    return -1;  // k out of range
}

int kthSmallest(Node root, int k) {
    Deque<Node> st = new ArrayDeque<>();
    Node cur = root;
    while (cur != null || !st.isEmpty()) {
        while (cur != null) { st.push(cur); cur = cur.left; }
        cur = st.pop();
        if (--k == 0) return cur.key;  // early exit
        cur = cur.right;
    }
    return -1;
}

def kth_smallest(root, k):
    st, cur = [], root
    while cur or st:
        while cur:                # descend to leftmost
            st.append(cur)
            cur = cur.left
        cur = st.pop()
        k -= 1
        if k == 0:               # early exit
            return cur.key
        cur = cur.right
    return -1                   # k out of range

With early exit this runs in O(h + k) time: O(h) to reach the leftmost leaf, then one stack operation per visited node up to k. In the worst case (skewed tree, or k near n) it degrades to O(n).

2. Kth Largest via Reverse In-order

To get the kth largest key, traverse in reverse in-order (right → root → left), which visits keys in descending order, and stop at the kth visited node.

Reverse in-order of the tree above:  14 13 10 8 7 6 4 3 1
visit count (k):                      1  2  3 4 5 6 7 8 9

kthLargest(k = 3) => 10

// kth largest via iterative reverse in-order (right, root, left)
int kthLargest(Node* root, int k) {
    stack<Node*> st;
    Node* cur = root;
    while (cur != nullptr || !st.empty()) {
        while (cur != nullptr) { st.push(cur); cur = cur->right; }
        cur = st.top(); st.pop();
        if (--k == 0) return cur->key;
        cur = cur->left;
    }
    return -1;
}

def kth_largest(root, k):
    st, cur = [], root
    while cur or st:
        while cur:            # descend to rightmost
            st.append(cur)
            cur = cur.right
        cur = st.pop()
        k -= 1
        if k == 0:
            return cur.key
        cur = cur.left
    return -1

3. Augmented BST: Storing Subtree Sizes

The traversal approach is O(n) in the worst case because it has no shortcut: it cannot skip past subtrees. The fix is to augment each node with one extra field: size, the number of nodes in the subtree rooted at that node (including the node itself). With sizes available, we can decide at each step how many keys live in the left subtree and prune accordingly.

Each node annotated as  key(size):

                 8(9)
               /      \
            3(5)        10(3)
           /    \           \
        1(1)    6(3)        14(2)
                /  \         /
             4(1)  7(1)   13(1)

size(node) = 1 + size(left) + size(right)

leftSize(node) = size(node.left)   (0 if no left child)
  e.g. leftSize(8) = size(3) = 5  -> 5 keys are less than 8

Maintaining size on insert

On the way down the search path during an insert, increment size on every node you pass through (each gains exactly one descendant). Equivalently, on the way back up from a recursive insert, recompute node.size = 1 + size(node.left) + size(node.right). Either way the update touches only the O(h) nodes on the path.

Maintaining size on delete

Decrement size on every node along the path to the deleted node, then recompute sizes back up to the root. When deleting a node with two children you typically swap with its in-order successor (the minimum of the right subtree); recompute sizes along the successor's path as well. All affected nodes lie on O(h) root-to-node paths, so the bookkeeping stays O(h).

Invariant to preserve: after every insert or delete, for every node node.size == 1 + size(node.left) + size(node.right), treating a null child's size as 0. If you ever rebalance (rotations in an AVL / red-black tree), update sizes inside the rotation: the rotated-down node's size is recomputed first, then the rotated-up node's.

4. Select (Kth Smallest) in O(h) Using Sizes

Here's the payoff. Instead of counting nodes one by one, the subtree sizes let you count whole subtrees in a single glance and jump past them. Standing at any node, the left subtree already holds leftSize keys smaller than this node, so this node's position in its own subtree is leftSize + 1. That one number tells you which way to go:

  • k == leftSize + 1 → you're standing on the answer.
  • k <= leftSize → the kth smallest is somewhere in the left subtree; go left, k unchanged.
  • k > leftSize + 1 → it's in the right subtree; go right, but first subtract the leftSize + 1 keys you just skipped past (the left subtree plus this node), so k -= leftSize + 1.

One root-to-leaf path, no wandering — that's why it's O(h) instead of O(n).

struct Node { int key, size; Node *left, *right; };

int sz(Node* n) { return n ? n->size : 0; }

// select: return kth smallest key (1-indexed), in O(h)
Node* select(Node* root, int k) {
    Node* cur = root;
    while (cur != nullptr) {
        int leftSize = sz(cur->left);
        if (k == leftSize + 1) return cur;         // found
        else if (k <= leftSize) cur = cur->left;       // go left
        else { k -= leftSize + 1; cur = cur->right; }  // go right
    }
    return nullptr;  // k out of range
}

static int sz(Node n) { return n == null ? 0 : n.size; }

Node select(Node root, int k) {
    Node cur = root;
    while (cur != null) {
        int leftSize = sz(cur.left);
        if (k == leftSize + 1) return cur;
        else if (k <= leftSize) cur = cur.left;
        else { k -= leftSize + 1; cur = cur.right; }
    }
    return null;
}

def _sz(n):
    return n.size if n else 0

def select(root, k):
    cur = root
    while cur:
        left_size = _sz(cur.left)
        if k == left_size + 1:        # found
            return cur
        elif k <= left_size:           # go left
            cur = cur.left
        else:                          # go right
            k -= left_size + 1
            cur = cur.right
    return None                        # k out of range

5. Rank of a Key in O(h) Using Sizes

rank(key) returns the number of keys strictly less than key. Walk down from the root accumulating a running count:

  • Going right from a node means the node and its entire left subtree are smaller than key, so add leftSize + 1 to the count.
  • Going left adds nothing.
  • When you land on a node equal to key, add its leftSize (all keys below it) and stop.

This is the inverse of select: rank(select(k).key) == k - 1.

rank(10) on the tree from Section 3  (count keys < 10):

  at 8(9):  10 > 8  =>  count += leftSize(8)+1 = 5+1 = 6, go right
  at 10(3): 10 == 10 =>  count += leftSize(10) = 0, stop
  result = 6      (keys 1 3 4 6 7 8 are < 10)  ✓

// rank: number of keys strictly less than `key`, in O(h)
int rank(Node* root, int key) {
    int count = 0;
    Node* cur = root;
    while (cur != nullptr) {
        if (key < cur->key) {
            cur = cur->left;
        } else if (key > cur->key) {
            count += sz(cur->left) + 1;       // node + left subtree are smaller
            cur = cur->right;
        } else {
            count += sz(cur->left);            // keys below this equal node
            break;
        }
    }
    return count;
}

int rank(Node root, int key) {
    int count = 0;
    Node cur = root;
    while (cur != null) {
        if (key < cur.key) {
            cur = cur.left;
        } else if (key > cur.key) {
            count += sz(cur.left) + 1;
            cur = cur.right;
        } else {
            count += sz(cur.left);
            break;
        }
    }
    return count;
}

def rank(root, key):
    count, cur = 0, root
    while cur:
        if key < cur.key:
            cur = cur.left
        elif key > cur.key:
            count += _sz(cur.left) + 1   # node + left subtree smaller
            cur = cur.right
        else:
            count += _sz(cur.left)         # keys below equal node
            break
    return count

6. Counting Keys in a Range [lo, hi]

Once rank is available, counting keys in an inclusive range is a one-liner. The number of keys strictly less than hi+1 minus the number strictly less than lo gives the count of keys in [lo, hi]:

countInRange(lo, hi) = rank(hi + 1) - rank(lo)

If keys are not integers (so hi + 1 is meaningless), define a rankUpper(key) that counts keys ≤ key (go right and also add leftSize + 1 on equality), then countInRange(lo, hi) = rankUpper(hi) - rank(lo). Either way it costs two O(h) descents. A direct single-pass traversal that prunes subtrees fully outside [lo, hi] using sizes is also possible and avoids visiting irrelevant subtrees.

7. Related Structures

Cross-references:
  • Fenwick / BIT order statistics: when keys come from a fixed, known universe (or after coordinate compression), a Fenwick Tree used as a frequency array supports the same select/rank queries. Bit-descent on the BIT finds the kth smallest in O(log n); a prefix sum gives rank. Prefer the BIT when the key set is dense and static; prefer the augmented BST when keys are sparse, arbitrary, or arrive with deletions.
  • Balanced order-statistic trees: to keep h = O(log n) in the worst case, layer the size augmentation onto a balanced BST (AVL or red-black), updating sizes inside rotations. In C++ the GNU policy-based data structures (__gnu_pbds::tree with tree_order_statistics_node_update) provide a ready-made order-statistic set exposing find_by_order(k) (select) and order_of_key(key) (rank), both O(log n). It behaves like an augmented std::set.

8. Complexity Summary

OperationMethodTime
Kth smallest / largestIn-order traversal (early exit)O(h + k), worst O(n)
Select (kth smallest)Augmented BST sizesO(h)
Rank (keys < key)Augmented BST sizesO(h)
Count in range [lo, hi]Two rank queriesO(h)
Insert / delete (size upkeep)Update path nodesO(h)

Here h is the tree height: O(log n) for a balanced tree, O(n) for a degenerate (skewed) one. Augmentation turns a query that was O(n) in the worst case into O(h) regardless of k, at the cost of one integer per node and O(h) maintenance per update.

Rule of thumb: augment a BST with subtree size the moment you need rank or select. It is the smallest possible augmentation, costs only O(h) extra work per insert/delete to maintain, and converts both queries from a full O(n) traversal into a single O(h) root-to-node walk.
