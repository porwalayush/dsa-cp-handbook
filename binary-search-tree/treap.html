Treap (Randomized BST)

Imagine you run a small bakery and want your customers lined up two ways at once — sorted left-to-right by last name so you can find anyone quickly, but also arranged so the person holding the highest lottery ticket number stands at the very front. That sounds impossible to satisfy both at the same time… yet there's exactly one arrangement that does, and a treap is the tree shape that pulls it off.

The name is a portmanteau: tree + heap = treap. Every node carries two values — a key and a randomly chosen priority. The tree is a binary search tree on the keys (left < node < right) and simultaneously a heap on the priorities (a parent's priority always beats its children's). The trick is that the random priorities, not any hand-coded balancing rule, are what keep the tree short.

Core Idea: Balance by Luck, Not by Rules

An AVL tree stays balanced by following strict rules: it measures every node's balance factor and fires off rotations the instant something tips past ±1. A red-black tree does the same with color invariants. Both work hard to guarantee a short tree on every single operation.

A treap takes the opposite, almost cheeky, approach: balance by luck. When you insert a key, you roll a die — assign it a random priority — and then just enforce the heap property. Because the priorities are random, the node that ends up at the root is essentially a random one of your keys, the node at the next level is a random split of each half, and so on. That is precisely how a randomly built BST behaves, and a randomly built BST has expected height O(log n). So with high probability the treap is balanced, even though it never measured a single balance factor.

The key insight: a treap's shape is completely determined by the (key, priority) pairs. Two treaps built from the same set of pairs are identical, no matter what order you insert them. Since the priorities are random, that fixed shape is a random BST — balanced in expectation, with the chance of being badly skewed shrinking exponentially.

We will use a max-heap on priority throughout this page (the largest priority sits at the root). A min-heap version works identically with the comparisons flipped — pick one convention and stay consistent.

A treap.  Notation:  key | priority      (BST by key, MAX-heap by priority)

                 (40 | 95)
                /          \
        (20 | 70)          (60 | 55)
        /      \                  \
 (10 | 30)  (30 | 25)          (70 | 40)

BST check (keys, in-order):  10 20 30 40 60 70   -> sorted ✓
Heap check (priorities):     every parent > its children
        95 > 70, 95 > 55, 70 > 30, 70 > 25, 55 > 40   ✓

Insert via Rotations

Insertion is a two-step dance, and step one is something you already know. First, insert the key as an ordinary BST leaf — walk down comparing keys, hang the new node where it belongs. At this point the BST property is satisfied but the heap property might be broken: your shiny new node was handed a random priority, and that priority could easily be bigger than its parent's.

Second, bubble the node up with rotations while its priority beats its parent's. This is the exact same leftRotate / rightRotate primitive from AVL trees — a rotation that re-hangs three nodes without disturbing the in-order key ordering. Each rotation lifts the node one level toward the root and pushes the weaker-priority parent down. You stop the moment the node's parent has a higher priority (or the node becomes the root).

🎬 Insert a Key → Roll a Priority → Bubble It Up
Type a key and press Insert. We assign it a random priority, drop it in as a BST leaf, then rotate it up step-by-step while its priority beats its parent's (max-heap). Each node shows key on top and its priority below. Watch the in-order key order never change.

Because the new node only travels upward and stops as soon as the heap property holds, the number of rotations equals the number of levels it climbs — on a balanced treap that is O(log n) in expectation.

struct Node {
    int key, priority;
    Node *left = nullptr, *right = nullptr;
    Node(int k) : key(k), priority(rand()) {}
};

// Right rotation: left child rises, node sinks right.
Node* rightRotate(Node* y) {
    Node* x = y->left;
    y->left = x->right;
    x->right = y;
    return x;                 // x is the new subtree root
}
// Left rotation: right child rises, node sinks left.
Node* leftRotate(Node* x) {
    Node* y = x->right;
    x->right = y->left;
    y->left = x;
    return y;
}

// Insert as BST leaf, then rotate up while child wins (max-heap).
Node* insert(Node* node, int key) {
    if (!node) return new Node(key);
    if (key < node->key) {
        node->left = insert(node->left, key);
        // left child climbed; fix heap if it now out-ranks us
        if (node->left->priority > node->priority)
            node = rightRotate(node);
    } else if (key > node->key) {
        node->right = insert(node->right, key);
        if (node->right->priority > node->priority)
            node = leftRotate(node);
    }
    return node;                // duplicate keys ignored
}

class Node {
    int key, priority;
    Node left, right;
    Node(int k, Random rng) { key = k; priority = rng.nextInt(); }
}

// Right rotation: left child rises, node sinks right.

    Node x = y.left;
    y.left = x.right;
    x.right = y;
    return x;
}
// Left rotation: right child rises, node sinks left.
Node leftRotate(Node x) {
    Node y = x.right;
    x.right = y.left;
    y.left = x;
    return y;
}

// Insert as BST leaf, then rotate up while child wins (max-heap).
Node insert(Node node, int key, Random rng) {
    if (node == null) return new Node(key, rng);
    if (key < node.key) {
        node.left = insert(node.left, key, rng);
        if (node.left.priority > node.priority)
            node = rightRotate(node);
    } else if (key > node.key) {
        node.right = insert(node.right, key, rng);
        if (node.right.priority > node.priority)
            node = leftRotate(node);
    }
    return node;
}

import random

class Node:
    def __init__(self, key):
        self.key = key
        self.priority = random.random()   # random priority in [0, 1)
        self.left = None
        self.right = None

# Right rotation: left child rises, node sinks right.
def right_rotate(y):
    x = y.left
    y.left = x.right
    x.right = y
    return x
# Left rotation: right child rises, node sinks left.
def left_rotate(x):
    y = x.right
    x.right = y.left
    y.left = x
    return y

# Insert as BST leaf, then rotate up while child wins (max-heap).
def insert(node, key):
    if node is None:
        return Node(key)
    if key < node.key:
        node.left = insert(node.left, key)
        if node.left.priority > node.priority:
            node = right_rotate(node)
    elif key > node.key:
        node.right = insert(node.right, key)
        if node.right.priority > node.priority:
            node = left_rotate(node)
    return node

Delete via Rotations

Deletion is the mirror image of insertion, and it's wonderfully simple: instead of bubbling a node up, you sink the target node down until it's a leaf, then snip it off. The catch is that you must sink it in a way that preserves the heap property the whole time.

The rule: at each step, look at the node's two children and rotate toward the child with the higher priority — that child rises to take the node's place (keeping the strongest priority on top), and the target sinks one level on the other side. Repeat until the node has no children, then delete it. If the node has only one child, rotate that child up. If it's already a leaf, just remove it.

delete(40) — sink it down toward the higher-priority child

        (40 | 95)                 (20 | 70)                  (20 | 70)
        /       \                 /       \                  /       \
 (20 | 70)  (60 | 55)   =>  (10|30)  (40 | 95)    =>   (10|30)  (60 | 55)
   /                                      \                          \
(10 | 30)                            (60 | 55)                    (40 | 95)  <- now a leaf

so rotate RIGHT: 20 rises, 40 sinks right.       its higher-priority child until

// Sink the matched node down toward the stronger child, then remove it.
Node* erase(Node* node, int key) {
    if (!node) return nullptr;
    if (key < node->key)       node->left  = erase(node->left, key);
    else if (key > node->key)  node->right = erase(node->right, key);
    else {
        // Found it. Base cases: zero or one child.
        if (!node->left)  { Node* r = node->right; delete node; return r; }
        if (!node->right) { Node* l = node->left;  delete node; return l; }
        // Two children: rotate the stronger-priority child up, recurse.
        if (node->left->priority > node->right->priority) {
            node = rightRotate(node);
            node->right = erase(node->right, key);
        } else {
            node = leftRotate(node);
            node->left = erase(node->left, key);
        }
    }
    return node;
}

// Sink the matched node down toward the stronger child, then remove it.
Node erase(Node node, int key) {
    if (node == null) return null;
    if (key < node.key)       node.left  = erase(node.left, key);
    else if (key > node.key)  node.right = erase(node.right, key);
    else {
        // Found it. Base cases: zero or one child.
        if (node.left == null)  return node.right;
        if (node.right == null) return node.left;
        // Two children: rotate the stronger-priority child up, recurse.
        if (node.left.priority > node.right.priority) {
            node = rightRotate(node);
            node.right = erase(node.right, key);
        } else {
            node = leftRotate(node);
            node.left = erase(node.left, key);
        }
    }
    return node;
}

def erase(node, key):

        node.left = erase(node.left, key)

            node.right = erase(node.right, key)
        else:
            node = left_rotate(node)
            node.left = erase(node.left, key)
    return node

Split and Merge: The Treap's Superpower

Here is where treaps leave AVL and red-black trees in the dust. A treap supports two extra operations that are awkward or impossible on other balanced trees, yet they're short to code and run in O(log n):

  • split(T, k) → (L, R): cut treap T into two treaps where every key in L is ≤ k and every key in R is > k. Think of it as slicing a sorted deck of cards at a chosen value.
  • merge(L, R) → T: glue two treaps back together, assuming every key in L is smaller than every key in R (the two ranges don't overlap). This is the inverse of split.

The beautiful part: once you have split and merge, you can express everything else with them. Insert(x) = split at x, merge the new node between the halves. Delete(x) = split out the single node and merge what's left. Both operations work by walking one root-to-leaf path and respecting the heap priority at each step, so they stay logarithmic.

// Split T into L (keys <= k) and R (keys > k).
void split(Node* t, int k, Node*& L, Node*& R) {
    if (!t) { L = R = nullptr; return; }
    if (t->key <= k) {           // t and its left go to L
        split(t->right, k, t->right, R);
        L = t;
    } else {                     // t and its right go to R
        split(t->left, k, L, t->left);
        R = t;
    }
}

// Merge L and R (all keys in L < all keys in R). Keep higher priority on top.
Node* merge(Node* L, Node* R) {
    if (!L || !R) return L ? L : R;
    if (L->priority > R->priority) {
        L->right = merge(L->right, R);
        return L;
    } else {
        R->left = merge(L, R->left);
        return R;
    }
}

// split returns a 2-element array {L, R}: L has keys <= k, R has keys > k.
Node[] split(Node t, int k) {
    if (t == null) return new Node[]{null, null};
    if (t.key <= k) {
        Node[] rs = split(t.right, k);
        t.right = rs[0];
        return new Node[]{t, rs[1]};
    } else {
        Node[] ls = split(t.left, k);
        t.left = ls[1];
        return new Node[]{ls[0], t};
    }
}

// Merge L and R (all keys in L < all keys in R).
Node merge(Node L, Node R) {
    if (L == null || R == null) return L != null ? L : R;
    if (L.priority > R.priority) {
        L.right = merge(L.right, R);
        return L;
    } else {
        R.left = merge(L, R.left);
        return R;
    }
}

# Split t into (L, R): L has keys <= k, R has keys > k.
def split(t, k):
    if t is None:
        return None, None
    if t.key <= k:
        t.right, R = split(t.right, k)
        return t, R
    else:
        L, t.left = split(t.left, k)
        return L, t

# Merge L and R (all keys in L < all keys in R).
def merge(L, R):
    if L is None or R is None:
        return L if L else R
    if L.priority > R.priority:
        L.right = merge(L.right, R)
        return L
    else:
        R.left = merge(L, R.left)
        return R

Implicit Treaps: Treating the Tree Like an Array

Now the mind-bending variant. In an implicit treap, you throw away the key entirely and instead let a node's position in the in-order traversal be its "key." In other words, the tree behaves like a dynamic array indexed 0, 1, 2, … — the leftmost node is index 0, the next in-order is index 1, and so on. You never store these indices; you recover them on the fly by keeping a subtree size count in every node.

Why bother? Because split and merge now let you do things to ranges of an array that are normally impossible in better than linear time:

  • Insert or erase an element at any position in O(log n) — split at the index, merge the piece in or leave it out.
  • Reverse a whole subarray [l, r] in O(log n) by splitting out that range and flipping a lazy "reverse" flag on its root (children get swapped when you next descend, like a lazy-propagation segment tree).
  • Apply a range update (add a value to every element in a range, find a range min/sum) by combining the size counts with lazy tags.

It's the same split/merge machinery you just saw — the only addition is a size field (and optional lazy flags) that you recompute whenever a node's children change. Don't worry if this feels like a lot at once: the takeaway is simply that a treap can impersonate a resizable array with logarithmic-time range surgery.

Quick mental model: a normal treap is a sorted set keyed by value. An implicit treap is a list keyed by position. Both are the same code; one stores keys, the other stores subtree sizes and lets position play the role of the key.

Why Competitive Programmers Love Treaps

If you read enough competitive-programming editorials you'll see treaps everywhere. Here's why they're a go-to tool:

  • Short to code. A full treap with split/merge is roughly 30 lines — far less than a red-black tree, and you can reconstruct it from memory under contest pressure. Insert and delete are even one-liners on top of split/merge.
  • Order statistics for free. Add a subtree-size field and you can answer "what is the k-th smallest key?" and "how many keys are < x?" (rank) in O(log n).
  • Split and merge. Slicing and gluing ordered sets in logarithmic time enables persistent structures, merging two sets, and divide-and-conquer tricks that ordinary balanced trees can't do cleanly.
  • Implicit-key range ops. The array-like variant handles insert/erase-at-index, range reverse, and range updates — the kind of problems that otherwise need a much fussier data structure.
  • Randomized, so no nasty case analysis. There are no four imbalance cases to memorize and no color invariants to maintain. The randomness does the balancing for you.
Two gotchas: (1) Use a good random source and enough bits of priority — if two nodes collide on the same priority you can break the strict heap order and degrade balance, so prefer a wide range (e.g. a 32/64-bit RNG). (2) The O(log n) bounds are expected, not worst-case. With high probability the tree is balanced, but an adversary who can see your random seed could theoretically force a bad shape — seed from a non-predictable source in adversarial settings.

Worst case caveat: like a hash table, a treap's guarantees are probabilistic. A single operation could in principle be O(n) if the priorities land pathologically, but the probability of the tree being more than a constant factor taller than log₂n is exponentially small. In practice treaps are reliably balanced.

Treap vs AVL vs Red-Black

Treap

  • Balance by random priority (expected O(log n))
  • Very short to implement; easy split/merge
  • Native order statistics & implicit-key range ops
  • Bounds are expected, not worst-case

AVL / Red-Black

  • Balance by strict rules (guaranteed O(log n))
  • More code; split/merge is awkward
  • Worst-case guarantees, no randomness
  • Backbone of standard library ordered maps
Rule of thumb: reach for an AVL or red-black tree when you need worst-case guarantees and a battle-tested library map. Reach for a treap when you want a balanced BST you can write from scratch quickly, especially if you need split/merge, order statistics, or array-like range operations — which is exactly the toolkit competitive programmers prize.
