Greedy Algorithms

A greedy algorithm builds an answer one step at a time, always grabbing the choice that looks best right now — and never reconsidering. Simple, fast, and surprisingly often correct… but not always.

The Greedy Mindset

   while problem not solved:
       pick the LOCALLY best option available
       commit to it (never undo)

   Hope: a sequence of local best choices = the global best.

   This hope is TRUE only for certain problems.
   You must PROVE it — usually with an "exchange argument".
  • Greedy-choice property: a global optimum can be reached by local optimal choices.
  • Optimal substructure: the best solution contains best solutions to subproblems.
  • If both hold → greedy works. If not → you probably need DP.

Interactive: Activity Selection

You have one room and many meetings, each with a start and end time. Goal: fit the most non-overlapping meetings. The winning greedy rule: always pick the meeting that finishes earliest, then skip everything that clashes with it.

0h2h4h6h8h10h

Why "earliest finish" wins

   Picking the meeting that ends soonest leaves the MOST
   room for everything after it.

   Sort by end time, then sweep:
     keep a meeting if its start ≥ last chosen end.

   Exchange argument: if some optimal solution didn't pick
   the earliest-finishing activity, you can swap it in
   without losing any meetings — so greedy is at least as good.

Code: Activity Selection

// each activity = {start, end}
int maxActivities(vector<pair<int,int>>& acts) {
    sort(acts.begin(), acts.end(),
         [](auto& a, auto& b){ return a.second < b.second; }); // by END time
    int count = 0, lastEnd = -1;
    for (auto& [s, e] : acts) {
        if (s >= lastEnd) {     // no clash → take it
            count++;
            lastEnd = e;
        }
    }
    return count;
}
int maxActivities(int[][] acts) {
    Arrays.sort(acts, (a, b) -> a[1] - b[1]); // by END time
    int count = 0, lastEnd = -1;
    for (int[] a : acts) {
        if (a[0] >= lastEnd) {  // no clash → take it
            count++;
            lastEnd = a[1];
        }
    }
    return count;
}
def max_activities(acts):
    acts.sort(key=lambda x: x[1])   # sort by END time
    count, last_end = 0, -1
    for s, e in acts:
        if s >= last_end:          # no clash → take it
            count += 1
            last_end = e
    return count

When Greedy FAILS

The classic trap: coin change. Greedy (take the biggest coin first) works for normal coins but breaks for weird denominations.

  • Coins = {1, 3, 4}, make 6.
  • Greedy: 4 + 1 + 1 = 3 coins.
  • Optimal: 3 + 3 = 2 coins. ❌ Greedy lost!
  • Here you need Dynamic Programming instead.

Greedy vs Dynamic Programming

Use Greedy when…

  • A local best choice is provably safe
  • You can sort + sweep
  • Speed matters: O(n log n)

Use DP when…

  • Local choices can backfire later
  • You must weigh many combinations
  • Subproblems overlap

Famous Greedy Algorithms

Activity Selection — most non-overlapping intervals (shown above).

Huffman Coding — optimal prefix codes for compression.

Dijkstra & Prim — shortest paths & minimum spanning trees.

Fractional Knapsack — take highest value/weight ratio first.

Job Sequencing — schedule deadlines for max profit.

How to Trust a Greedy Solution

1. State the greedy rule clearly (e.g. "earliest finish first").

2. Exchange argument: show any optimal solution can be morphed into the greedy one without getting worse.

3. Try to break it: hunt for a small counterexample. If you can't find one after honest effort, you're likely safe.

4. When unsure → fall back to DP and compare on small inputs.
