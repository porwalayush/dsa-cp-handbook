Recursion & Backtracking

Recursion is a function that calls itself on a smaller version of the problem. Backtracking adds one move: when a choice leads to a dead end, undo it and try the next one.

Recursion in Plain Words

Every recursive function needs exactly two things:

  1. Base case: the smallest problem you can answer immediately (stops the recursion).
  2. Recursive case: solve a smaller piece, then build on it.
  • Missing base case → infinite recursion → stack overflow.
  • Each call gets its own copy of local variables, stacked on the call stack.

The Call Stack: factorial(4)

   Calls go DOWN (push), results come back UP (pop):

   factorial(4)                      returns 4 * 6 = 24
     └─ factorial(3)                 returns 3 * 2 = 6
          └─ factorial(2)            returns 2 * 1 = 2
               └─ factorial(1)       returns 1  (BASE CASE)

   The stack unwinds bottom-to-top, multiplying as it returns.

Backtracking = Recursion + Undo

The template is always the same three steps, repeated for each choice:

   for each choice:
       1. CHOOSE   → make the move
       2. EXPLORE  → recurse deeper
       3. UNDO     → take the move back (backtrack!)

   "Try it. If it doesn't work out, pretend it never happened."

Interactive: N-Queens Backtracking

Place N queens on an N×N board so none attack each other. Watch the solver try a square, place it if safe, hit a dead end, and backtrack.

Code: N-Queens

bool safe(vector<int>& pos, int row, int col) {
    for (int r = 0; r < row; r++) {
        int c = pos[r];
        if (c == col) return false;              // same column
        if (abs(c - col) == abs(r - row)) return false; // diagonal
    }
    return true;
}
void solve(vector<int>& pos, int row, int n, int& count) {
    if (row == n) { count++; return; }     // BASE: all placed
    for (int col = 0; col < n; col++) {
        if (safe(pos, row, col)) {
            pos[row] = col;             // 1. CHOOSE
            solve(pos, row + 1, n, count); // 2. EXPLORE
            pos[row] = -1;            // 3. UNDO (backtrack)
        }
    }
}
boolean safe(int[] pos, int row, int col) {
    for (int r = 0; r < row; r++) {
        int c = pos[r];
        if (c == col) return false;
        if (Math.abs(c - col) == Math.abs(r - row)) return false;
    }
    return true;
}

    if (row == n) { count++; return; }  // BASE
    for (int col = 0; col < n; col++) {
        if (safe(pos, row, col)) {
            pos[row] = col;           // CHOOSE

def safe(pos, row, col):
    for r in range(row):
        c = pos[r]
        if c == col: return False            # same column
        if abs(c - col) == abs(r - row): return False  # diagonal
    return True

def solve(pos, row, n):
    if row == n:                  # BASE: all placed
        return 1
    total = 0
    for col in range(n):
        if safe(pos, row, col):
            pos[row] = col          # CHOOSE
            total += solve(pos, row + 1, n)  # EXPLORE
            pos[row] = -1         # UNDO (backtrack)
    return total

Recursion vs Iteration

Recursion shines when

  • The problem is naturally nested (trees, grids)
  • You need to explore many branches
  • "Try all possibilities" / generate combinations

Watch out for

  • Deep recursion → stack overflow (n > ~10⁴)
  • Repeated subproblems → add memoization (that's DP!)
  • Function-call overhead vs a simple loop

Classic Backtracking Problems

Permutations & Combinations — generate every arrangement / subset.

N-Queens — place non-attacking queens (shown above).

Sudoku Solver — fill cells, backtrack on contradictions.

Word Search / Maze — explore paths, undo when blocked.

Subset Sum / Partition — include or exclude each element.

Pruning: Making Backtracking Fast

  • Prune early: abandon a branch the moment it can't possibly succeed (like safe() rejecting a square before recursing).
  • Order choices smartly: try the most constrained option first.
  • Good pruning can turn an impossible search into an instant one.

Common Pitfalls

1. Forgetting to undo: if you CHOOSE but never UNDO, later branches see stale state.

2. No base case: infinite recursion → crash.

3. Sharing mutable state wrong: be careful passing lists/arrays by reference.

4. Not pruning: exploring obviously-dead branches makes it exponentially slow.
