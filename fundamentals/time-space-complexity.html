Time & Space Complexity

Before you write a single line, you should be able to answer one question: "Will this be fast enough?" Complexity analysis is how we answer it — without ever running the code.

The Big Idea (in plain words)

Big-O describes how the work grows as the input grows. We don't care about exact seconds — those change with the computer. We care about the shape of the growth.

  • Think of input size as n (array length, number of nodes, etc.).
  • Big-O answers: "if n doubles, how much more work do I do?"
  • We always drop constants and smaller terms: 3n + 5 → O(n).

The Three Notations

   Big-O    (O)  →  upper bound   →  "no worse than"   →  worst case
   Big-Theta (Θ) →  tight bound   →  "exactly this"    →  typical case
   Big-Omega (Ω) →  lower bound   →  "no better than"  →  best case

   Example: linear search in an array of n items
     best  case: item is first   → Ω(1)
     worst case: item is last     → O(n)
     we usually quote the worst case → O(n)

How to Read Code & Spot the Big-O

1. A single loop over n → O(n)

2. A loop inside a loop → O(n²)

3. Halving each step (i = i / 2) → O(log n)

4. Two separate loops → O(n) + O(n) = O(n), not O(n²)

5. Recursion that splits in half and does O(n) work → O(n log n)

Code: Spotting Complexity

// O(1) - constant: one operation no matter the size
int first = arr[0];

// O(n) - linear: touch every element once
for (int i = 0; i < n; i++) sum += arr[i];

// O(n^2) - quadratic: every pair
for (int i = 0; i < n; i++)
    for (int j = 0; j < n; j++)
        check(arr[i], arr[j]);

// O(log n) - logarithmic: halve the range each step
while (lo < hi) { int mid = (lo + hi) / 2; /* ... */ }

Space Complexity

Same idea, but for memory instead of time. Count the extra space your algorithm allocates as n grows — not counting the input itself.

O(1) space

  • A few variables (counters, pointers)
  • In-place swaps
  • Two-pointer techniques

O(n) space

  • A copy of the array
  • A hash map of all elements
  • Recursion depth of n (call stack!)

Amortized Analysis (the "average over many ops" trick)

Some operations are usually cheap but occasionally expensive. Amortized analysis spreads the rare expensive cost across all the cheap ones.

  • Dynamic array push: usually O(1). When full, it doubles and copies — O(n) that once. Averaged out → O(1) amortized.
  • Because doubling happens rarely, the total cost of n pushes is still O(n).

The Practical Rule of Thumb

  • A typical judge allows ~10⁸ simple operations per second.
  • n ≤ 10 → even O(n!) is fine.   n ≤ 20 → O(2ⁿ) ok.
  • n ≤ 5000 → O(n²) ok.   n ≤ 10⁶ → need O(n) or O(n log n).
  • n ≤ 10⁹ → need O(log n) or O(1).

Common Pitfalls

1. Hidden loops: s.find(), list.contains(), string concatenation in a loop can each be O(n).

2. Recursion stack: deep recursion uses O(depth) memory — easy to forget.

3. Average vs worst: hash maps are O(1) average but O(n) worst case on collisions.

4. Dropping the wrong term: O(n + m) is NOT always O(n) — keep both if n and m are independent.
