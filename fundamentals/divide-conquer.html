Divide & Conquer

A three-word recipe for hard problems: divide the problem into smaller pieces, conquer each piece (usually by recursion), then combine the answers into one.

The Three Steps

   1. DIVIDE   →  split the input into 2+ smaller subproblems
   2. CONQUER  →  solve each subproblem recursively
                  (base case: small enough to solve directly)
   3. COMBINE  →  merge the sub-answers into the final answer

   The magic is in step 3 — a clever COMBINE is what makes it fast.
  • If subproblems don't overlap, plain divide & conquer works.
  • If they do overlap (same subproblem solved many times), you want Dynamic Programming instead.

Interactive: Merge Sort, Step by Step

Merge sort is the poster child of divide & conquer. It splits the array down to single elements (divide), then merges them back together in sorted order (combine).

Why It's O(n log n)

   log n levels of splitting (each cut halves the size)
   ─────────────────────────────────────────────
   [8 7 6 5 4 3 2 1]                  level 0
   [8 7 6 5] [4 3 2 1]                level 1
   [8 7][6 5][4 3][2 1]               level 2
   [8][7][6][5][4][3][2][1]           level 3 (base)

   Each LEVEL does O(n) total work to merge.
   Number of levels = log₂ n.
   Total = O(n) × O(log n) = O(n log n)

Code: Merge Sort

void merge(vector<int>& a, int l, int m, int r) {
    vector<int> tmp;
    int i = l, j = m + 1;
    while (i <= m && j <= r)            // COMBINE step
        tmp.push_back(a[i] <= a[j] ? a[i++] : a[j++]);
    while (i <= m) tmp.push_back(a[i++]);
    while (j <= r) tmp.push_back(a[j++]);
    for (int k = 0; k < tmp.size(); k++) a[l + k] = tmp[k];
}
void mergeSort(vector<int>& a, int l, int r) {
    if (l >= r) return;            // base case
    int m = (l + r) / 2;          // DIVIDE
    mergeSort(a, l, m);             // CONQUER left
    mergeSort(a, m + 1, r);         // CONQUER right
    merge(a, l, m, r);              // COMBINE
}
void merge(int[] a, int l, int m, int r) {
    int[] tmp = new int[r - l + 1];
    int i = l, j = m + 1, k = 0;
    while (i <= m && j <= r)
        tmp[k++] = (a[i] <= a[j]) ? a[i++] : a[j++];
    while (i <= m) tmp[k++] = a[i++];
    while (j <= r) tmp[k++] = a[j++];
    System.arraycopy(tmp, 0, a, l, tmp.length);
}
void mergeSort(int[] a, int l, int r) {
    if (l >= r) return;        // base case
    int m = (l + r) / 2;      // DIVIDE
    mergeSort(a, l, m);         // CONQUER left
    mergeSort(a, m + 1, r);     // CONQUER right
    merge(a, l, m, r);          // COMBINE
}
def merge_sort(a):
    if len(a) <= 1:           # base case
        return a
    mid = len(a) // 2        # DIVIDE
    left = merge_sort(a[:mid])   # CONQUER left
    right = merge_sort(a[mid:])  # CONQUER right
    # COMBINE
    res, i, j = [], 0, 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            res.append(left[i]); i += 1
        else:
            res.append(right[j]); j += 1
    res.extend(left[i:]); res.extend(right[j:])
    return res

The Master Theorem (made simple)

A formula to instantly find the Big-O of a divide & conquer recurrence of the form T(n) = a·T(n/b) + O(nᵈ), where you split into a subproblems of size n/b and do O(nᵈ) work to combine.

   Compare a   vs   b^d :

   a < b^d   →   T(n) = O(n^d)         (combine dominates)
   a = b^d   →   T(n) = O(n^d · log n) (balanced)
   a > b^d   →   T(n) = O(n^(log_b a)) (splitting dominates)

   Merge sort:   a=2, b=2, d=1  →  2 = 2^1  →  O(n log n) ✓
   Binary search: a=1, b=2, d=0 →  1 = 2^0  →  O(log n)   ✓

Where You'll Use It

Merge Sort & Quick Sort — sorting in O(n log n).

Binary Search — divide the range, conquer one half.

Karatsuba / FFT — fast multiplication of big numbers / polynomials.

Closest Pair of Points — geometry in O(n log n).

Count Inversions — a free byproduct of merge sort.

Divide & Conquer vs Dynamic Programming

Divide & Conquer

  • Subproblems are independent
  • Each solved once, no overlap
  • Merge sort, binary search

Dynamic Programming

  • Subproblems overlap
  • Cache results to avoid recompute
  • Fibonacci, LCS, knapsack

Common Pitfalls

1. Weak combine step: the whole speedup lives here — a sloppy merge ruins the complexity.

2. Off-by-one in the split: mid = (l+r)/2 and the boundaries must line up exactly.

3. Missing base case: forgetting l >= r → infinite recursion.

4. Using D&C on overlapping subproblems: exponential blowup — use DP instead.
