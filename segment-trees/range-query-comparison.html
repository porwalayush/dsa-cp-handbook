Range Query Data Structures Compared

Choosing the right range query structure can make the difference between TLE and AC. This page compares the five main approaches: Prefix Sums, Fenwick Tree (BIT), Segment Tree, Sparse Table, and Sqrt Decomposition. Each has a sweet spot where it outperforms the rest.

Overview of Structures

1. Prefix Sums

The simplest range query technique. Precompute cumulative sums so that any range sum [l, r] is answered in O(1) as prefix[r] - prefix[l-1]. Works only for static arrays (no updates). Extends to 2D with inclusion-exclusion.

2. Fenwick Tree (Binary Indexed Tree)

A compact tree stored in a flat array that supports point updates and prefix queries in O(log n). Relies on the binary representation of indices. Best for invertible operations like sum and XOR where prefix decomposition is possible.

3. Segment Tree

The most versatile range query structure. Supports any associative operation (sum, min, max, GCD, etc.) with O(log n) queries and updates. Can be extended with lazy propagation for range updates and with persistence for historical queries.

4. Sparse Table

Precomputes answers for all power-of-two length intervals. Achieves O(1) queries for idempotent operations (min, max, GCD) on static arrays. Uses O(n log n) space and build time. Cannot handle updates.

5. Sqrt Decomposition

Divides the array into blocks of size ~sqrt(n). Simple to implement and flexible. Particularly useful for problems that are hard to solve with trees (e.g., Mo's algorithm for offline range queries, distinct element counting).

When BIT Wins Over Segment Tree

If your problem only requires point updates + prefix/range sum queries (or XOR, or any invertible operation), the Fenwick Tree is almost always the better choice:

BIT Advantages

  • Simpler code: 5-10 lines vs 30-50 lines
  • Faster constant: ~2-3x faster in practice due to fewer cache misses and simpler operations
  • Less memory: single array of size n+1 vs 4n array
  • Lower bug risk: fewer edge cases and off-by-one errors
  • Easy to extend: 2D BIT is straightforward

Typical BIT Problems

  • Range sum with point updates
  • Inversion counting
  • Coordinate compression + frequency
  • Order statistics (kth smallest)
  • 2D prefix sums with updates
  • Range update + point query (difference BIT)
// BIT: entire implementation in ~10 lines
int bit[N];
void update(int i, int v) { for (; i < N; i += i & -i) bit[i] += v; }
int query(int i) { int s = 0; for (; i > 0; i -= i & -i) s += bit[i]; return s; }
int query(int l, int r) { return query(r) - query(l - 1); }

When Segment Tree is Necessary

The Segment Tree becomes the only practical choice when:

  • Non-invertible operations: min, max, GCD cannot use prefix decomposition, so BIT cannot help. Segment Tree handles any associative merge function.
  • Lazy propagation: Range updates + range queries (e.g., "add v to all elements in [l,r], then query sum of [l,r]") require lazy propagation, which only Segment Tree supports naturally.
  • Persistence: You need to query historical versions of the array. Persistent Segment Tree creates new path nodes per update in O(log n) time and space.
  • Complex merge operations: Maximum subarray sum, count of distinct values, merging intervals -- these need custom node structures that only Segment Tree accommodates.
  • Segment Tree Beats: Maintaining min/max with "set all values in range to min(val, x)" operations (Ji Driver Segment Tree).
// Segment Tree with Lazy Propagation (range add + range sum)
struct SegTree {
    int n;
    vector<long long> tree, lazy;

    SegTree(int n) : n(n), tree(4 * n, 0), lazy(4 * n, 0) {}

    void push(int v, int tl, int tr) {
        if (lazy[v]) {
            tree[v] += lazy[v] * (tr - tl + 1);
            if (tl != tr) {
                lazy[2*v] += lazy[v];
                lazy[2*v+1] += lazy[v];
            }
            lazy[v] = 0;
        }
    }

    void update(int v, int tl, int tr, int l, int r, long long val) {
        push(v, tl, tr);
        if (l > tr || r < tl) return;
        if (l <= tl && tr <= r) {
            lazy[v] += val;
            push(v, tl, tr);
            return;
        }
        int mid = (tl + tr) / 2;
        update(2*v, tl, mid, l, r, val);
        update(2*v+1, mid+1, tr, l, r, val);
        tree[v] = tree[2*v] + tree[2*v+1];
    }

    long long query(int v, int tl, int tr, int l, int r) {
        push(v, tl, tr);
        if (l > tr || r < tl) return 0;
        if (l <= tl && tr <= r) return tree[v];
        int mid = (tl + tr) / 2;
        return query(2*v, tl, mid, l, r) + query(2*v+1, mid+1, tr, l, r);
    }
};

class SegTree {
    int n;
    long[] tree, lazy;

    SegTree(int n) {
        this.n = n;
        tree = new long[4 * n];
        lazy = new long[4 * n];
    }

    void push(int v, int tl, int tr) {
        if (lazy[v] != 0) {
            tree[v] += lazy[v] * (tr - tl + 1);
            if (tl != tr) {
                lazy[2*v] += lazy[v];
                lazy[2*v+1] += lazy[v];
            }
            lazy[v] = 0;
        }
    }

    void update(int v, int tl, int tr, int l, int r, long val) {
        push(v, tl, tr);
        if (l > tr || r < tl) return;
        if (l <= tl && tr <= r) { lazy[v] += val; push(v, tl, tr); return; }
        int mid = (tl + tr) / 2;
        update(2*v, tl, mid, l, r, val);
        update(2*v+1, mid+1, tr, l, r, val);
        tree[v] = tree[2*v] + tree[2*v+1];
    }

    long query(int v, int tl, int tr, int l, int r) {
        push(v, tl, tr);
        if (l > tr || r < tl) return 0;
        if (l <= tl && tr <= r) return tree[v];
        int mid = (tl + tr) / 2;
        return query(2*v, tl, mid, l, r) + query(2*v+1, mid+1, tr, l, r);
    }
}

class SegTree:
    def __init__(self, n):
        self.n = n
        self.tree = [0] * (4 * n)
        self.lazy = [0] * (4 * n)

    def push(self, v, tl, tr):
        if self.lazy[v]:
            self.tree[v] += self.lazy[v] * (tr - tl + 1)
            if tl != tr:
                self.lazy[2*v] += self.lazy[v]
                self.lazy[2*v+1] += self.lazy[v]
            self.lazy[v] = 0

    def update(self, v, tl, tr, l, r, val):
        self.push(v, tl, tr)
        if l > tr or r < tl: return
        if l <= tl and tr <= r:
            self.lazy[v] += val
            self.push(v, tl, tr)
            return
        mid = (tl + tr) // 2
        self.update(2*v, tl, mid, l, r, val)
        self.update(2*v+1, mid+1, tr, l, r, val)
        self.tree[v] = self.tree[2*v] + self.tree[2*v+1]

    def query(self, v, tl, tr, l, r):
        self.push(v, tl, tr)
        if l > tr or r < tl: return 0
        if l <= tl and tr <= r: return self.tree[v]
        mid = (tl + tr) // 2
        return self.query(2*v, tl, mid, l, r) + self.query(2*v+1, mid+1, tr, l, r)

When Sqrt Decomposition is Useful

Sqrt Decomposition often gets overlooked because its O(sqrt n) per query seems slow compared to O(log n). But it shines in specific scenarios:

  • Mo's Algorithm: For offline range queries where the answer can be maintained incrementally (add/remove one element). Answers Q queries in O((N + Q) * sqrt(N)). Classic use: count distinct elements in range.
  • Simpler implementation: When the merge operation is complex and hard to express in a segment tree node, sqrt decomposition lets you process partial blocks with brute force and full blocks with precomputed answers.
  • Cache-friendly: Processing blocks of contiguous memory can be faster in practice than tree pointer-chasing for certain problems.
  • Update flexibility: Some operations (like "sort a range") are trivial with sqrt decomposition but extremely hard with segment trees.
// Mo's Algorithm: count distinct elements in range queries
int block;
struct Query { int l, r, idx; };

bool cmp(Query& a, Query& b) {
    int ba = a.l / block, bb = b.l / block;
    if (ba != bb) return ba < bb;
    return (ba & 1) ? (a.r > b.r) : (a.r < b.r);  // zigzag optimization
}

int cnt[MAXVAL], cur_ans = 0;
void add(int x) { if (++cnt[x] == 1) cur_ans++; }
void rem(int x) { if (--cnt[x] == 0) cur_ans--; }

void solve(vector<int>& a, vector<Query>& queries) {
    block = max(1, (int)sqrt(a.size()));
    sort(queries.begin(), queries.end(), cmp);

    int cl = 0, cr = -1;
    for (auto& q : queries) {
        while (cr < q.r) add(a[++cr]);
        while (cl > q.l) add(a[--cl]);
        while (cr > q.r) rem(a[cr--]);
        while (cl < q.l) rem(a[cl++]);
        ans[q.idx] = cur_ans;
    }
}

// Mo's Algorithm: count distinct elements in range queries
int block;
int[] cnt = new int[MAXVAL];
int curAns = 0;

void add(int x) { if (++cnt[x] == 1) curAns++; }
void rem(int x) { if (--cnt[x] == 0) curAns--; }

void solve(int[] a, int[][] queries) {
    block = Math.max(1, (int) Math.sqrt(a.length));
    // Sort queries by (l/block, r) with zigzag
    Arrays.sort(queries, (p, q) -> {
        int ba = p[0] / block, bb = q[0] / block;
        if (ba != bb) return ba - bb;
        return (ba % 2 == 0) ? p[1] - q[1] : q[1] - p[1];
    });

    int cl = 0, cr = -1;
    for (int[] q : queries) {
        while (cr < q[1]) add(a[++cr]);
        while (cl > q[0]) add(a[--cl]);
        while (cr > q[1]) rem(a[cr--]);
        while (cl < q[0]) rem(a[cl++]);
        ans[q[2]] = curAns;
    }
}

# Mo's Algorithm: count distinct elements in range queries
import math

def solve(a, queries):
    n = len(a)
    block = max(1, int(math.sqrt(n)))

    # Sort by (l // block, r) with zigzag
    def mo_cmp(q):
        b = q[0] // block
        return (b, q[1] if b % 2 == 0 else -q[1])

    queries = sorted(enumerate(queries), key=lambda x: mo_cmp(x[1]))

    cnt = [0] * (max(a) + 1)
    cur_ans = 0
    cl, cr = 0, -1
    ans = [0] * len(queries)

    def add(x):
        nonlocal cur_ans
        cnt[x] += 1
        if cnt[x] == 1: cur_ans += 1

    def rem(x):
        nonlocal cur_ans
        cnt[x] -= 1
        if cnt[x] == 0: cur_ans -= 1

    for idx, (l, r) in queries:
        while cr < r: cr += 1; add(a[cr])
        while cl > l: cl -= 1; add(a[cl])
        while cr > r: rem(a[cr]); cr -= 1
        while cl < l: rem(a[cl]); cl += 1
        ans[idx] = cur_ans
    return ans

Static range min Sparse Table ~100ms
Point update + range sum Fenwick Tree ~200ms
Point update + range min Segment Tree ~400ms
Range update + range sum Segment Tree + Lazy ~500ms
Offline distinct count Mo's Algorithm ~800ms
Quick Decision Rules:
  • No updates? Use Prefix Sums (sum) or Sparse Table (min/max/GCD).
  • Point updates + sum/XOR? Use Fenwick Tree -- it is simpler and faster.
  • Point updates + min/max? Use Segment Tree -- BIT cannot do non-invertible ops.
  • Range updates? Use Segment Tree with Lazy Propagation.
  • Offline queries, hard merge? Consider Mo's Algorithm (Sqrt Decomposition).
  • Need persistence? Persistent Segment Tree is your only real option.
  • When in doubt? Segment Tree handles everything (just with more code).
