Offline Algorithms & CDQ Divide-and-Conquer

When all queries are known up front, you can reorder and batch them to beat any online approach. CDQ divide-and-conquer is the flagship: it handles multi-dimensional "dominance" problems by splitting on time and letting the left half's updates answer the right half's queries.

Online vs Offline

   ONLINE   — must answer each query before seeing the next.
              (real-time systems; harder constraints)

   OFFLINE  — read ALL queries first, then answer in any order.
              Lets you: sort by a coordinate, sweep, batch,
              use Mo's algorithm, or recurse on time (CDQ).

CDQ in One Picture (3D dominance)

   Count, for each point, how many earlier points are
   ≤ it in ALL dimensions (x, y, z).

   1. Sort by the 1st dimension (time / x).
   2. DIVIDE into left half L and right half R.
   3. Recurse on L, recurse on R.
   4. CONQUER across the split: only L's points can affect R's
      (they're already ≤ in dim 1). Merge by dim 2; use a BIT
      on dim 3 to count contributions L → R.

   Three nested "sort + sweep + BIT" layers → O(n log²n).

Interactive: Count Dominating Pairs (2D)

For each point we count how many earlier points have both x ≤ and y ≤ it. CDQ splits the timeline; the left half contributes to the right half across the merge. Step through to see the cross-contribution.

Code: CDQ Divide-and-Conquer (2D counting)

// points sorted by x (time). Count, per point, earlier pts
// with x <= and y <= (here via merge-sort on y).
struct Pt { int x, y, ans = 0; };
vector<Pt> p;

void cdq(int lo, int hi) {
    if (hi - lo <= 1) return;
    int mid = (lo + hi) / 2;
    cdq(lo, mid); cdq(mid, hi);
    // merge by y; left contributes to right
    int i = lo, j = mid, cntLeft = 0;
    vector<Pt> tmp;
    while (j < hi) {
        while (i < mid && p[i].y <= p[j].y) { cntLeft++; tmp.push_back(p[i++]); }
        p[j].ans += cntLeft;          // all those have x<= and y<=
        tmp.push_back(p[j++]);
    }
    while (i < mid) tmp.push_back(p[i++]);
    for (int k = lo; k < hi; k++) p[k] = tmp[k - lo];
}
// call: sort p by x, then cdq(0, n). p[i].ans is the count.
# points as [x, y, ans], pre-sorted by x
def cdq(p, lo, hi):
    if hi - lo <= 1:
        return
    mid = (lo + hi) // 2
    cdq(p, lo, mid); cdq(p, mid, hi)
    # merge by y; count left points with y <= current right point
    i, j, cnt_left, tmp = lo, mid, 0, []
    while j < hi:
        while i < mid and p[i][1] <= p[j][1]:
            cnt_left += 1; tmp.append(p[i]); i += 1
        p[j][2] += cnt_left          # x<= (by split) and y<= (by merge)
        tmp.append(p[j]); j += 1
    while i < mid:
        tmp.append(p[i]); i += 1
    p[lo:hi] = tmp

# sort by x first, then cdq(p, 0, len(p))

Other Offline Techniques

  • Sweep line: sort events by coordinate, process with a BIT/segment tree.
  • Mo's algorithm: reorder range queries by √-blocks, maintain an add/remove answer.
  • Offline DSU: process union/query events in a chosen order (e.g. sorted by weight for Kruskal-style).
  • Time-travel / rollback: undo operations to answer queries at different "times".

When to Go Offline

All queries given in the input (no forced ordering / no interaction).

Multi-dimensional dominance ("count points ≤ in every coordinate") → CDQ.

Range queries without updates, or with offline-able updates → Mo's / sweep.

Hard online versions that become easy once you can sort by a key.

Common Pitfalls

1. Equal coordinates: ties in a dimension need a consistent tie-break (often by index/time) to avoid double counting.

2. Mixing updates & queries: in CDQ, only LEFT-half updates may affect RIGHT-half queries — enforce it by the split.

3. Resetting the BIT: undo (don't rebuild) the BIT between merges to keep it O(n log²n).

4. Forgetting it's offline: if the problem forces answers before later queries arrive, none of this applies.
