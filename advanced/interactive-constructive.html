Interactive & Constructive Problems

Two CP genres that reward cleverness over heavy machinery. Interactive problems let you query a judge and adapt; constructive problems ask you to build any valid object meeting the constraints — existence, not optimization.

Interactive Problems

  • You print queries; the judge replies; you adapt — usually within a query budget (e.g. ≤ 20 questions).
  • Flush output after every query (cout.flush() / System.out.flush() / sys.stdout.flush()) or you'll hang.
  • Binary search is the classic tool: each query halves the search space.

Interactive: Guess the Number (binary search)

The judge picked a secret number in [1, 32]. Each "guess" the midpoint and the judge says higher/lower — you find it in ⌈log₂ 32⌉ = 5 queries. Watch the range halve.

Code: Interactive Binary Search

// Judge knows a secret in [1, n]. We may ask "is secret >= x?"
int lo = 1, hi = n;
while (lo < hi) {
    int mid = (lo + hi) / 2;
    cout << "? " << mid << endl;   // endl flushes!
    cout.flush();                // be explicit
    string resp; cin >> resp;     // "yes" or "no"
    if (resp == "yes") hi = mid;  // secret <= mid
    else lo = mid + 1;
}
cout << "! " << lo << endl;       // final answer
// Judge knows a secret in [1, n].
int lo = 1, hi = n;
while (lo < hi) {

    System.out.println("? " + mid);
    System.out.flush();              // crucial!

Constructive Algorithms

   "Output ANY arrangement / sequence / graph that satisfies …"

   Strategy toolkit:
   • Find a PATTERN on small cases (n=1,2,3,4) → generalize.
   • Build greedily / incrementally, fixing violations as you go.
   • Use symmetry (pair up, alternate, mirror).
   • Reduce to a known structure (Euler path, bipartite, parity).
   • Prove impossibility cases FIRST (often a parity/counting bound).

A Worked Constructive Example

Problem: arrange 1..n so no adjacent pair sums to a value divisible by... actually simpler: arrange so adjacent differences are all distinct.

Construction: 1, n, 2, n−1, 3, n−2, … — the zig-zag gives differences n−1, n−2, n−3, …, all distinct. Found by trying n=4 by hand, spotting the pattern, then proving it.

Lesson: small cases → pattern → quick proof. No data structure required.

Constructive Mindset

  • Existence, not optimality: any valid answer scores — don't over-optimize.
  • Brute-force small n locally to discover the pattern, then generalize.
  • Handle impossibility explicitly (print −1 / "NO") — often a parity or counting argument.
  • Symmetry & invariants are your friends — pair, alternate, or balance.

Common Pitfalls

1. Not flushing (interactive): the #1 cause of mysterious TLE/hangs — flush after every query.

2. Exceeding the query budget: count your worst-case queries; binary search is usually optimal.

3. Missing impossible cases (constructive): always check whether a valid construction even exists.

4. Over-engineering: these problems reward a clean pattern, not a heavy algorithm — if you're coding a segment tree, reconsider.
