Binary Trees: Structure & Fundamentals

Think of a family tree or a company's org chart. One person sits at the top. Below them are the people they're directly connected to, and below those are more people, branching out as you go down. A binary tree is the same idea with one simple rule: each box (we call it a node) can point to at most two boxes below it — one on the left and one on the right.

That "left vs. right" matters. Even if a node has only one child, we still keep track of whether it's the left or the right one — they are not interchangeable. This is the only thing that separates a binary tree from a general "anything goes" tree.

Once you can picture that shape, you've got the foundation for a huge slice of computer science. Binary trees power binary search trees and their balanced cousins (AVL, red-black) used in maps and sets, heaps behind priority queues, expression trees in calculators and compilers, Huffman trees in file compression, and more. Learn the vocabulary and the shapes here, and the rest of the tree algorithms become much easier to follow.

Node Structure

A binary tree is built from nodes. Each node holds a value and two pointers/references: left and right, each pointing to a child node or being null/empty when the child is absent. The tree itself is identified by a reference to its root node.

// A single binary tree node
struct TreeNode {
    int val;
    TreeNode* left;
    TreeNode* right;

    // Construct a leaf node (no children)
    TreeNode(int v) : val(v), left(nullptr), right(nullptr) {}
};

// Example: build a tiny tree
//        1
//       / \
//      2   3
TreeNode* root = new TreeNode(1);
root->left  = new TreeNode(2);
root->right = new TreeNode(3);

// A single binary tree node
class TreeNode {
    int val;
    TreeNode left;
    TreeNode right;

    TreeNode(int v) {
        val = v;
        left = null;
        right = null;
    }
}

// Example: build a tiny tree
//        1
//       / \
//      2   3
TreeNode root = new TreeNode(1);
root.left  = new TreeNode(2);
root.right = new TreeNode(3);

# A single binary tree node
class TreeNode:
    def __init__(self, val=0, left=None, right=None):
        self.val = val
        self.left = left
        self.right = right

# Example: build a tiny tree

Terminology

Tree words are easiest to learn against a picture, so let's borrow the family-tree analogy one more time. Before the formal table, here are the words in plain English:

  • The person at the very top is the root — the one node with nobody above it.
  • A node directly above another is its parent; the ones directly below are its children. Two children of the same parent are siblings.
  • A node with no children — the "bottom" of a branch — is a leaf. Everything else (a node with at least one child) is an internal node.
  • Depth = how far down a node sits from the root (root is depth 0). Height = how far up a node is from its deepest leaf. The tree's height is just the depth of its deepest node.
  • A node's ancestors are everyone on the path back up to the root; its subtree is the node itself plus everyone hanging below it.

Depth vs. height — a common confusion:
  • Depth is measured downward from the root to the node.
  • Height is measured upward from the deepest leaf to the node.
  • Some textbooks count nodes instead of edges, making both quantities one larger. This page uses the edge-count convention (a single-node tree has height 0).

Types of Binary Trees

Binary trees are classified by their shape — how children are distributed and how balanced the structure is.

Full (a.k.a. proper / strict)

Every node has either 0 or 2 children — no node has exactly one child.

       /     \
    ( 2 )   ( 3 )
            /   \
        ( 4 )   ( 5 )

Complete

All levels are completely filled except possibly the last, and the last level is filled left to right with no gaps. This is exactly the shape a binary heap uses, which makes array storage compact.

        ( 1 )
       /     \
    ( 2 )   ( 3 )
    /   \   /
 ( 4 )( 5 )( 6 )      last level fills left-to-right

Perfect

Every internal node has two children and all leaves are on the same level. A perfect tree of height h has exactly 2h+1−1 nodes.

          ( 1 )
        /       \
     ( 2 )     ( 3 )
     /   \     /   \
  ( 4 )( 5 )( 6 )( 7 )    all leaves at the same depth

Balanced

The height is kept O(log n), typically by bounding the height difference between the left and right subtrees of every node (e.g. AVL trees keep it ≤ 1). Balance guarantees fast search/insert/delete.

       /     \
    ( 2 )   ( 3 )
    /         \
 ( 4 )       ( 5 )    heights of subtrees differ by ≤ 1

Degenerate / Skewed

Each parent has only one child, so the tree collapses into a linked list with height n−1. This is the worst case for an unbalanced BST and destroys logarithmic performance.

Relationships: Every perfect tree is both full and complete. Every complete tree is balanced, but a complete tree need not be full (its last internal node may have a single left child). A degenerate tree is the opposite extreme of a balanced tree.

Properties & Math Facts

These identities follow directly from the at-most-two-children rule and are worth memorizing — they justify the O(log n) bounds of balanced trees and the array layout of heaps.

Worked sanity check on the perfect tree above (height h = 2, n = 7 nodes):
  • Max nodes at level 2 = 22 = 4 ✓ (nodes 4,5,6,7).
  • Max nodes of height 2 = 23−1 = 7 ✓.
  • Leaves L = 4, internal nodes = L−1 = 3 ✓ (nodes 1,2,3).
  • Edges = n−1 = 6 ✓.

Representations

There are two standard ways to store a binary tree in memory.

Linked representation

  • Each node is an object holding val, left, right.
  • Flexible: works for any shape, grows on demand.
  • Pointer overhead; nodes scattered in memory (poor cache locality).
  • The general-purpose default.

Array representation

  • Nodes stored by position in a flat array, no pointers.
  • Compact and cache-friendly for complete trees (heaps).
  • Wastes space for sparse / skewed trees (gaps must be reserved).
  • Index arithmetic replaces child links.

In the 0-indexed array layout, for a node at index i:

  • Left child is at 2i + 1
  • Right child is at 2i + 2
  • Parent is at (i - 1) / 2 (integer division)

       /     \        value: [A, B, C, D, E, F, G]

    /   \   /   \     node A (i=0): left=1 (B), right=2 (C)

The helper functions below encapsulate that index arithmetic, with bounds checks so missing children return a sentinel:

// 0-indexed array layout of a binary tree
// Returns -1 when the requested node does not exist.
int leftChild(int i, int n) {
    int l = 2 * i + 1;
    return (l < n) ? l : -1;
}

int rightChild(int i, int n) {
    int r = 2 * i + 2;
    return (r < n) ? r : -1;
}

int parent(int i) {
    return (i <= 0) ? -1 : (i - 1) / 2;
}

// 0-indexed array layout of a binary tree
// Returns -1 when the requested node does not exist.
static int leftChild(int i, int n) {
    int l = 2 * i + 1;
    return (l < n) ? l : -1;
}

static int rightChild(int i, int n) {
    int r = 2 * i + 2;
    return (r < n) ? r : -1;
}

static int parent(int i) {
    return (i <= 0) ? -1 : (i - 1) / 2;
}

# 0-indexed array layout of a binary tree
# Returns None when the requested node does not exist.
def left_child(i, n):
    l = 2 * i + 1
    return l if l < n else None

def right_child(i, n):
    r = 2 * i + 2
    return r if r < n else None

def parent(i):
    return None if i <= 0 else (i - 1) // 2
Watch out: the array layout is only space-efficient for complete trees. For a degenerate/skewed tree of n nodes, the array would need up to 2n−1 slots — use the linked representation for arbitrary shapes.
Where binary trees show up:
  • Expression trees — operators as internal nodes, operands as leaves, evaluated bottom-up.
  • Heaps — complete binary trees backing priority queues (array-stored).
  • Binary search trees (BSTs) — ordered keys enabling O(log n) search when balanced.
  • Tries — conceptually a branching tree; binary tries branch on individual bits.
  • Huffman trees — built greedily to assign shortest codes to frequent symbols in compression.
  • Parsing & ASTs — compilers represent nested syntax as trees for analysis and code generation.
