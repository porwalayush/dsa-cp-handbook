Constructing Binary Trees from Traversals

Think of a traversal as a fingerprint the tree leaves behind. If you walk the tree in a fixed order and write down the values you pass, you get a sequence. The question this page answers is the reverse: given the fingerprints, can you rebuild the exact original tree?

It turns out one fingerprint is never enough — but the right two are. The reason is a neat division of labor: one traversal tells you who the root is, and a second one (inorder) tells you which nodes go left of the root and which go right. Put those two facts together, recurse, and the whole tree falls out. The step-through builder below shows exactly this happening, one node at a time.

Why One Traversal Is Not Enough

A single traversal is ambiguous: many different tree shapes leave the exact same fingerprint. Consider the preorder sequence [1, 2]. Two completely different trees yield it:

preorder = [1, 2]  -- two valid trees, same sequence

   Tree A            Tree B
     1                  1
    /                    \
   2                      2

Both visit root 1 first, then 2. Preorder alone cannot tell us
whether 2 is a left child or a right child.

The fix is to combine traversals. The key property we exploit is that inorder tells us, for any chosen root, exactly which nodes fall in the left subtree and which fall in the right subtree (everything left of the root in inorder is the left subtree, everything right is the right subtree). A second traversal (preorder or postorder) tells us who the root is. Together they reconstruct the tree.

Which Pairs Uniquely Determine a Tree?

Traversal PairUnique?Reason
Preorder + InorderYesPreorder gives root; inorder splits left/right subtrees.
Postorder + InorderYesPostorder gives root (last element); inorder splits subtrees.
Level-order + InorderYesLevel-order gives root; inorder splits, restricted per subtree.
Preorder + PostorderOnly for full treesCannot distinguish a single child as left vs right; not unique in general.
Any single traversalNoNo way to locate subtree boundaries.
Assumption for all constructions below: node values are distinct. If duplicates exist, the value-to-index lookup is ambiguous and these algorithms break down.

1. Build from Preorder + Inorder

This is the workhorse pairing, and it's exactly what the builder at the top of the page animates. The recipe relies on two facts:

  • Preorder visits root, then left subtree, then right subtree — so the first unused preorder element is always the next root.
  • Inorder visits left subtree, then root, then right subtree — so once we know the root's position in inorder, everything to its left is the left subtree and everything to its right is the right subtree.

Walk-through

preorder = [3, 9, 20, 15, 7]
inorder  = [9, 3, 15, 20, 7]

Step 1: root = preorder[0] = 3
        find 3 in inorder at index 1
        inorder left  = [9]          (1 node)
        inorder right = [15, 20, 7]  (3 nodes)

              3
             / \
          [9]   [15,20,7]

Step 2: left subtree, next preorder = 9
        find 9 in inorder -> [9] empty on both sides -> leaf 9

Step 3: right subtree, next preorder = 20
        find 20 in inorder [15, 20, 7] at index 1
        inorder left = [15], inorder right = [7]

              3
             / \
            9   20
               /  \
             15    7

Final tree:
              3
             / \
            9    20
                /  \
              15     7

A naive implementation does a linear scan of inorder to find the root each time — O(n) per node, O(n²) total. We get O(n) by precomputing a hashmap value → inorder index and using a single moving preorder pointer that advances as we consume roots.

struct TreeNode {
    int val;
    TreeNode *left, *right;
    TreeNode(int v) : val(v), left(nullptr), right(nullptr) {}
};

class Solution {
    vector<int> pre, in;
    unordered_map<int, int> idx;  // value -> inorder index
    int preIdx = 0;            // moving preorder pointer

    TreeNode* build(int inL, int inR) {
        if (inL > inR) return nullptr;
        int rootVal = pre[preIdx++];        // next root
        TreeNode* root = new TreeNode(rootVal);
        int mid = idx[rootVal];             // split point in inorder
        root->left  = build(inL, mid - 1); // left first (preorder order)
        root->right = build(mid + 1, inR);
        return root;
    }
public:
    TreeNode* buildTree(vector<int>& preorder, vector<int>& inorder) {
        pre = preorder; in = inorder;
        for (int i = 0; i < (int)in.size(); i++) idx[in[i]] = i;
        return build(0, in.size() - 1);
    }
};

class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int v) { val = v; }
}

class Solution {
    int[] pre;
    Map<Integer, Integer> idx = new HashMap<>();
    int preIdx = 0;

    TreeNode build(int inL, int inR) {
        if (inL > inR) return null;
        int rootVal = pre[preIdx++];
        TreeNode root = new TreeNode(rootVal);
        int mid = idx.get(rootVal);
        root.left  = build(inL, mid - 1);
        root.right = build(mid + 1, inR);
        return root;
    }

    TreeNode buildTree(int[] preorder, int[] inorder) {
        pre = preorder;
        for (int i = 0; i < inorder.length; i++) idx.put(inorder[i], i);
        return build(0, inorder.length - 1);
    }
}

class TreeNode:
    def __init__(self, val=0, left=None, right=None):
        self.val, self.left, self.right = val, left, right

class Solution:
    def buildTree(self, preorder, inorder):
        idx = {v: i for i, v in enumerate(inorder)}  # value -> index
        self.pre_idx = 0                              # moving pointer

        def build(in_l, in_r):
            if in_l > in_r:
                return None
            root_val = preorder[self.pre_idx]
            self.pre_idx += 1
            root = TreeNode(root_val)
            mid = idx[root_val]
            root.left  = build(in_l, mid - 1)        # left before right
            root.right = build(mid + 1, in_r)
            return root

        return build(0, len(inorder) - 1)

2. Build from Postorder + Inorder

Postorder visits left subtree, then right subtree, then root — so the last postorder element is the root. We scan the postorder pointer backwards, and because the right subtree is consumed before the left in reverse, we must build the right subtree before the left.

postorder = [9, 15, 7, 20, 3]
inorder   = [9, 3, 15, 20, 7]

Step 1: root = postorder[last] = 3, find in inorder at index 1
        left  = [9], right = [15, 20, 7]
        (move postorder pointer left, build RIGHT subtree first)

Step 2: next root (from back) = 20, in inorder [15,20,7] at index 1
        left = [15], right = [7]

Resulting tree (identical to section 1):
              3
             / \
            9    20
                /  \
              15     7

class Solution {
    vector<int> post;
    unordered_map<int, int> idx;  // value -> inorder index
    int postIdx;               // moving pointer, starts at end

    TreeNode* build(int inL, int inR) {
        if (inL > inR) return nullptr;
        int rootVal = post[postIdx--];     // last element = root
        TreeNode* root = new TreeNode(rootVal);
        int mid = idx[rootVal];
        root->right = build(mid + 1, inR); // right BEFORE left
        root->left  = build(inL, mid - 1);
        return root;
    }
public:
    TreeNode* buildTree(vector<int>& inorder, vector<int>& postorder) {
        post = postorder;
        postIdx = post.size() - 1;
        for (int i = 0; i < (int)inorder.size(); i++) idx[inorder[i]] = i;
        return build(0, inorder.size() - 1);
    }
};

class Solution {
    int[] post;
    Map<Integer, Integer> idx = new HashMap<>();
    int postIdx;

    TreeNode build(int inL, int inR) {
        if (inL > inR) return null;
        int rootVal = post[postIdx--];
        TreeNode root = new TreeNode(rootVal);
        int mid = idx.get(rootVal);
        root.right = build(mid + 1, inR);  // right BEFORE left
        root.left  = build(inL, mid - 1);
        return root;
    }

    TreeNode buildTree(int[] inorder, int[] postorder) {
        post = postorder;
        postIdx = postorder.length - 1;
        for (int i = 0; i < inorder.length; i++) idx.put(inorder[i], i);
        return build(0, inorder.length - 1);
    }
}

class Solution:
    def buildTree(self, inorder, postorder):
        idx = {v: i for i, v in enumerate(inorder)}
        self.post_idx = len(postorder) - 1      # start at end

        def build(in_l, in_r):
            if in_l > in_r:
                return None
            root_val = postorder[self.post_idx]
            self.post_idx -= 1
            root = TreeNode(root_val)
            mid = idx[root_val]
            root.right = build(mid + 1, in_r)     # right BEFORE left
            root.left  = build(in_l, mid - 1)
            return root

        return build(0, len(inorder) - 1)

3. Build from Preorder + Postorder (Full Binary Trees)

Without inorder we lose the clean left/right split, so this pair is generally not enough to recover a unique tree. The construction works only when we assume the tree is full (every node has 0 or 2 children).

The idea: preorder[0] is the root, and preorder[1] must be the root of the left subtree. We locate preorder[1] in postorder — everything up to and including it forms the left subtree, the rest (minus the final root) forms the right subtree.

preorder  = [1, 2, 4, 5, 3, 6, 7]
postorder = [4, 5, 2, 6, 7, 3, 1]

root = preorder[0] = 1
left root = preorder[1] = 2
  find 2 in postorder at index 2 -> left subtree size = 2 + 1 = 3
  left  postorder = [4, 5, 2],  right postorder = [6, 7, 3]

           1
          / \
         2    3
        / \  / \
       4  5 6   7
Not unique in general. If a node has exactly one child, preorder + postorder cannot tell whether that child is on the left or the right. The reconstruction is only well defined when every node has 0 or 2 children (a full / proper binary tree). With single-child nodes, multiple distinct trees share the same pre/post sequences. 
class Solution:
    # Assumes a FULL binary tree (every node has 0 or 2 children).
    def constructFromPrePost(self, preorder, postorder):
        post_idx = {v: i for i, v in enumerate(postorder)}
        self.pre_idx = 0

        def build(post_l, post_r):
            if post_l > post_r:
                return None
            root = TreeNode(preorder[self.pre_idx])
            self.pre_idx += 1
            if post_l == post_r:           # leaf: no children
                return root
            # next preorder value is the left subtree's root
            left_root = preorder[self.pre_idx]
            mid = post_idx[left_root]      # its position in postorder
            root.left  = build(post_l, mid)
            root.right = build(mid + 1, post_r - 1)
            return root

        return build(0, len(postorder) - 1)

4. Build a Balanced BST from a Sorted Array

A sorted array already encodes the inorder traversal of any BST. To get a height-balanced BST, pick the middle element as the root at every step: the left half becomes the left subtree, the right half the right subtree. This guarantees O(log n) height.

sorted = [1, 2, 3, 4, 5, 6, 7]

mid index = 3 -> root = 4
  left  half = [1, 2, 3]   right half = [5, 6, 7]
    mid -> 2                  mid -> 6
    left [1]  right [3]       left [5]  right [7]

              4
            /   \
           2     6
          / \   / \
         1   3 5   7

Height = O(log n) -- perfectly balanced.

class Solution {
    TreeNode* build(vector<int>& a, int lo, int hi) {
        if (lo > hi) return nullptr;
        int mid = lo + (hi - lo) / 2;   // middle = root
        TreeNode* root = new TreeNode(a[mid]);
        root->left  = build(a, lo, mid - 1);
        root->right = build(a, mid + 1, hi);
        return root;
    }
public:
    TreeNode* sortedArrayToBST(vector<int>& nums) {
        return build(nums, 0, nums.size() - 1);
    }
};

class Solution {
    TreeNode build(int[] a, int lo, int hi) {
        if (lo > hi) return null;
        int mid = lo + (hi - lo) / 2;   // middle = root
        TreeNode root = new TreeNode(a[mid]);
        root.left  = build(a, lo, mid - 1);
        root.right = build(a, mid + 1, hi);
        return root;
    }

    TreeNode sortedArrayToBST(int[] nums) {
        return build(nums, 0, nums.length - 1);
    }
}

class Solution:
    def sortedArrayToBST(self, nums):
        def build(lo, hi):
            if lo > hi:
                return None
            mid = (lo + hi) // 2          # middle = root
            root = TreeNode(nums[mid])
            root.left  = build(lo, mid - 1)
            root.right = build(mid + 1, hi)
            return root

        return build(0, len(nums) - 1)
See the BST section for more on binary search tree properties, insertion, and validation. The sorted-array-to-BST trick is the inverse of an inorder traversal: an inorder walk of the result reproduces the original sorted array.

5. Build from Level-order + Inorder

Level-order (BFS) lists nodes top-to-bottom, left-to-right, so the first level-order element that appears within a given inorder range is that subtree's root. Inorder then splits the range into left and right halves as before.

Approach:

  1. Record each value's position in level-order with a hashmap (so "earliest in level-order" is an O(1) comparison).
  2. Build the inorder index map value → inorder index.
  3. Recurse on an inorder range [l, r]: scan the candidates and pick the one with the smallest level-order position as the root (or, more efficiently, pass down the candidate root for each side). The root splits inorder into [l, mid-1] and [mid+1, r].

The clean O(n) version filters the level-order array into two sub-sequences (left-subtree values and right-subtree values, preserving level order) at each step, then recurses. A practical pattern: keep a global level-order sequence and, for each inorder sub-range, the first level-order value lying inside that range is its root.

levelorder = [3, 9, 20, 15, 7]
inorder    = [9, 3, 15, 20, 7]

range [9,3,15,20,7]: first level-order value present = 3 -> root
   left inorder = [9], right inorder = [15,20,7]
range [15,20,7]: first level-order value present = 20 -> root
   left = [15], right = [7]

              3
             / \
            9    20
                /  \
              15     7

6. Complexity Comparison

The reconstruction logic is identical — the only difference is how you locate the root inside inorder. A linear scan is the naive trap; a hashmap makes it O(1).

StrategyRoot LookupTotal TimeSpace
Naive (linear scan of inorder)O(n) per nodeO(n²)O(n) recursion
Hashmap of inorder indicesO(1) per nodeO(n)O(n) map + recursion
Sorted array → balanced BSTO(1) (middle index)O(n)O(log n) recursion
The universal trick. Every construction in this guide reduces to the same two-part recipe:
  • A hashmap of inorder indices (value → index) so the left/right split is found in O(1) instead of scanning — this is what turns O(n²) into O(n).
  • A single global pointer into the preorder (advance forward) or postorder (advance backward) array, so you never re-slice the sequence; the recursion only carries the current inorder [lo, hi] bounds.
Match the recursion order to the pointer direction: preorder builds left before right; postorder (consumed from the back) builds right before left.
