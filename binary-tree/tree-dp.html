Tree DP & Rerooting

Picture a company org chart. You’re a manager, and your boss asks: “How many people are under you, total?” You don’t count them yourself. You ask each of your direct reports the same question, wait for their answers, add them up, add yourself, and report that one number up the chain. That tiny ritual — ask the children, fold their answers into one small summary, hand it to your parent — is the entire idea behind tree dynamic programming.

The path-problems page solved a whole family of questions with one DFS skeleton. This page zooms out: that skeleton was a special case of a general pattern called tree DP, where each node computes a compact “summary” from its children’s summaries. Once you see the pattern, problems that look wildly different — robbing houses, placing security cameras, summing distances — all collapse into the same shape.

1. The Tree-DP Mindset

Tree DP is just the org-chart ritual made precise. Process the tree in postorder (children before parent), and have each node return a small fixed-size summary — often a single number or a tiny tuple — computed purely from its children’s summaries plus the node’s own value. The parent never re-examines a grandchild; it trusts the summary its child handed up.

On the Tree Path Problems page, the Maximum Path Sum solution did exactly this: each node returned its “best gain ending here” (a one-number summary) while quietly updating a global answer. That was tree DP in disguise. The mental upgrade here is to stop thinking “path trick” and start thinking “what is the smallest summary a node must report so its parent can do its own job?” Answer that question and the recursion writes itself.

The tree-DP skeleton (postorder)
--------------------------------
solve(node):
    if node is null: return base-case summary
    childSummaries = [solve(c) for c in node.children]
    mySummary = combine(node.value, childSummaries)   # the only creative step
    return mySummary

The whole craft is designing "summary" and "combine":
  * summary must be SMALL  (O(1) per node) and SELF-CONTAINED
  * combine uses only the node + its children's summaries
  * sometimes a global answer is updated as a side effect
How to design the summary. Ask: “If my children already solved their subtrees perfectly and handed me an answer, what do I need from them to solve mine?” If one number isn’t enough, return a tuple. House Robber III needs two numbers per node; cameras need three states. The summary is always O(1), so the whole pass is O(n).

2. House Robber III

You’re a (cartoon) burglar and the houses are arranged as a tree. Each house has some cash. The one rule: you can’t rob two houses that are directly connected (parent and child), or the silent alarm trips. Maximize the cash you take. This is the classic line “House Robber” problem bent onto a tree.

The summary insight: for each node, the only thing the parent cares about is two numbers — how much can I make in this subtree if I rob this node, and how much if I skip it. Return that pair {rob, skip} and the parent combines them:

  • If I rob this node: I take its cash, but then I must skip both children. So rob = node.val + left.skip + right.skip.
  • If I skip this node: each child is free to do whatever is best for it — rob or skip, whichever is larger. So skip = max(left.rob, left.skip) + max(right.rob, right.skip).

The final answer is max(root.rob, root.skip). One postorder pass, O(n) time, O(1) per node. Try the step-through widget below to watch the {rob, skip} pair get computed bottom-up.

struct TreeNode { int val; TreeNode *left, *right; };

// returns {rob, skip} for the subtree rooted at node
pair<int, int> dfs(TreeNode* node) {
    if (!node) return {0, 0};       // base case
    auto [lRob, lSkip] = dfs(node->left);
    auto [rRob, rSkip] = dfs(node->right);
    int rob  = node->val + lSkip + rSkip;          // take node -> skip kids
    int skip = max(lRob, lSkip) + max(rRob, rSkip); // kids free
    return {rob, skip};
}

int rob(TreeNode* root) {
    auto [r, s] = dfs(root);
    return max(r, s);
}

class TreeNode { int val; TreeNode left, right; }

// returns {rob, skip} for the subtree rooted at node
int[] dfs(TreeNode node) {
    if (node == null) return new int[]{0, 0};
    int[] L = dfs(node.left);
    int[] R = dfs(node.right);
    int rob  = node.val + L[1] + R[1];        // take node -> skip kids
    int skip = Math.max(L[0], L[1])
             + Math.max(R[0], R[1]);          // kids free
    return new int[]{rob, skip};
}

int rob(TreeNode root) {
    int[] res = dfs(root);
    return Math.max(res[0], res[1]);
}

def rob(root):
    def dfs(node):                          # returns (rob, skip)
        if not node:
            return (0, 0)                    # base case
        l_rob, l_skip = dfs(node.left)
        r_rob, r_skip = dfs(node.right)
        rob_here  = node.val + l_skip + r_skip          # skip kids
        skip_here = max(l_rob, l_skip) + max(r_rob, r_skip)
        return (rob_here, skip_here)
    return max(dfs(root))
Why two numbers, not one? If a node returned only its best subtree total, the parent couldn’t tell whether that total used the child — and the adjacency rule depends on exactly that. Carrying both the “used me” and “didn’t use me” values lets the parent make a legal choice. This “return one value per relevant state” idea generalizes to the camera problem next.

3. Binary Tree Cameras (Greedy Postorder)

Now you’re installing security cameras. A camera on a node watches itself, its parent, and its direct children. You want every node watched using the fewest cameras. This feels like an optimization nightmare, but a small greedy insight tames it: never waste a camera on a leaf — a leaf is covered for free by putting the camera on its parent, which also covers the parent and the grandparent. So decisions should bubble up from the bottom.

Each node reports one of three states to its parent:

  • 0 — “I need coverage”: I have no camera and nothing below me is watching me. My parent must place a camera.
  • 1 — “I’m covered, no camera here”: a child of mine has a camera watching me. My parent can relax about me.
  • 2 — “I have a camera”: I’m watching upward, so my parent is covered too.

The combine rule, postorder: if either child says “I need coverage” (0), I must place a camera here (return 2, count++). Else if either child has a camera (2), I’m already covered (return 1). Otherwise both children are merely covered, so I’m exposed — return 0 and let my parent handle me. Treat null children as state 1 (covered) so leaves end up reporting 0. One final check at the root: if the root reports 0, add one camera.

// states: 0 = needs cover, 1 = covered, 2 = has camera
int cameras = 0;

int dfs(TreeNode* node) {
    if (!node) return 1;            // null is "covered"
    int L = dfs(node->left);
    int R = dfs(node->right);
    if (L == 0 || R == 0) {        // a child is exposed
        cameras++;
        return 2;                      // place camera here
    }
    if (L == 2 || R == 2) return 1; // a child covers me
    return 0;                          // exposed, ask parent
}

int minCameraCover(TreeNode* root) {
    cameras = 0;
    if (dfs(root) == 0) cameras++;  // root left exposed
    return cameras;
}

Same shape, different summary. House Robber III returned a 2-tuple of sums; cameras return a single state code and mutate a global counter. Both are postorder, both fold children into an O(1) report. Once you internalize “design the summary,” the two problems feel like cousins rather than strangers.

4. The Children-Sum Property

A tree has the children-sum property when, for every non-leaf node, the node’s value equals the sum of its children’s values (a missing child counts as 0). Think of it as a budget rule: every manager’s number must equal the sum of their direct reports’ numbers. Leaves are free to hold anything.

Checking it

Pure tree DP: each node reports its own value upward; a parent compares its value against the sum of what its children reported. If any node fails, the whole tree fails. One postorder pass.

bool isChildrenSum(TreeNode* node) {
    if (!node || (!node->left && !node->right))
        return true;                       // null or leaf: ok
    int sum = (node->left  ? node->left->val  : 0)
            + (node->right ? node->right->val : 0);
    return node->val == sum
        && isChildrenSum(node->left)
        && isChildrenSum(node->right);
}

boolean isChildrenSum(TreeNode node) {
    if (node == null || (node.left == null && node.right == null))
        return true;                       // null or leaf: ok
    int sum = (node.left  != null ? node.left.val  : 0)
            + (node.right != null ? node.right.val : 0);
    return node.val == sum
        && isChildrenSum(node.left)
        && isChildrenSum(node.right);
}

def is_children_sum(node):
    if not node or (not node.left and not node.right):
        return True                      # null or leaf: ok
    s = (node.left.val  if node.left  else 0) \
      + (node.right.val if node.right else 0)
    return (node.val == s
            and is_children_sum(node.left)
            and is_children_sum(node.right))

Converting a tree to satisfy it

You’re allowed to only increase node values. The trick is a two-way conversation per node:

  • Top-down nudge: if the children’s total is already ≥ the parent, bump the parent up to match it (we can only increase, so the larger side wins). If the parent is larger, push the surplus down into a child so the children can grow to meet the parent.
  • Bottom-up fix: after recursing, recompute the parent as the sum of its (now possibly larger) children, so any growth that happened deeper is reflected upward.

That postorder “recompute parent from children after the recursion” step is the tree-DP move — the parent’s final value is a summary of the subtree below it.

void convert(TreeNode* node) {
    if (!node || (!node->left && !node->right)) return;
    int childSum = (node->left  ? node->left->val  : 0)
                 + (node->right ? node->right->val : 0);
    if (childSum >= node->val) {
        node->val = childSum;                 // bump parent up
    } else {
        int diff = node->val - childSum;       // push surplus down
        if (node->left)  node->left->val  += diff;
        else            node->right->val += diff;
    }
    convert(node->left);
    convert(node->right);
    int total = (node->left  ? node->left->val  : 0)
             + (node->right ? node->right->val : 0);
    node->val = total;                        // bottom-up fix
}

5. Rerooting: An Answer for Every Node, in O(n)

Here’s the question that breaks naive tree DP: for every node, compute the sum of distances from that node to all other nodes. The obvious approach — run a fresh BFS/DFS from each node — is O(n²). For a million-node tree that’s a trillion steps. Rerooting gets the same answer in O(n).

The intuition is a relay race between neighbors. Imagine you’ve computed the answer for the root. Now you want the answer for a child c. Instead of recomputing from scratch, ask: what changes when I “move the camera” from the parent to c? Every node in c’s own subtree gets one step closer to c, and every node not in c’s subtree gets one step farther. So:

ans[child] = ans[parent] - (nodes inside child's subtree)
                          + (nodes outside child's subtree)

Let cnt[c] = size of c's subtree, n = total nodes.
nodes outside = n - cnt[c], so:

ans[child] = ans[parent] - cnt[child] + (n - cnt[child])
           = ans[parent] + n - 2 * cnt[child]

That one-line update is the whole magic. The algorithm is two passes:

  1. Pass 1 (postorder, bottom-up): compute cnt[v] = subtree size, and ans[root] = total distance from the root, by accumulating child contributions. (When you pull a child’s subtree up one level, you add cnt[child] to the running distance — one extra edge for each node down there.)
  2. Pass 2 (preorder, push-down): walk from the root downward; for each child apply ans[child] = ans[parent] + n - 2*cnt[child], reusing the parent’s freshly finalized answer.

The first pass is ordinary tree DP. The second pass is what “rerooting” means: you reuse a neighbor’s answer to get yours in O(1), so visiting all nodes is O(n) total.

The mental model: Pass 1 answers “everything below me.” Pass 2 hands each node the missing piece — “everything above me” — which the parent already knows, adjusted by one step. Down-contribution (pass 1) + up-contribution (pass 2) = the full answer for every node. Many rerooting problems (sum of distances, longest path from each node, counting via subtree + complement) follow this exact two-pass template; only the combine rule changes. 
// general tree as adjacency list; sum of distances for every node
vector<vector<int>> adj;
vector<int> cnt, ans;
int n;

void postorder(int u, int parent) {
    cnt[u] = 1;
    for (int v : adj[u]) if (v != parent) {
        postorder(v, u);
        cnt[u]  += cnt[v];
        ans[0] += 0;                 // (ans[root] built below)
        ans[u] += ans[v] + cnt[v];   // +1 edge per node below v
    }
}

void preorder(int u, int parent) {
    for (int v : adj[u]) if (v != parent) {
        ans[v] = ans[u] + n - 2 * cnt[v];   // reroot move
        preorder(v, u);
    }
}

vector<int> sumOfDistances(int N) {
    n = N; cnt.assign(n, 0); ans.assign(n, 0);
    postorder(0, -1);   // ans[0] = total distance from root
    preorder(0, -1);    // push answers to every node
    return ans;
}

def sum_of_distances(n, adj):
    cnt = [0] * n
    ans = [0] * n

    def postorder(u, parent):
        cnt[u] = 1
        for v in adj[u]:
            if v != parent:
                postorder(v, u)
                cnt[u] += cnt[v]
                ans[u] += ans[v] + cnt[v]   # +1 edge per node below

    def preorder(u, parent):
        for v in adj[u]:
            if v != parent:
                ans[v] = ans[u] + n - 2 * cnt[v]  # reroot move
                preorder(v, u)

    postorder(0, -1)   # ans[0] = total distance from root
    preorder(0, -1)    # push answers to every node
    return ans
Watch out: rerooting works on a general tree (adjacency list, no fixed parent pointers), so always pass the parent to avoid walking back up an edge. The reroot formula assumes the contribution combines cleanly when you move the root by one edge — true for distance sums and subtree sizes, but double-check the algebra for your specific quantity before trusting the one-liner.

Every algorithm here visits each node O(1) times with an O(1) combine, so all are O(n). h is the tree height (recursion stack), O(log n) balanced and O(n) skewed. Rerooting stores per-node arrays, hence O(n) space.

The patterns to remember:
  • Design the summary first. Decide the smallest O(1) report a node must hand its parent (one number, a tuple, or a state code), then the postorder combine almost writes itself.
  • One value per relevant state. When a parent’s legal choices depend on what a child did, return one value per state — {rob, skip} for House Robber, three states for cameras.
  • Reroot to go from one-node to every-node. If you can answer for the root with a postorder pass and express a child’s answer as a small adjustment of the parent’s, a second push-down pass gives all n answers in O(n) — no per-node restart.
