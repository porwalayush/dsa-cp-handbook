Hash Maps & Sets

A hash table maps keys to slots via a hash function, giving expected O(1) insert, lookup, and delete. The whole art is handling collisions — when two keys want the same slot.

How Hashing Works

   key  --hash()-->  big number  --% capacity-->  slot index

   hash(42) % 8 = 2   →   put 42 in bucket 2
   hash(10) % 8 = 2   →   COLLISION! 10 also wants bucket 2

   Two strategies to resolve collisions:
   • CHAINING        — each bucket holds a list of entries
   • OPEN ADDRESSING — probe to the next free slot

Interactive: Open Addressing (linear probing)

Chaining vs Open Addressing

Chaining

  • Each bucket = a linked list / vector
  • Simple deletes; tolerates high load
  • Extra pointers / cache misses

Open Addressing

  • All data in one array — cache friendly
  • Probing: linear, quadratic, double hash
  • Deletes need tombstones; needs load < ~0.7

Load Factor & Resizing

  • Load factor = entries / buckets. As it rises, collisions and probe lengths grow.
  • Most libraries resize (double capacity & rehash) when load passes a threshold (~0.7) → amortized O(1).
  • Keep open-addressing tables below ~70% full or performance degrades sharply.

What You'll Use Them For

Frequency counting — Counter / freq map.

Membership / dedup — set lookups.

Grouping — anagrams, buckets by key.

Memoization — cache results by argument.

Two-sum-style — value → index maps.

Common Pitfalls

1. Anti-hash tests: default unordered_map can be forced to O(n²) — add a randomized custom hash in CP.

2. Hashing mutable keys: never mutate a key after inserting (Python: only hashable/immutable keys).

3. Deletion in open addressing: use tombstones, or lookups past the gap break.

4. Iteration order: hash containers are unordered — use a tree map / sort if order matters.
