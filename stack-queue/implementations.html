Building Stacks & Queues From Scratch

Here's a favorite interview move: "You have a stack. Build me a queue." Or the reverse. Or: "Build a queue with no linked list — just a plain fixed array." These questions feel like riddles, but each one has a tidy trick once you see the picture.

The whole game is about order. A stack hands back the last thing you put in (LIFO — last in, first out). A queue hands back the first thing you put in (FIFO — first in, first out). When someone asks you to build one out of the other, you're really being asked: how do I flip the order around? Let's build all three classics, slowly.

1. Queue Using Two Stacks

Picture your desk. You have an inbox tray where new mail lands on top of the pile, and an outbox tray you actually read from. Both trays are stacks — you only ever touch the top.

New mail? Drop it on the inbox. Want to read the oldest letter? Here's the magic: if your outbox is empty, you grab the whole inbox pile and flip it over into the outbox — one letter at a time, top to top. The letter that was buried at the bottom of the inbox (the oldest) is now sitting right on top of the outbox. That flip reverses the order, turning LIFO into FIFO.

So we keep two stacks:

  • in — everything you enqueue gets pushed here. Always O(1).
  • out — you dequeue by popping from here. If it's empty, pour all of in into it first (this is the reversing flip), then pop.

The clever part is the cost. A single dequeue that triggers a transfer looks expensive — it moves every element. But each element is moved from in to out exactly once in its whole life, and is popped once. Spread that cost over all the operations and each one averages out to constant time. That's what amortized O(1) means.

Amortized intuition: don't judge a single operation in isolation — judge the average over a long run. One dequeue might shove 100 items across, but those 100 items will never be moved again. It's like paying a yearly bill once: pricey that day, cheap per day across the year. 
class MyQueue {
    std::stack<int> in, out;
public:
    // Always cheap: just drop onto the in-stack.
    void enqueue(int x) { in.push(x); }

    int dequeue() {
        if (out.empty()) {            // out is dry — pour in over
            while (!in.empty()) {       // this reverses the order
                out.push(in.top());
                in.pop();
            }
        }
        int front = out.top();
        out.pop();
        return front;
    }
};

class MyQueue {
    Deque<Integer> in = new ArrayDeque<>();
    Deque<Integer> out = new ArrayDeque<>();

    // Always cheap: just drop onto the in-stack.
    void enqueue(int x) { in.push(x); }

    int dequeue() {
        if (out.isEmpty()) {          // out is dry — pour in over
            while (!in.isEmpty()) {     // this reverses the order
                out.push(in.pop());
            }
        }
        return out.pop();
    }
}

class MyQueue:
    def __init__(self):
        self._in = []     # push/pop from the end
        self._out = []

    def enqueue(self, x):
        self._in.append(x)            # always cheap

    def dequeue(self):
        if not self._out:               # out is dry — pour in over
            while self._in:            # this reverses the order
                self._out.append(self._in.pop())
        return self._out.pop()

2. Stack Using Two Queues

Now flip the riddle. You only have queues — FIFO lines where new people join the back and you can only serve the front. But you need a stack, where the newest item must come out first. How do you make the back-of-the-line newcomer jump to the front?

The trick: every time you push, you immediately rotate the line so the new arrival ends up at the very front. Enqueue the new value, then take every older element from the front and send it to the back. After this shuffle, the most recent element is first in line — and a normal FIFO dequeue now behaves exactly like a stack pop.

You can do this with just one queue, which is the cleanest version. On push, enqueue x, then move the previous size - 1 elements (everything that was there before x) to the back, one rotation at a time.

Where's the cost? This is a push-heavy design. The rotation touches every existing element, so push is O(n). But pop and top are then a plain O(1) dequeue. (There's also a pop-heavy variant that flips which operation pays — pick based on which one you call more.) 
// Single-queue rotate trick: newest always rotated to the front.
class MyStack {
    std::queue<int> q;
public:
    void push(int x) {
        q.push(x);                       // new element joins the back
        for (int i = 0; i < (int)q.size() - 1; i++) {
            q.push(q.front());           // move older items behind it
            q.pop();
        }
    }
    int pop()  { int v = q.front(); q.pop(); return v; }
    int top()  { return q.front(); }
    bool empty() { return q.empty(); }
};

// Single-queue rotate trick: newest always rotated to the front.
class MyStack {
    Queue<Integer> q = new LinkedList<>();

    void push(int x) {
        q.add(x);                        // new element joins the back
        for (int i = 0; i < q.size() - 1; i++) {
            q.add(q.remove());           // move older items behind it
        }
    }
    int pop()  { return q.remove(); }
    int top()  { return q.peek(); }
    boolean empty() { return q.isEmpty(); }
}

from collections import deque

# Single-queue rotate trick: newest always rotated to the front.
class MyStack:
    def __init__(self):
        self.q = deque()

    def push(self, x):
        self.q.append(x)                 # new element joins the back
        for _ in range(len(self.q) - 1):
            self.q.append(self.q.popleft())  # rotate older items behind

    def pop(self):  return self.q.popleft()
    def top(self):  return self.q[0]
    def empty(self): return not self.q

3. Circular Array Queue

What if you're handed nothing but a plain fixed-size array — no linked list, no growing? A naive queue would dequeue from the front and shift every other element left, which is O(n) per dequeue. Wasteful.

Instead, think of the array as a ring. Two pointers chase each other around it: front marks where the next item leaves, and rear marks where the next item enters. When a pointer runs off the end of the array, it wraps back to index 0 using the modulo operator (% capacity). Nothing ever shifts — the pointers just circle, reusing the freed slots behind them.

Enqueue writes at rear, then advances rear with wraparound. Dequeue reads at front, then advances front the same way. We also track size so we always know how many items are really in the ring.

The off-by-one trap — full vs. empty: in a ring, front == rear happens both when the queue is completely empty and when it's completely full — the pointers land on the same slot either way. You cannot tell those two apart from the pointers alone. The cleanest fix is to track a separate size counter (full when size == capacity, empty when size == 0). The alternative — leaving one slot permanently empty — works too, but the size approach is harder to get wrong. 
class CircularQueue {
    std::vector<int> buf;
    int front = 0, size = 0, cap;
public:
    CircularQueue(int capacity) : buf(capacity), cap(capacity) {}

    bool enqueue(int x) {
        if (size == cap) return false;     // ring is full
        int rear = (front + size) % cap;  // wrap around
        buf[rear] = x;
        size++;
        return true;
    }

    int dequeue() {
        if (size == 0) return -1;        // ring is empty
        int v = buf[front];
        front = (front + 1) % cap;        // advance, wrapping
        size--;
        return v;
    }
};

class CircularQueue {
    private int[] buf;
    private int front = 0, size = 0, cap;

    CircularQueue(int capacity) {
        buf = new int[capacity];
        cap = capacity;
    }

    boolean enqueue(int x) {
        if (size == cap) return false;     // ring is full
        int rear = (front + size) % cap;  // wrap around
        buf[rear] = x;
        size++;
        return true;
    }

    int dequeue() {
        if (size == 0) return -1;        // ring is empty
        int v = buf[front];
        front = (front + 1) % cap;        // advance, wrapping
        size--;
        return v;
    }
}

class CircularQueue:
    def __init__(self, capacity):
        self.buf = [0] * capacity
        self.front = 0
        self.size = 0
        self.cap = capacity

    def enqueue(self, x):
        if self.size == self.cap:
            return False                 # ring is full
        rear = (self.front + self.size) % self.cap  # wrap
        self.buf[rear] = x
        self.size += 1
        return True

    def dequeue(self):
        if self.size == 0:
            return -1                    # ring is empty
        v = self.buf[self.front]
        self.front = (self.front + 1) % self.cap   # advance
        self.size -= 1
        return v

Complexity at a Glance

Three constructions, three different cost profiles. Notice how each one pushes the cost onto a different operation — there's no free lunch, just trade-offs you choose on purpose.

How to pick in an interview: if you can use any structure, the circular array queue gives true O(1) on both ends with zero wasted shifting — it's what real ring buffers use. The two-stacks and two-queues builds are about proving you understand order reversal, so be ready to explain the LIFO ↔ FIFO flip out loud, not just code it.
