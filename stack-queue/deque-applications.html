Deque & Sliding Window Maximum

Picture a line of people — but a strange one, where someone can join or leave from either end. New arrivals can slip in at the front or the back, and you can wave goodbye to whoever is at the front or the back. That two-sided line is a deque (pronounced "deck", short for double-ended queue).

A deque is the swiss-army knife of linear containers. Use only the back and it behaves like a stack. Add at the back, remove from the front, and it's a plain queue. Because it generalizes both, it's the perfect tool for problems where you need to drop things from both sides — like our star problem below.

The Six Operations

Every operation on a deque is O(1) — constant time, no matter how many elements are inside:

  • pushFront(x) — add x to the front.
  • pushBack(x) — add x to the back.
  • popFront() — remove and return the front element.
  • popBack() — remove and return the back element.
  • peekFront() — look at the front without removing it.
  • peekBack() — look at the back without removing it.

Every language ships a ready-made deque. Here's the basic usage in all three:

// std::deque supports both ends in O(1)
#include <deque>
std::deque<int> dq;

dq.push_back(10);     // back:  [10]
dq.push_front(5);     // front: [5, 10]

int f = dq.front();    // peek front -> 5
int b = dq.back();     // peek back  -> 10

dq.pop_front();        // removes 5  -> [10]
dq.pop_back();         // removes 10 -> []

bool empty = dq.empty();

// ArrayDeque is the go-to deque in Java
import java.util.ArrayDeque;
ArrayDeque<Integer> dq = new ArrayDeque<>();

dq.addLast(10);     // back:  [10]
dq.addFirst(5);     // front: [5, 10]

int f = dq.peekFirst();  // peek front -> 5
int b = dq.peekLast();   // peek back  -> 10

dq.pollFirst();          // removes 5  -> [10]
dq.pollLast();           // removes 10 -> []

boolean empty = dq.isEmpty();

# collections.deque is a doubly linked list
from collections import deque
dq = deque()

dq.append(10)         # back:  [10]
dq.appendleft(5)      # front: [5, 10]

f = dq[0]            # peek front -> 5
b = dq[-1]           # peek back  -> 10

dq.popleft()           # removes 5  -> [10]
dq.pop()               # removes 10 -> []

empty = not dq

The Star Application: Sliding Window Maximum

Here's the problem that makes the deque shine. You're given an array and a window of fixed size k. The window slides one step at a time from left to right, and at each position you must report the maximum inside it.

The lazy approach: at every window, scan all k elements for the max. That's O(n·k) — fine for tiny inputs, painful for big ones. The deque gets us all the way down to O(n), where each element is added and removed at most once.

The Intuition

We keep a monotonic decreasing deque of indices. The trick is captured in one sentence: the front of the deque always holds the index of the current window's maximum.

To keep that promise as a new element arrives at position i, we do three things:

  1. Clean the back. While the element at the back of the deque is smaller than the new value, pop it. Why? A smaller, older element can never be the max again while this bigger, newer one is around — it's permanently overshadowed. Then push i at the back.
  2. Expire the front. If the front index has slid out of the window (its index is <= i - k), pop it from the front. It's too old to count anymore.
  3. Record the answer. Once the window has fully formed (i >= k - 1), the value at the front index is the maximum of the current window.

Because elements only ever leave when they're too small or too old, the deque stays sorted (largest at the front, decreasing toward the back) for free.

Here's the full algorithm in all three languages, using a deque of indices:

// Sliding window maximum — deque of indices, O(n)
std::vector<int> maxSlidingWindow(std::vector<int>& nums, int k) {
    std::deque<int> dq;            // holds indices, values decreasing
    std::vector<int> res;
    for (int i = 0; i < (int)nums.size(); ++i) {
        // 1. clean the back: drop smaller, overshadowed values
        while (!dq.empty() && nums[dq.back()] < nums[i])
            dq.pop_back();
        dq.push_back(i);
        // 2. expire the front if it slid out of the window
        if (dq.front() <= i - k)
            dq.pop_front();
        // 3. record once the window has fully formed
        if (i >= k - 1)
            res.push_back(nums[dq.front()]);
    }
    return res;
}

// Sliding window maximum — ArrayDeque of indices, O(n)
public int[] maxSlidingWindow(int[] nums, int k) {
    ArrayDeque<Integer> dq = new ArrayDeque<>(); // indices, values decreasing
    int[] res = new int[nums.length - k + 1];
    int r = 0;
    for (int i = 0; i < nums.length; i++) {
        // 1. clean the back: drop smaller, overshadowed values
        while (!dq.isEmpty() && nums[dq.peekLast()] < nums[i])
            dq.pollLast();
        dq.addLast(i);
        // 2. expire the front if it slid out of the window
        if (dq.peekFirst() <= i - k)
            dq.pollFirst();
        // 3. record once the window has fully formed
        if (i >= k - 1)
            res[r++] = nums[dq.peekFirst()];
    }
    return res;
}

# Sliding window maximum — deque of indices, O(n)
from collections import deque

def maxSlidingWindow(nums, k):
    dq = deque()          # holds indices, values decreasing
    res = []
    for i, v in enumerate(nums):
        # 1. clean the back: drop smaller, overshadowed values
        while dq and nums[dq[-1]] < v:
            dq.pop()
        dq.append(i)
        # 2. expire the front if it slid out of the window
        if dq[0] <= i - k:
            dq.popleft()
        # 3. record once the window has fully formed
        if i >= k - 1:
            res.append(nums[dq[0]])
    return res

Complexity

ApproachTimeSpaceNotes
Brute force (rescan each window)O(n·k)O(1)Re-scans k elements per step
Monotonic dequeO(n)O(k)Each index pushed & popped once
Any single deque operationO(1)—pushFront/Back, popFront/Back, peek

Why O(n) and not O(n·k)? Each index enters the deque exactly once and leaves at most once. Even though a single step might pop several elements off the back, those pops are "paid for" by the earlier pushes — across the whole run the total work is bounded by 2n. The deque never holds more than k indices, so space is O(k).

Why store indices, not values? The deque needs two pieces of information about its front: what the value is (to report the max) and when it entered (to know if it has expired). If you stored only values you could read the max, but you'd have no way to tell that the front element slid out of the window. Storing the index gives you both — nums[front] recovers the value, and front <= i - k checks expiry.
Three bugs that bite everyone:
  • Comparing values but expiring by index. The back-cleaning step compares values (nums[dq.back()] < nums[i]), but the front-expiry step compares indices (dq.front() <= i - k). Mixing these up — e.g. expiring by value — is a classic mistake.
  • > vs >= on duplicates. Using nums[back] < nums[i] (strict) keeps equal values in the deque, which is correct and safe. If you pop on <= you discard duplicates that may still be valid maxima of later windows.
  • Recording too early. Only push to the result once i >= k - 1 — that's when the first full window exists. Recording before the window has formed produces a spurious leading answer.
