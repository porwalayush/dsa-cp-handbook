Queues: The Basics

Picture the line at a ticket counter. You walk up, take your place at the back, and wait. The person who arrived first gets served first; the person who just showed up waits the longest. Nobody cuts. That's a queue — a fair, orderly line where service follows arrival.

The rule has a name: FIFO — First In, First Out. The first item to enter is the first to leave. That's the exact opposite of a stack, which is LIFO (Last In, First Out) — like a pile of plates where you always grab the top one you just put down. A queue is a line; a stack is a pile.

The Two Ends That Matter

A queue has two ends, and keeping them straight is the whole game:

  • The rear (or back) — where new items join the line.
  • The front — where items leave the line.

Add at the rear, remove from the front. Just like real people in a real line.

Core Operations

Everything a queue does boils down to four moves — and each one is O(1), meaning it takes the same tiny amount of time no matter how long the line is:

  • enqueue(x) — add x to the back of the line.
  • dequeue() — remove and return the item at the front.
  • front() / peek() — look at the front item without removing it.
  • isEmpty() — is the line empty?

Queue Operations in Code

Every language ships a ready-made queue. Here's enqueue, dequeue, and front in each:

// std::queue wraps a deque under the hood
#include <queue>

std::queue<int> q;

q.push(5);            // enqueue at the rear
q.push(7);
q.push(3);            // queue: front [5, 7, 3] rear

int f = q.front();   // peek front -> 5  (does NOT remove)
q.pop();             // dequeue the front -> removes 5

bool empty = q.empty();

// Queue is an interface; back it with a LinkedList or ArrayDeque
Queue<Integer> q = new ArrayDeque<>();

q.offer(5);          // enqueue at the rear (offer or add)
q.offer(7);
q.offer(3);          // queue: front [5, 7, 3] rear

int f = q.peek();    // peek front -> 5  (does NOT remove)
int x = q.poll();    // dequeue the front -> returns and removes 5

boolean empty = q.isEmpty();

# collections.deque gives O(1) at BOTH ends
from collections import deque

q = deque()

q.append(5)         # enqueue at the rear
q.append(7)
q.append(3)         # queue: front [5, 7, 3] rear

f = q[0]            # peek front -> 5  (does NOT remove)
x = q.popleft()    # dequeue the front -> returns and removes 5

empty = not q
In Python, always use collections.deque for a queue — not a plain list. list.pop(0) looks innocent but is O(n): it shifts every remaining element down one slot. deque.popleft() is O(1).
Dequeueing an empty queue is a classic crash. In C++ calling pop()/front() on an empty queue is undefined behavior; in Java poll() returns null but remove() throws. Always check isEmpty() first.

Why a Naive Array Queue Is Slow

Here's the trap. You might reach for a plain array: enqueue just appends to the end (fast), and dequeue removes index 0. Easy, right?

Not quite. Removing the front of an array means every other element has to shift one slot to the left to fill the gap. For a queue of n items, that's n moves on every single dequeue — O(n). Do that in a loop and your "simple" queue is quietly quadratic.

Naive array dequeue

  • Remove arr[0].
  • Shift all remaining items left by one.
  • Cost: O(n) per dequeue. 😟

Circular buffer dequeue

  • Just move the front pointer forward.
  • Nothing shifts; the slot is reused later.
  • Cost: O(1) per dequeue. 🎉

The Fix: a Circular Buffer

Instead of physically moving elements, we keep them put and move two pointers instead:

  • front — the index of the item to dequeue next.
  • rear — the index where the next enqueue goes.

When a pointer runs off the end of the fixed-size array, it wraps back around to index 0 — as if the array's last slot were glued to its first, forming a ring. The wrap is just one line of arithmetic: index = (index + 1) % capacity. The modulo (%) does the wrapping for free.

Because we only ever bump pointers, both enqueue and dequeue stay O(1) — no shifting, ever.

Empty ring. front = 0, rear = 0.
struct CircularQueue {
    std::vector<int> buf;
    int front = 0, count = 0, cap;
    CircularQueue(int c) : buf(c), cap(c) {}

    bool enqueue(int x) {
        if (count == cap) return false;   // full
        int rear = (front + count) % cap;  // wrap!
        buf[rear] = x;
        count++;
        return true;
    }
    int dequeue() {
        int x = buf[front];
        front = (front + 1) % cap;       // wrap!
        count--;
        return x;
    }
};

class CircularQueue {
    int[] buf;
    int front = 0, count = 0, cap;
    CircularQueue(int c) { buf = new int[c]; cap = c; }

    boolean enqueue(int x) {
        if (count == cap) return false;   // full
        int rear = (front + count) % cap;  // wrap!
        buf[rear] = x;
        count++;
        return true;
    }
    int dequeue() {
        int x = buf[front];
        front = (front + 1) % cap;       // wrap!
        count--;
        return x;
    }
}

class CircularQueue:
    def __init__(self, cap):
        self.buf = [0] * cap
        self.front = 0
        self.count = 0
        self.cap = cap

    def enqueue(self, x):
        if self.count == self.cap:
            return False            # full
        rear = (self.front + self.count) % self.cap   # wrap!
        self.buf[rear] = x
        self.count += 1
        return True

    def dequeue(self):
        x = self.buf[self.front]
        self.front = (self.front + 1) % self.cap   # wrap!
        self.count -= 1
        return x

Complexity Summary

OperationTimeNotes
enqueueO(1)add at the rear
dequeueO(1)remove at the front (deque / circular buffer)
front / peekO(1)look, don't remove
isEmptyO(1)check count or pointers
dequeue (naive array)O(n)⚠️ shifts every element — avoid this
Rule of thumb: never dequeue from the front of a raw array in a loop. Use your language's deque/queue, or a circular buffer if you need a fixed-capacity ring. Both give you O(1) at both ends.

Where Queues Show Up

Once you see FIFO, you'll spot queues everywhere "first come, first served" is the right behavior:

  • Task / job scheduling — the OS run queue hands the CPU to processes in arrival order.
  • Printer queue — documents print in the order they were submitted.
  • BFS traversal — breadth-first search uses a queue to explore a graph level by level.
  • Message buffers — Kafka, SQS, and friends queue messages between producers and consumers.
  • Request handling — web servers queue incoming requests until a worker is free.
Quick recap: a queue is a line. Add at the rear, remove from the front, first in is first out. Use a deque or circular buffer so every operation stays O(1) — and never pop(0) a list.
