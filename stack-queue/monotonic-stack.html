The Monotonic Stack

Picture a line of people of different heights, all facing right. For each person you want to answer one question: who is the next taller person to my right? The lazy approach is to make each person crane their neck and scan everyone ahead of them — but that's a lot of repeated looking, and it costs O(n²).

Here's the clever idea. Keep a small group of people who are still waiting to find someone taller. As soon as a tall person walks up, every shorter person who was waiting gets their answer at once — it's this newcomer. That waiting group is a monotonic stack: a stack whose values always stay in sorted order. The moment a new element breaks that order, you start popping, and each pop resolves one answer.

A monotonic stack is just a regular stack with one rule: its contents are always kept in sorted order — either strictly increasing or strictly decreasing from bottom to top. You enforce the rule by popping anything that would violate it before you push.

The Core Mechanic

You scan the array once, left to right. The stack holds the indices of elements that are still waiting for an answer. For "next greater element," we keep the stack values decreasing from bottom to top.

At each new element, ask one thing: does it break the order? For each index on top of the stack whose value is smaller than the current value, that waiting element has just found its next greater — so pop it and record the answer. Keep popping while the rule is violated. Then push the current index, because it is now waiting too.

Why is this fast? Each index is pushed exactly once and popped at most once. That's at most 2n stack operations across the whole scan, so the entire algorithm is O(n) — even though it looks like a nested loop.

Worked Example: Next Greater Element

Let's run it on [2, 1, 2, 4, 3]. The stack holds indices; we fill answer[i] the moment index i is popped. Anything still on the stack at the end never found a greater element, so it gets -1.

Notice how step 4 resolved two waiting elements in a single moment. That batching is exactly why the work stays linear.

The Code

Here is Next Greater Element to the right, returning an array. The pattern is the same in every language: a stack of indices, fill answer[i] when index i is popped, and leftovers stay -1.

// Next greater element to the right; -1 if none.
vector<int> nextGreater(vector<int>& a) {
    int n = a.size();
    vector<int> ans(n, -1);
    stack<int> st;            // holds indices
    for (int i = 0; i < n; i++) {
        // current beats whatever is waiting on top
        while (!st.empty() && a[st.top()] < a[i]) {
            ans[st.top()] = a[i];
            st.pop();
        }
        st.push(i);
    }
    return ans;             // leftovers stay -1
}

// Next greater element to the right; -1 if none.
int[] nextGreater(int[] a) {
    int n = a.length;
    int[] ans = new int[n];
    Arrays.fill(ans, -1);
    Deque<Integer> st = new ArrayDeque<>();  // indices
    for (int i = 0; i < n; i++) {
        while (!st.isEmpty() && a[st.peek()] < a[i]) {
            ans[st.peek()] = a[i];
            st.pop();
        }
        st.push(i);
    }
    return ans;             // leftovers stay -1
}

# Next greater element to the right; -1 if none.
def next_greater(a):
    n = len(a)
    ans = [-1] * n
    st = []                  # holds indices
    for i in range(n):
        while st and a[st[-1]] < a[i]:
            ans[st.pop()] = a[i]
        st.append(i)
    return ans              # leftovers stay -1

One Idea, a Whole Family of Problems

Once the mechanic clicks, you'll start seeing it everywhere. These famous problems are all the same stack with a tweak:

Next Greater Element

The canonical version. For each item, the first larger value to its right. Decreasing stack of indices.

Daily Temperatures

"How many days until it gets warmer?" Identical pattern — but store the distance i - poppedIndex instead of the value.

Stock Span

"How many consecutive prior days was the price ≤ today's?" Same stack, looking left, counting how far back it stays dominated.

Largest Rectangle in Histogram

Use the stack to find, for each bar, the nearest shorter bar on each side. Those boundaries give the widest rectangle each bar can anchor.

The recurring signal in an interview: "for each element, find the next/previous greater/smaller one." That phrasing almost always means a monotonic stack.

Complexity

ApproachTimeSpaceWhy
Monotonic stackO(n)O(n)Each index pushed and popped at most once
Brute force (scan ahead)O(n²)O(1)For each element, look at all elements to its right
The insight to remember: the inner while loop looks like it makes the algorithm O(n²), but it doesn't. Across the entire outer loop, every element is pushed once and popped once — so the total pop work is bounded by n, not n per step. Amortized, the whole thing is linear.
Common bugs to watch for:
  • Pushing values instead of indices. You usually need the index to write into answer[i] or to compute a distance. Push indices, read values via a[index].
  • Forgetting the leftovers. Whatever is still on the stack at the end has no greater element — initialize answer to -1 so those stay correct.
  • Wrong comparison direction. For next greater keep a decreasing stack and pop while a[top] < a[i]. For next smaller, flip it: keep an increasing stack and pop while a[top] > a[i].
  • Strict vs. non-strict. Using < vs <= decides how ties are handled — pick deliberately based on whether equal values should "block" each other.
