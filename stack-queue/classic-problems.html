Classic Stack Problems & Design

Once the push / pop / top rhythm feels natural, the fun part begins: using a stack to solve real problems. The trick is almost never the stack itself — it's spotting that a problem secretly has stack-shaped logic hiding inside it.

The signature clue is simple. Whenever you find yourself thinking "I need to remember things, deal with the most recent one first, and sometimes undo it" — that's a stack. This page walks through four classics that show off that pattern, plus the most famous design problem of them all.

1. Min Stack — the classic design problem

Here's the puzzle. Build a stack that supports the usual push, pop, and top, plus a fourth operation: getMin(), which returns the smallest value currently in the stack. The catch — all four must run in O(1).

The naive idea is to scan the whole stack whenever someone asks for the minimum. But scanning is O(n), and we promised O(1). We need a way to already know the answer before the question is asked.

The insight: keep a second stack running alongside the main one — a "min stack" that remembers, at every level, what the minimum was at that point. When you push a value x, you push min(x, currentMin) onto the min stack. So the top of the min stack is always the current minimum of everything below it. Asking for the min is just reading that top: O(1).

And because both stacks grow and shrink in lockstep, when you pop the main stack you simply pop the min stack too. The history rewinds itself automatically — the new min-stack top is whatever the minimum was before you pushed.

The design insight worth keeping: when you need an aggregate (a min, a max, a running sum) answered instantly on a stack, keep a parallel stack of that aggregate. Each entry stores "what the answer was at this height." Reading it is O(1), and pop rewinds it for free. This idea generalizes far beyond min — it's the heart of many "design a data structure" interview questions.

The bug everyone hits: when you pop the main stack, you must also pop the min stack — forget that, and the two drift out of sync and getMin lies. Watch out for duplicate minimums too: if you only store the min when it's strictly smaller (a popular space optimization), you must pop it only when the value being removed equals the current min, handling repeats carefully. The two-parallel-stacks approach shown here sidesteps that entirely by always pushing and always popping in lockstep.

Min Stack in code

Each language stores the value alongside the running minimum — either as two stacks, or one stack of pairs. Both are O(1) per operation.

// Store a pair: (value, min-so-far). Top always knows the min.
#include <stack>
#include <algorithm>

class MinStack {
    std::stack<std::pair<int, int>> st;  // {value, minSoFar}
public:
    void push(int x) {
        int m = st.empty() ? x : std::min(x, st.top().second);
        st.push({x, m});                  // store value + running min
    }
    void pop()      { st.pop(); }         // pops value AND its min together
    int  top()      { return st.top().first; }
    int  getMin()   { return st.top().second; }  // O(1) — just read the top
};

// Two parallel stacks: values, and the running minimum.
class MinStack {
    private Deque<Integer> data = new ArrayDeque<>();
    private Deque<Integer> mins = new ArrayDeque<>();

    public void push(int x) {
        data.push(x);
        int m = mins.isEmpty() ? x : Math.min(x, mins.peek());
        mins.push(m);                     // mirror with the running min
    }
    public void pop() {
        data.pop();
        mins.pop();                       // MUST pop both — keep them in sync
    }
    public int top()    { return data.peek(); }
    public int getMin() { return mins.peek(); }   // O(1)
}

# Two lists. mins[-1] is always the current minimum.
class MinStack:
    def __init__(self):
        self.data = []
        self.mins = []

    def push(self, x):
        self.data.append(x)
        m = x if not self.mins else min(x, self.mins[-1])
        self.mins.append(m)               # mirror with running min

    def pop(self):
        self.data.pop()
        self.mins.pop()                   # pop BOTH — stay in lockstep

    def top(self):
        return self.data[-1]

    def getMin(self):
        return self.mins[-1]            # O(1)

2. Trapping Rain Water (stack approach)

Picture a row of vertical bars of different heights — like a city skyline. After a rainstorm, water pools in the dips between taller bars. The question: how much water gets trapped?

The stack insight: walk left to right keeping a stack of bar indices whose heights are decreasing. As long as bars keep getting shorter, water has nothing to push against yet, so we just stack them up. The moment a taller bar appears, it acts like the right wall of a container — and whatever sits on the stack is the floor.

So when the new bar is taller than the stack top, we pop the top (that's the floor of a puddle), then look at the new stack top (the left wall). The trapped water above that popped floor is width × boundedHeight, where the height is bounded by the shorter of the two walls minus the floor. Keep popping while the new bar stays taller.

Heads up: there's a slicker two-pointer solution to this problem that runs in O(1) extra space. We show the stack method here because it's the one that teaches the "a taller bar closes a container" intuition — and that same intuition powers monotonic-stack problems like Largest Rectangle. Knowing both is ideal. 
int trap(std::vector<int>& h) {
    std::stack<int> st;          // stack of indices, decreasing heights
    int water = 0;
    for (int i = 0; i < (int)h.size(); ++i) {
        while (!st.empty() && h[i] > h[st.top()]) {
            int floor = st.top(); st.pop();   // bottom of the puddle
            if (st.empty()) break;            // no left wall -> no water
            int left = st.top();              // left wall index
            int width = i - left - 1;
            int bounded = std::min(h[left], h[i]) - h[floor];
            water += width * bounded;
        }
        st.push(i);
    }
    return water;
}

int trap(int[] h) {
    Deque<Integer> st = new ArrayDeque<>();  // indices, decreasing
    int water = 0;
    for (int i = 0; i < h.length; i++) {
        while (!st.isEmpty() && h[i] > h[st.peek()]) {
            int floor = st.pop();             // puddle floor
            if (st.isEmpty()) break;          // no left wall
            int left = st.peek();
            int width = i - left - 1;
            int bounded = Math.min(h[left], h[i]) - h[floor];
            water += width * bounded;
        }
        st.push(i);
    }
    return water;
}

def trap(h):
    st = []                       # indices with decreasing heights
    water = 0
    for i, hi in enumerate(h):
        while st and hi > h[st[-1]]:
            floor = st.pop()      # bottom of the puddle
            if not st:            # no left wall -> no water held
                break
            left = st[-1]
            width = i - left - 1
            bounded = min(h[left], hi) - h[floor]
            water += width * bounded
        st.append(i)
    return water

3. Asteroid Collision

Asteroids fly along a single row. Each is a number: the magnitude is its size, the sign is its direction — positive means moving right, negative means moving left. Same-direction asteroids never catch each other. But a right-mover followed by a left-mover are on a collision course. When they hit, the smaller one explodes; equal sizes destroy each other.

This is a perfect stack simulation. The stack holds the asteroids that have survived so far. A new right-moving asteroid never collides with anything to its left, so it just gets pushed. A new left-moving asteroid, though, may collide with right-movers sitting on top of the stack — so we resolve those collisions in a loop before deciding whether the newcomer survives.

The key guard: a collision only happens when the stack top is moving right (positive) and the new asteroid is moving left (negative). Any other combination — two left-movers, two right-movers, or a left-mover already past everything — means no collision.

Only opposite-facing pairs collide. The single most common bug is comparing magnitudes without first checking directions. Two asteroids moving the same way, or a left-mover meeting a left-mover, never collide — they simply coexist. Always confirm top > 0 && current < 0 before you even think about who wins. 
std::vector<int> asteroidCollision(std::vector<int>& a) {
    std::vector<int> st;            // survivors (use a vector as a stack)
    for (int x : a) {
        bool alive = true;
        while (alive && !st.empty() && st.back() > 0 && x < 0) {
            if (st.back() < -x)      st.pop_back();   // top explodes, keep going
            else if (st.back() == -x) { st.pop_back(); alive = false; } // both die
            else                     alive = false;       // newcomer explodes
        }
        if (alive) st.push_back(x);
    }
    return st;
}

int[] asteroidCollision(int[] a) {
    Deque<Integer> st = new ArrayDeque<>();  // survivors
    for (int x : a) {
        boolean alive = true;
        while (alive && !st.isEmpty() && st.peek() > 0 && x < 0) {
            if (st.peek() < -x)      st.pop();        // top explodes
            else if (st.peek() == -x) { st.pop(); alive = false; } // both die
            else                     alive = false;     // newcomer dies
        }
        if (alive) st.push(x);
    }
    // ArrayDeque.push adds to the front — reverse to restore order
    int[] out = new int[st.size()];
    for (int i = out.length - 1; i >= 0; i--) out[i] = st.pop();
    return out;
}

def asteroidCollision(a):
    st = []                       # survivors
    for x in a:
        alive = True
        while alive and st and st[-1] > 0 and x < 0:
            if st[-1] < -x:           # top is smaller -> it explodes
                st.pop()
            elif st[-1] == -x:        # equal -> both explode
                st.pop()
                alive = False
            else:                     # top wins -> newcomer explodes
                alive = False
        if alive:
            st.append(x)
    return st

4. Simplify Path (Unix path)

Given an absolute Unix-style path like /a/./b/../../c/, produce its canonical form. The rules are exactly the ones your terminal follows: . means "stay here" (ignore it), .. means "go up one directory" (undo the last step), and empty pieces from double slashes are skipped.

"Undo the last step" is the giveaway — that's a stack. Split the path on /, then walk the pieces: push real directory names, pop on .. (going up), and ignore . and empties. At the end, join the stack with slashes and you have the clean path.

std::string simplifyPath(std::string path) {
    std::vector<std::string> st;
    std::stringstream ss(path);
    std::string part;
    while (std::getline(ss, part, '/')) {
        if (part == "" || part == ".") continue;   // skip empties and "."
        if (part == "..") { if (!st.empty()) st.pop_back(); } // go up
        else st.push_back(part);                       // a real dir name
    }
    std::string out;
    for (auto& d : st) out += "/" + d;
    return out.empty() ? "/" : out;
}

String simplifyPath(String path) {
    Deque<String> st = new ArrayDeque<>();
    for (String part : path.split("/")) {
        if (part.isEmpty() || part.equals(".")) continue;  // skip
        if (part.equals("..")) { if (!st.isEmpty()) st.pop(); } // up
        else st.push(part);                            // real dir
    }
    StringBuilder sb = new StringBuilder();
    Iterator<String> it = st.descendingIterator();   // bottom -> top order
    while (it.hasNext()) sb.append("/").append(it.next());
    return sb.length() == 0 ? "/" : sb.toString();
}

def simplifyPath(path):
    st = []
    for part in path.split("/"):
        if part == "" or part == ".":      # skip empties and "."
            continue
        if part == "..":                    # go up one directory
            if st:
                st.pop()
        else:
            st.append(part)               # a real directory name
    return "/" + "/".join(st)           # "/" alone if stack is empty
Two quick bonus classics, same flavor: Remove All Adjacent Duplicates — push each character, but if it matches the current top, pop instead; the leftover stack is your answer. And Backspace String Compare — treat # as a backspace: push real characters, pop on #, then compare the two resulting stacks. Both are just "push normally, pop to undo" — the same shape as Simplify Path.

Complexity at a Glance

Every problem here is linear in time — each element is pushed and popped at most once. The min-stack operations are individually constant time, which is the whole reason it's a famous design problem.

ProblemTimeSpace
Min Stack (push / pop / top / getMin)O(1) eachO(n)
Trapping Rain Water (stack)O(n)O(n)
Asteroid CollisionO(n)O(n)
Simplify PathO(n)O(n)

In each linear-time problem, "n" is the input size and every item enters and leaves the stack at most once — that amortizes the inner while loops down to O(n) overall.

The Pattern Behind All of Them

Step back and the four problems rhyme. Each keeps a stack of "things decided so far," and the newest arrival either gets pushed or triggers a cascade of pops that resolve the past.

Design problems

  • Min Stack — a parallel aggregate stack answers min/max in O(1).
  • Max Stack, "stack with O(1) sum" — same parallel-stack trick.
  • Insight: store the answer at each height, pop to rewind.

Simulation problems

  • Asteroid Collision — stack of survivors, resolve on contact.
  • Simplify Path — push dirs, pop on ...
  • Insight: the stack IS the running state; pop = undo.
Pattern to remember: if a problem asks for an instant aggregate (min/max/sum) — design a parallel stack. If it asks you to process a sequence where new items cancel, collide with, or undo recent ones — simulate with a stack of survivors. Spot which of the two you're facing and the rest of the solution writes itself.
