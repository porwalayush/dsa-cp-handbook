Expression Parsing & Evaluation

Here's a question that sounds easy until you try it: how does a computer read 3 + 4 * 2 and know the answer is 11, not 14? It has to honor "multiply before add" — and it has to handle parentheses, nesting, and operators all the way down.

The surprising answer is that almost every piece of this puzzle is solved with one humble tool: the stack. Matching brackets, reordering operators, evaluating math — they're all stack problems wearing different hats. This is the single most famous family of stack applications, and once you see the pattern, you'll spot it everywhere.

We'll build it up in four steps, easiest first: checking brackets, converting infix to postfix, evaluating postfix, and finally a basic calculator that ties it together.

1. Balanced / Valid Parentheses

Start with the simplest version. Forget math — just ask: is ([]{}) properly nested, and is ([)] broken? A human checks this by eye, but what's the actual rule?

The rule is: every opening bracket must be closed by the matching type, in the right order. And "right order" is the giveaway — the last bracket you opened must be the first one you close. That's LIFO. That's a stack.

So the algorithm reads left to right:

  • See an opening bracket — (, [, { — push it.
  • See a closing bracket — the stack's top must be its matching opener. If the stack is empty (nothing to close) or the top is the wrong type, it's invalid. Otherwise pop the match and move on.
  • At the very end, the stack must be empty. Anything left over means an opener never got closed.

Think of it like nesting boxes: you can only close the box you most recently opened.

#include <stack>
#include <string>

bool isValid(const std::string& s) {
    std::stack<char> st;
    for (char c : s) {
        if (c == '(' || c == '[' || c == '{') {
            st.push(c);                  // opener: remember it
        } else {
            if (st.empty()) return false;  // closer with nothing open
            char top = st.top();
            if ((c == ')' && top != '(') ||
                (c == ']' && top != '[') ||
                (c == '}' && top != '{'))
                return false;            // wrong TYPE — mismatch
            st.pop();                    // matched pair
        }
    }
    return st.empty();               // nothing left open = balanced
}

import java.util.*;

boolean isValid(String s) {
    Deque<Character> st = new ArrayDeque<>();
    Map<Character, Character> match = Map.of(')', '(', ']', '[', '}', '{');
    for (char c : s.toCharArray()) {
        if (c == '(' || c == '[' || c == '{') {
            st.push(c);                          // opener
        } else {
            // closer: stack must be non-empty AND top must match the type
            if (st.isEmpty() || st.pop() != match.get(c))
                return false;
        }
    }
    return st.isEmpty();                      // must end empty
}

def is_valid(s):
    stack = []
    match = {')': '(', ']': '[', '}': '{'}
    for c in s:
        if c in '([{':
            stack.append(c)              # opener: push
        else:
            # closer: need a matching opener on top
            if not stack or stack.pop() != match[c]:
                return False
    return not stack                     # empty stack = balanced

2. Infix → Postfix (the Shunting-Yard Algorithm)

Humans write math infix: the operator sits between its operands, like a + b. It reads nicely, but it forces you to remember precedence rules and juggle parentheses. Machines hate that ambiguity.

Postfix (also called Reverse Polish Notation, RPN) puts the operator after its operands: a b +. The magic: postfix needs no parentheses and no precedence rules. The order is baked right into the sequence. a + b * c becomes a b c * + — and there's only one way to read it.

Edsger Dijkstra's shunting-yard algorithm converts infix to postfix using — you guessed it — a stack, this time holding operators while operands flow straight to the output. The name comes from a railway shunting yard: cars (operators) get parked on a side track and released in the right order.

Scanning left to right:

  • Operand (a number or variable) → send it straight to output.
  • Operator → while the operator on top of the stack has higher or equal precedence (for left-associative operators), pop it to output. Then push the new operator.
  • Left paren ( → just push it (it's a barrier).
  • Right paren ) → pop operators to output until you hit the matching (, then discard the pair.
  • End of input → pop any remaining operators to output.

Operator precedence & associativity

"Higher precedence" just means "binds tighter / done first." Here's the standard ladder:

OperatorMeaningPrecedenceAssociativity
^exponent3 (highest)right
* /multiply, divide2left
+ -add, subtract1 (lowest)left

Associativity breaks ties between equal-precedence operators. + - * / are left-associative: a - b - c means (a - b) - c, so we pop an equal-precedence top before pushing. ^ is right-associative: 2 ^ 3 ^ 2 means 2 ^ (3 ^ 2), so we do not pop an equal-precedence ^ — we let it stack up.

#include <stack>
#include <string>
#include <cctype>

int prec(char op) {
    if (op == '^') return 3;
    if (op == '*' || op == '/') return 2;
    if (op == '+' || op == '-') return 1;
    return 0;
}

std::string toPostfix(const std::string& s) {
    std::stack<char> ops;
    std::string out;
    for (char c : s) {
        if (isalnum(c)) {
            out += c;                          // operand -> output
        } else if (c == '(') {
            ops.push(c);
        } else if (c == ')') {
            while (!ops.empty() && ops.top() != '(') { out += ops.top(); ops.pop(); }
            if (!ops.empty()) ops.pop();         // discard the '('
        } else {                                // an operator
            // '^' is right-assoc: stop popping at EQUAL precedence
            bool rightAssoc = (c == '^');
            while (!ops.empty() && ops.top() != '(' &&
                   (prec(ops.top()) > prec(c) ||
                    (prec(ops.top()) == prec(c) && !rightAssoc))) {
                out += ops.top(); ops.pop();
            }
            ops.push(c);
        }
    }
    while (!ops.empty()) { out += ops.top(); ops.pop(); }  // drain
    return out;
}

import java.util.*;

int prec(char op) {
    if (op == '^') return 3;
    if (op == '*' || op == '/') return 2;
    if (op == '+' || op == '-') return 1;
    return 0;
}

String toPostfix(String s) {
    Deque<Character> ops = new ArrayDeque<>();
    StringBuilder out = new StringBuilder();
    for (char c : s.toCharArray()) {
        if (Character.isLetterOrDigit(c)) {
            out.append(c);                       // operand -> output
        } else if (c == '(') {
            ops.push(c);
        } else if (c == ')') {
            while (!ops.isEmpty() && ops.peek() != '(') out.append(ops.pop());
            if (!ops.isEmpty()) ops.pop();         // drop '('
        } else {
            boolean rightAssoc = (c == '^');
            while (!ops.isEmpty() && ops.peek() != '(' &&
                   (prec(ops.peek()) > prec(c) ||
                    (prec(ops.peek()) == prec(c) && !rightAssoc))) {
                out.append(ops.pop());
            }
            ops.push(c);
        }
    }
    while (!ops.isEmpty()) out.append(ops.pop());   // drain
    return out.toString();
}

def to_postfix(s):
    prec = {'^': 3, '*': 2, '/': 2, '+': 1, '-': 1}
    ops, out = [], []
    for c in s:
        if c.isalnum():
            out.append(c)                    # operand -> output
        elif c == '(':
            ops.append(c)
        elif c == ')':
            while ops and ops[-1] != '(':
                out.append(ops.pop())
            if ops:
                ops.pop()                    # discard '('
        else:                                 # operator
            right_assoc = c == '^'
            while (ops and ops[-1] != '(' and
                   (prec[ops[-1]] > prec[c] or
                    (prec[ops[-1]] == prec[c] and not right_assoc))):
                out.append(ops.pop())
            ops.append(c)
    while ops:                            # drain remaining operators
        out.append(ops.pop())
    return ''.join(out)

Why bother converting? Postfix needs no parentheses and no precedence rules at evaluation time — that's exactly why compilers and calculators convert infix to postfix (or to a tree) first. Once it's postfix, evaluating it is almost trivial, as you'll see next.

3. Evaluate Postfix (RPN)

Now the payoff. Postfix is built to be evaluated by a stack in a single left-to-right pass — no precedence logic at all. This is the classic LeetCode problem "Evaluate Reverse Polish Notation."

The rule is delightfully short:

  • See a number → push it.
  • See an operator → pop two operands, apply the operator, push the result back.
  • When the input ends, exactly one value remains on the stack — that's your answer.

Order matters when you pop. The first value you pop is the right operand, the second is the left. So for a b - you compute a - b (not b - a). Get this backwards and subtraction and division silently give wrong answers.

Worked example — evaluate 3 4 2 * + 1 5 + - (the postfix our visualizer produces for 3+4*2-(1+5)): push 3, push 4, push 2, hit * → 4*2=8 → stack [3,8], hit + → 3+8=11 → [11], push 1, push 5, hit + → 6 → [11,6], hit - → 11-6=5. Answer: 5. ✓

#include <stack>
#include <vector>
#include <string>

int evalRPN(const std::vector<std::string>& tokens) {
    std::stack<int> st;
    for (const std::string& t : tokens) {
        if (t == "+" || t == "-" || t == "*" || t == "/") {
            int b = st.top(); st.pop();   // RIGHT operand (popped first)
            int a = st.top(); st.pop();   // LEFT operand
            if (t == "+") st.push(a + b);
            else if (t == "-") st.push(a - b);
            else if (t == "*") st.push(a * b);
            else st.push(a / b);          // a / b, NOT b / a
        } else {
            st.push(std::stoi(t));        // number -> push
        }
    }
    return st.top();                  // the one value left
}

import java.util.*;

int evalRPN(String[] tokens) {
    Deque<Integer> st = new ArrayDeque<>();
    for (String t : tokens) {
        switch (t) {
            case "+": case "-": case "*": case "/": {
                int b = st.pop();             // RIGHT operand
                int a = st.pop();             // LEFT operand
                if (t.equals("+")) st.push(a + b);
                else if (t.equals("-")) st.push(a - b);
                else if (t.equals("*")) st.push(a * b);
                else st.push(a / b);          // a / b — order matters
                break;
            }
            default:
                st.push(Integer.parseInt(t));   // number
        }
    }
    return st.pop();
}

def eval_rpn(tokens):
    st = []
    ops = {
        '+': lambda a, b: a + b,
        '-': lambda a, b: a - b,
        '*': lambda a, b: a * b,
        '/': lambda a, b: int(a / b),   # truncate toward zero
    }
    for t in tokens:
        if t in ops:
            b = st.pop()                  # RIGHT operand (popped first)
            a = st.pop()                  # LEFT operand
            st.append(ops[t](a, b))       # apply, push result
        else:
            st.append(int(t))             # number -> push
    return st[0]                        # last value standing
Three classic bugs to watch for:
  • Forgetting the empty-stack check at the end of bracket matching. "(" alone never triggers a mismatch — it's only caught because the stack isn't empty when you finish. Skip that final check and you wrongly call it balanced.
  • Matching count, not type. "(]" has one opener and one closer, so a naive counter says "balanced." It isn't — the type must match the top of the stack.
  • Operand order in postfix. The first pop is the right operand. For a b - it's a - b; for a b / it's a / b. Swap them and - and / break. Also beware integer division and how your language truncates negatives.

4. Basic Calculator

The grand finale: evaluate an infix string like 3+4*2-(1+5) directly, with correct precedence and parentheses. This is LeetCode's "Basic Calculator" family.

You have two honest options, and both lean on stacks:

  • Two-pass: run shunting-yard (step 2) to get postfix, then evaluate it (step 3). Clean and easy to reason about — you've already built both halves.
  • One-pass: walk the string keeping a stack of values plus the last operator seen. When you hit a higher-precedence operator like * or /, apply it immediately to the top of the stack; + and - just push the (signed) number. Parentheses recurse (or push the current state and resume after the matching )). At the end, sum the value stack.

Here's the compact one-pass version handling + - * / and parentheses via recursion:

def calculate(s):
    def helper(it):
        st, num, op = [], 0, '+'
        while True:
            c = next(it, ')')        # treat end like a closing paren
            if c.isdigit():
                num = num * 10 + int(c)  # build multi-digit numbers
            elif c == '(':
                num = helper(it)         # recurse: value of the sub-expression
            elif c == ' ':
                continue
            else:                         # operator or ')' — flush pending num with last op
                if op == '+': st.append(num)
                elif op == '-': st.append(-num)
                elif op == '*': st.append(st.pop() * num)  # precedence: act now
                elif op == '/': st.append(int(st.pop() / num))
                num, op = 0, c
                if c == ')':
                    return sum(st)        # sub-expression done
    return helper(iter(s))

The trick: + and - defer (push the signed number), while * and / apply immediately to the stack top — that's how precedence falls out naturally. Parentheses just recurse and return their summed value as a single number. If you'd rather not reason about all that at once, the two-pass route (shunting-yard then RPN eval) is just as correct and easier to debug.

Complexity at a Glance

Every algorithm here makes a single left-to-right pass and touches each token a constant number of times (push once, pop once). So they're all linear.

AlgorithmTimeSpace
Valid parenthesesO(n)O(n)
Infix → postfix (shunting-yard)O(n)O(n)
Evaluate postfix (RPN)O(n)O(n)
Basic calculatorO(n)O(n)

Space is O(n) because in the worst case (e.g. all openers, or a long chain of equal-precedence operators) the stack can hold a constant fraction of the input.

The big picture: infix is for humans, postfix is for machines, and a stack is the bridge between them. Bracket matching, conversion, and evaluation are the same idea — "remember what's open, resolve it in last-in-first-out order" — applied three times. Spot that shape and these problems stop being scary.
