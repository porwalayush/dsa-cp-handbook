Stacks: The Basics

Picture a pile of plates in a cafeteria. You wash a plate and set it on top of the pile. The next person grabs a plate — and they take the one on top, the one you just put there. The plate at the very bottom might sit there all day.

That's a stack. You only ever touch the top. The last thing you put on is the first thing you take off. Computer scientists gave this a name: LIFO — Last In, First Out.

A pile of books works the same way. So does a stack of pancakes, or the spring-loaded plate dispenser at a buffet. If you've ever pressed Undo, you've used a stack — it undoes the most recent thing first.

The Four Core Operations

A stack is deliberately tiny. There are really only four things you can do with it — and almost all of them touch only the top, which is exactly why they're so fast (O(1), constant time).

  • push(x) — put a new item on top of the stack. (Set a plate down.)
  • pop() — remove and return the top item. (Take the top plate.)
  • peek() / top() — look at the top item without removing it. (Glance at the top plate; don't pick it up.)
  • isEmpty() — ask "is there anything here at all?" before you reach in.

Notice what's missing: you can't peek at the middle, you can't pull a plate from the bottom, you can't jump to "item #3." That restriction is the feature — it keeps every operation instant and the mental model dead simple.

Using a Stack in Code

You almost never build a stack by hand — every language ships one in its standard library. Here's how to declare one and run the four operations in each language.

// C++ ships std::stack
#include <stack>

std::stack<int> s;

s.push(5);          // stack: [5]
s.push(7);          // stack: [5, 7]
s.push(3);          // stack: [5, 7, 3]  top = 3

int t = s.top();    // peek -> 3 (not removed)
s.pop();             // remove top (3). NOTE: pop() returns void

if (!s.empty()) {    // always check before touching the top
    int x = s.top();  // 7
}

// Java: prefer Deque (ArrayDeque) as a stack — faster than legacy Stack
Deque<Integer> s = new ArrayDeque<>();

s.push(5);            // stack: [5]
s.push(7);            // stack: [5, 7]
s.push(3);            // stack: [5, 7, 3]  top = 3

int t = s.peek();     // peek -> 3 (not removed)
int x = s.pop();      // removes AND returns 3

if (!s.isEmpty()) {    // guard before reading
    int y = s.peek();   // 7
}

# Python: a plain list IS a stack — no import needed
s = []

s.append(5)        # push  -> [5]
s.append(7)        # push  -> [5, 7]
s.append(3)        # push  -> [5, 7, 3]  top = 3

top = s[-1]        # peek  -> 3 (not removed)
x = s.pop()        # pop   -> removes AND returns 3

if s:              # an empty list is falsy — this is your isEmpty check
    y = s[-1]      # 7
Tip: in Python you don't need a special class — a regular list already behaves like a stack. Use .append(x) to push and .pop() to remove the top. (Avoid list.pop(0) for stack work — that pops the front and is O(n).)
The #1 mistake: touching an empty stack. Calling pop or peek when the stack is empty is the classic bug. In C++, s.top() or s.pop() on an empty stack is undefined behavior (it may crash or silently corrupt data). In Java it throws NoSuchElementException; in Python [].pop() raises IndexError. Always check isEmpty() first.

Complexity at a Glance

The whole point of a stack is speed. Every core operation is constant time because it only ever touches the top — there's no shifting, no searching, no walking through items.

OperationTime Complexity
push (add to top)O(1)
pop (remove top)O(1)
peek / top (read top)O(1)
isEmptyO(1)
search (find a value)O(n)

Search is O(n) because a stack gives you no shortcut to the middle — you'd have to pop items off one by one to look through them.

Where Stacks Show Up in the Real World

Once you know the LIFO shape, you start seeing stacks everywhere. Whenever a problem cares about the most recent thing first, a stack is usually the answer.

Everyday software

  • Undo / Redo — each action pushed on; undo pops the latest.
  • Browser back button — pages you visited, popped newest-first.
  • Function call stack — each call pushed; returning pops it off.

Classic algorithms

  • Balanced-bracket checking — push openers, pop to match closers.
  • Expression evaluation — postfix math and infix-to-postfix conversion.
  • Depth-first search (DFS) — the recursion is a stack.

The function call stack is worth a second look: every time your program calls a function, it pushes a frame (the local variables, the return address). When the function returns, that frame is popped. That's literally why a runaway recursion causes a "stack overflow" — you pushed more frames than there was room for.

Pattern to remember: if a problem mentions "most recent," "matching pairs," "undo," or "reverse order," reach for a stack first. The match is almost always there.
